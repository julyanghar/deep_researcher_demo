# 手把手看懂 vLLM 的"后缀树":从字典树开始,零基础

> 缘起:拆 vLLM suffix 投机解码([18-vllm-suffix-decoding.md](18-vllm-suffix-decoding.md))时绕不开它底层那棵树。本文假设你**只知道字典树(Trie)**,其余一切从零讲起:每一节只引入一个新想法,用一个 7 token 的小序列把树**一步步建出来、再一步步用一遍**。
> 读法:§一~§六 是正文;§七 术语表(忘了名词回来查);§八 是进阶(教科书后缀树、SAM 和它的区别),跳过不影响理解 vLLM。

## 一、先分清三个词:前缀、后缀、子串

拿序列 `我 爱 苹果`(3 个 token)说:

- **前缀** = 从头开始、切到某处:`我`、`我 爱`、`我 爱 苹果`;
- **后缀** = 从某处开始、一直到结尾:`我 爱 苹果`、`爱 苹果`、`苹果`;
- **子串** = 中间随便连续截一段:上面全算,还有 `爱` 这种夹在中间的。

一个马上要用的事实:**任何子串,都是"某个后缀的前缀"**。(`爱` 是后缀 `爱 苹果` 的前缀。)记住它,整个后缀 trie 就靠这一句。

## 二、从 Trie 到 vLLM 那棵树:四步,每步只加一个新想法

### 第 1 步:Trie(你已经会的,复述一遍)

Trie 是个**容器**:你给它一堆词,它把词按公共前缀共享着存。边 = 一个字符(在我们场景里 = 一个 token),**从根到某节点的路径拼出来 = 一个串**,一个节点的所有子孙共享这条路径当前缀。查询永远**从根开始**、逐字符往下走。

它的局限:只认"从头开始"的匹配。你把 `我 爱 苹果` 整条插进去,能查 `我 爱`(前缀),**查不了 `爱 苹果`**(它不从头开始)。

### 第 2 步:后缀 Trie = 同一棵 Trie,换一种"喂法"

**后缀 trie 没有任何新结构——它就是 Trie,区别只在你往里喂什么。**

- 普通 Trie:你手动给它一堆词(比如一本词典);
- 后缀 Trie:**喂的词固定为"某一条序列的所有后缀"**。

为什么这样喂?慢慢推,三步:

**① 先钉死 Trie 的本领(定义性质)。** 在 Trie 里,"从根出发能一步步走出 P" **当且仅当** "P 是**某个被插入的词**的开头一段(前缀)"。为什么:根路径都是插入词的时候一笔一笔刻出来的;反过来,查询走到半路停下也完全合法,不需要走到什么词尾。Trie 的一切能力都是这一条。

**② 看清"查子串"卡在哪。** Trie 只会**从词的开头**开始匹配。假如把整条 S 当**一个词**插进去,根路径就只有"S 的前缀"——想查 `爱 苹果`?它从 S 的**位置 2** 开始,不是 S 的开头,从根第一步(边上是 `我`)就走不通了。**子串的麻烦 = 它可以从 S 的任何位置开始,而 Trie 只认"开头"。**

**③ 那就给每个位置人造一个"开头"。** 把"从位置 i 一直到结尾"的那一段——也就是**后缀**——当作一个独立的词插进去,位置 i 就当上了"某个词的开头"。7 个位置插 7 条后缀,插完之后 **S 的每一个位置都是某个词的开头**。这时拿任意子串 P 来查:P 总是从 S 的某个位置 i 开始的,而"从 i 开始的后缀"已经作为词躺在 Trie 里,P 恰好是它的开头一段——按①,P 必然能从根走出来。反方向也不会误报:根路径全是各条后缀刻的,凡是走得通的 P 都是某条后缀的开头一段,也就必然是 S 里真实存在的子串。

拿例子过一遍:S = `我 爱 苹果 , 我 爱 香蕉`,查 P = `爱 苹果`。整条 S 当一个词时查不到(开头是 `我`);但**位置 2 的后缀** `爱 苹果 , 我 爱 香蕉` 被插入时,已经从根刻出了 `根→爱→苹果→,→…` 这条路,P 正是这条路的前两步 → 走得通 ✓。

所以 §一那句"任何子串 = 某个后缀的前缀"不是文字游戏,它说的就是:**每个子串都有自己的起点 i,而"从 i 开始的后缀"替它把路修到了根上。**"从根查前缀"升级成"查任意子串",升级的不是 Trie,是我们把"每个位置"都变成了开头。

顺手再白捡一个宝贝:给每个节点配个计数 `count` = "有多少条后缀经过这个节点"。你会发现 **count 恰好等于"这个子串在序列里出现了几次"**。频次不用额外算,建树时顺手就有了——vLLM 后面估"置信度"全靠它。

### 第 3 步:广义后缀 Trie = 多条序列共用一棵

- **狭义(普通)后缀 trie**:只为**一条**序列建,喂这一条的所有后缀;
- **广义(generalized)后缀 trie**:**多条序列共用同一棵树**——每条序列各自把自己的后缀喂进来,`count` 跨序列**累加**,同时每个分支记着"这段是谁贡献的"(源码里的 `seq_id` / `endpoints`),以便日后单独删掉某条序列。

"广义"就这点事,没有更多玄机("广义后缀树"也是同样的用法:字符串数据结构里,"广义 = 多串版")。为什么需要它?vLLM 的**全局树**要把上万个历史请求的输出装进同一棵树里跨请求共享——见 §五。

容易理解偏的一点:"把多条序列放在一起"有三种做法,广义后缀 Trie 特指第三种——
- **(a) 几棵独立的树并排摆着**(每序列一棵):查询要逐棵查、自己合并,频次投票无从谈起。这不叫广义;不过 vLLM 的**本地树**恰是这个形态(dict,每请求一棵、互不相干、用完整棵扔)。
- **(b) 拼接成一条再建一棵**:接缝处会刻出"A 结尾+B 开头"的**假子串**。例:A=`我 爱 苹果`、B=`你 爱 香蕉`,拼成一条后,"从 苹果 开始的窗口"是 `苹果 你 爱`——树里凭空出现 `苹果→你` 这条路,查 `苹果 你` 会说"存在",可它在 A、B 里都不存在;放到 vLLM 里更荒谬:speculate 会把"请求 B 的第一个词"当成"请求 A 结尾的续写"猜出来,纯白验证。(教科书的广义后缀树确实走拼接路线,靠给每条序列配不同终结符 `$1 $2…` 挡住跨界匹配,工程上繁琐,而且照样删不掉单条序列。)
- **(c) 一棵树共享节点,按 seq_id 分账** ← 广义后缀 Trie:相同的子串**走同一条路径、同一个节点**(A 里 `爱` 出现 2 次、B 里 1 次 → 不是两个节点,是同一个节点 count 2→3),所以**一次走树 = 同时问遍所有序列**,投票自动跨序列合并;不同的归属靠 `endpoints`/`seq_id` 记账,所以能单独删除某条序列(count 减回去、独有分支剪掉),别的序列毫发无损。

### 第 4 步:限深 = 每条后缀只喂开头 24 个 token

后缀 trie 有个致命伤:一条长 n 的序列有 n 条后缀、平均长 n/2,节点数 **O(n²)**——1 万 token 的 prompt 就是五千万级节点,根本建不起。

vLLM 的解法简单粗暴:**每条后缀只插入它开头的 ≤24 个 token**(24 = `max_tree_depth`)。等价的说法:只把"每个起始位置往后数 24 个 token 的窗口"插进树。节点数瞬间从 O(n²) 掉到 **O(24n)**。

为什么敢砍?因为查询端(§四)拿去匹配的 pattern **本来就最长 24**——索引再深也没人查得到。

**到此,vLLM 那棵树的全名就凑齐了:限深的、带频次的、广义的后缀 Trie。**(源码里还有一个纯工程优化:路径压缩——"一条只有独苗孩子的链"合并成一个节点、只存对原序列的引用不拷贝 token,省内存,不改变任何逻辑。)

## 三、手把手建一棵(教学用深度 3,真实是 24)

序列 A(prompt,7 个 token):

```
位置:  1   2   3    4   5   6   7
token: 我  爱  苹果  ,  我  爱  香蕉
```

深度限 3,所以要插的"窗口"是每个起点往后 ≤3 个:

```
[我 爱 苹果] [爱 苹果 ,] [苹果 , 我] [, 我 爱] [我 爱 香蕉] [爱 香蕉] [香蕉]
```

怎么插?vLLM 用**滑动指针队列**(源码 `_active_nodes`,C++ 里就是 `std::deque<Node*>`,每条序列一个),一遍扫过序列就把所有窗口插完。

**队列的元素是什么?** 不是 token、不是窗口内容,也**不是 Node 结构体的拷贝**,而是**指向树节点的指针**(`Node*`,8 字节的地址)——树上的结构体只有一份,队列只是"手指头指着它";后面说的"走一步",就是把这个指针变量改指到孩子节点的地址。每个指针的语义:一个"还没长满的窗口"的**断点书签**——指向"从根沿这个窗口目前的内容走下去所到达的节点"。为什么要存书签:如果每来一个 token、每个窗口都从根重走,一个 token 的成本是 O(24²);书签让每个窗口只需从上次的位置**再走一步**,降到 O(24)——断点续传。

规则,每来一个新 token——

1. **入队**(每 token 恰好一次):队尾 push 一个**指向根**的新指针 = "从当前位置开始的窗口"诞生(**此刻还没吃进 token**,所以书签在根);
2. **出队**(超长才发生):队列超过 3 个(真实 24)就**弹掉队头**——队头永远是最老、恰好已吃满上限的窗口,弹掉 = 封口 = 不再续传。**弹掉不删除树上任何东西**,它刻的路径和 count 永远留着;
3. **走一步**(队里每个指针都做):指针指着节点 u,看 u 有没有"新 token"这个孩子——有就走进去、孩子 count+1;没有就新建(count=1)。指针更新为走到的节点。

三个不变量,记住就不会乱:①队里指针按窗口长度**严格递减**(队头最长、队尾刚出生长度 1),所以"弹队头"自动弹的就是刚长满的那个,不用显式记长度;②每个位置都当过一次队尾(所有窗口都被创建)、每个窗口都被续传到长满或序列结束(所有窗口都插完整)——合起来 = 全部 ≤24 窗口都进了树;③元素里不需要存起点/内容/长度,续传只要知道"从哪个节点接着走",一个 `Node*` 够了(窗口终点记在节点 `endpoints` 上,是给删除用的账本)。

逐步走一遍(指针用"它代表的窗口串"表示):

| 来的 token | 弹谁 | 走完后队列里的指针(=窗口) | 树上发生了什么 |
|---|---|---|---|
| 我 | — | `我` | 根下新建 我(1) |
| 爱 | — | `我爱`、`爱` | 我下新建 爱(1);根下新建 爱(1) |
| 苹果 | — | `我爱苹果`、`爱苹果`、`苹果` | 三处各新建 苹果(1) |
| , | **弹 `我爱苹果`**(满 3 封口) | `爱苹果,`、`苹果,`、`,` | 三处各新建 ,(1) |
| 我 | 弹 `爱苹果,` | `苹果,我`、`,我`、`我` | 两处新建 我(1);**根下的 我 已存在 → count 1→2** |
| 爱 | 弹 `苹果,我` | `,我爱`、`我爱`、`爱` | ,我 下新建 爱(1);**我→爱 已存在 count→2;根→爱 已存在 count→2** |
| 香蕉 | 弹 `,我爱` | `我爱香蕉`、`爱香蕉`、`香蕉` | 三处各新建 香蕉(1) |

两个容易看岔的地方(拿第 3 行"苹果"说):

1. **"队列"一列的记法**:写的是每个指针**走完这一步之后**所指节点的"根路径名"——`我爱苹果` 不是一个叫这名字的元素,而是"某个 `Node*`,当前指向路径 我→爱→苹果 末端的节点"。新指针入队瞬间确实在根,但规则 3 对**包括它在内**的所有指针执行,同一步内它立刻走了第一步——表里永远看不到停在根的指针,不是它不存在,是它活不过半步。单步显微镜:

   ```
   处理 token"苹果"(第 3 步)的内部时序:
   开始:   队列 [P1@节点(我→爱),  P2@节点(爱)]
   ① 入队: 队列 [P1@我爱,  P2@爱,  P3@根]      ← P3 此刻确实指向根
   ② 长度 3 ≤ 3,不弹
   ③ 所有指针——包括刚入队的 P3——各沿"苹果"走一步:
      P1: 节点(我→爱)下没有"苹果"这个孩子 → 新建 → P1 改指它
      P2: 节点(爱)下没有 → 新建 → P2 改指它
      P3: 根下没有 → 新建 → P3 改指它
   结束:   队列 [P1@我爱苹果,  P2@爱苹果,  P3@苹果]   ← 表里写的是这一刻
   ```

2. **"三处各新建 苹果(1)"不矛盾**:因为**同一个 token 可以出现在树的很多个节点上**——节点的身份不是边上的 token,是"从根到它的整条路径"(= 它代表的子串)。上面 ③ 里三个指针脚下各自都没有"苹果"这个孩子,于是在三个不同位置各建一个:

   ```
   根 ├─ 我(1) ── 爱(1) ── 苹果(1)   ← P1 停在这,代表子串「我 爱 苹果」
      ├─ 爱(1) ── 苹果(1)            ← P2 停在这,代表子串「爱 苹果」
      └─ 苹果(1)                     ← P3 停在这,代表子串「苹果」
   ```

   它们分别代表 `我 爱 苹果`、`爱 苹果`、`苹果`——本来就是三个不同的子串(§二说过:任何子串都要能从根走出来,这三个都得有自己的路),理应是三个不同节点,只是进入各自的最后一条边恰好都标着 苹果。Node 结构体的 `token` 字段存的正是"进入我的那条边上的 token",它在整棵树里重复出现任意多次都正常。

扫完,树长这样(括号里是 count):

```
根 ├─ 我(2) ── 爱(2) ──┬─ 苹果(1)
   │                   └─ 香蕉(1)
   ├─ 爱(2) ──┬─ 苹果(1) ── ,(1)
   │          └─ 香蕉(1)
   ├─ 苹果(1) ── ,(1) ── 我(1)
   ├─ ,(1) ── 我(1) ── 爱(1)
   └─ 香蕉(1)
```

自己验证两个 count:`我` 在序列里出现 2 次(位置 1、5)→ 根下 我(2) ✓;`我 爱` 出现 2 次 → 我→爱 count 2 ✓。**每个节点的 count = 它代表的子串的出现次数**,§二第 2 步的承诺兑现了。

**count 是怎么"算"的?根本没有统计这一步——它是建树时攒出来的。** 规则就一句:**插入**时指针每**走进**一个节点,该节点 count+1(新建节点出生时 count=1);注意只有插入在加数,**查询**(speculate 拿 pattern 来匹配)时走过节点不改 count——count 记的是"多少条窗口住在这",不是"被查过多少次"。在上表里可以核对 `根→爱` 这个节点的一生:第 2 步(token 爱)由新根指针创建,count=1;第 6 步(又一个 爱)另一条窗口的指针走进它,count 1→2。为什么攒出来恰好是出现次数:子串 P 在位置 i 出现一次 ⟺ 从 i 开始的那条窗口以 P 开头 ⟺ 插那条窗口时指针恰好路过一次 P 的节点——一次出现对应一次 +1,不多不少。删除序列时同理反着来(路过的节点 -1)。这种"实时攒"对 vLLM 是刚需:树每个 decode step 都在长、speculate 每步都要读 count,不存在"建完冻结、再统计一遍"的机会(对照 §八:教科书后缀树/SAM 的频次都要在冻结后的结构上跑一遍预处理)。源码里还多做一步:count 一变,兄弟链表就按 count 降序**实时重排**(`Group` 机制),让"最高频孩子"(`head_child`)永远 O(1) 可取——speculate 的投票就是直接拿队首。

成本:每个 token 只动 ≤3 个指针(真实系统 ≤24),1 万 token 的 prompt ≈ 24 万次指针操作,毫秒级——这就是"每个请求来了现建一棵、每步还能往里追加"的底气。

## 四、手把手用一棵:speculate(猜草稿)

现在模型在生成,已经写出了 `……所以 我 爱`("所以"是它自己的话,不在 prompt 里)。suffix 要猜:接下来是什么?

**第 1 步:取 pattern。** 拿当前序列的**结尾 ≤3 个 token**(真实 24):`[所以 我 爱]`。

**第 2 步:枚举"结尾的后缀",逐个去树里配。** 从短到长:

- **m=1**:`爱` → 根下有 爱(2) ✓。它的孩子:苹果(1)、香蕉(1)——"爱"后面接过两种东西,**投票**:最高票 苹果 占 1/2 = **0.5**(这就是置信度:`孩子count ÷ 自己count`)。0.5 ≥ 门槛 0.1,敢猜。但**预算 = 匹配长度 × 1.0 = 1**,只敢猜 1 个 → 草稿 `苹果`(置信度 0.5),score = 0.5。
- **m=2**:`我 爱` → 根→我→爱 ✓(count 2)。孩子还是 苹果/香蕉 各 1,首 token 还是 苹果(0.5);预算 = 2,但选完苹果后这条路在深度 3 处到头了 → 草稿 `苹果`(0.5),score = 0.5。
- **m=3**:`所以 我 爱` → 根下没有"所以" ✗,**枚举到此为止**(短的配不上,更长的必配不上)。

**第 3 步:score 择优。** m=1 和 m=2 都是 0.5,平手取更长的 m=2。最终草稿:`苹果`。

**第 4 步:验证(这发生在下一个 decode step 的前向里)。** 模型把 [真 token + 草稿] 一起过一遍前向,逐位置比较草稿和自己的 argmax:

- 若模型本来就想说 `苹果` → 接受,还白送一个 bonus token(草稿末尾的 argmax 反正已算出)→ 这一步净吐 2 个,省了 1 步;
- 若模型想说 `香蕉` → 草稿被拒,用 argmax 兜底吐出 `香蕉` → 这一步净吐 1 个,和不开投机一样,只是白验证了一次。

**两种结局输出都和不开投机一模一样——无损**;赌输赌赢只影响速度。

**追问一:置信度的一般式——m 很大(比如 24)时,"孩子count ÷ 自己count"算的是谁除以谁?**

比值永远发生在"**当前走到的节点**"和"**它被选中的孩子**"之间,而节点代表的是**一整段上下文**(从根到它的路径),不是单个 token:

- **第一个草稿 token**(任意 m):分母节点 = 匹配上的那 **m 个 token 整体**,所以比值 = count(m 个 token + X) ÷ count(m 个 token)——**这里的 + 是拼接(concat),不是加法**:分子数的是"这 m 个 token 后面紧跟着 X"构成的**长度 m+1 的子串**的出现次数。整个比值 = "prompt 里所有出现过**这一整段**的位置中,接 X 的比例"。m=1 时恰好退化成你理解的"生成 token → 推测 token";m=24 时条件就是整段 24-gram。(本文后面所有 count(…+…) 里的 + 同义,都是拼接。)

  拿数字算一遍(序列 `我 爱 苹果 。 我 爱 香蕉 。 我 爱 苹果 汁`,"我 爱"出现 3 次):

  ```
  根 ─ 我(3) ─ 爱(3) ─┬─ 苹果(2) ─ …
                      └─ 香蕉(1) ─ …
  ```

  匹配 m=2(`我 爱`)停在节点 爱(3)。**分母不用现算——它就躺在这个节点上**:§三说过"节点 count = 它代表的子串的出现次数",这一条就是公式和实现之间的桥。3 就是"我 爱"出现的 3 次;孩子 苹果(2) 的 count 同理 = 拼接串"我 爱 苹果"出现的 2 次,**这就是分子**。比值 2/3 的含义:"3 处出现'我 爱'的地方,2 处接的是苹果"——频率版的 P(下一个=苹果 | 前文是 我 爱)。所以 **"count(m 个+X) ÷ count(m 个)"和"孩子count ÷ 自己count"是同一件事的两种说法**——前者是含义,后者是实现:正因为分子分母都是建树时按"窗口经过就 +1"攒好、分别存在相邻两个节点上的(见 §三"count 怎么算"),查询时打一次分才只是一次整数除法,不做任何统计。(细节:各孩子的 count 之和 ≤ 父节点 count,差额是"我 爱"恰好出现在序列结尾/窗口尽头、没有下文的那几次——它们坐在分母里、不进任何分子,天然拉低所有续写的占比。)
- **第二个及以后的草稿 token**:分母换成"匹配串 + 已猜部分"的节点——每一步都是"**给定到目前为止的一切**,下一个是 X 的频率"。展开举例,序列换成 `我 爱 苹果 汁 。 我 爱 香蕉 。 我 爱 苹果 派`("我 爱"3 次;其中接 苹果 2 次;"我 爱 苹果"之后 汁、派 各 1 次):

  ```
  根 ─ 我(3) ─ 爱(3) ─┬─ 苹果(2) ─┬─ 汁(1) ─ …
                      │           └─ 派(1) ─ …
                      └─ 香蕉(1) ─ …
  ```

  匹配 m=2(`我 爱`),预算 = 2,可以猜两个:

  | 草稿第几个 | 当前节点(=分母) | 它代表的上下文 | 孩子(=候选分子) | 本步比值 | 累积 prob |
  |---|---|---|---|--:|--:|
  | 1 | 爱(**3**) | `我 爱`(纯匹配串) | 苹果(2)/香蕉(1) → 选苹果 | 2/3 | 0.67 |
  | 2 | 苹果(**2**) | `我 爱 苹果`(匹配串**+已猜的苹果**) | 汁(1)/派(1) → 平手选汁 | 1/2 | 0.33 |

  看第 2 行:**"分母换人"不是重新查了什么,它就是"指针往下走了一步"的自动结果**——写完苹果,指针站进了苹果(2)节点,它代表的路径天然就是"匹配串+已猜串",于是分母自动从 count(`我 爱`)=3 变成 count(`我 爱 苹果`)=2。第 2 步的问句是"**在出现过 `我 爱 苹果` 的那 2 处里**,下一个接什么"——注意条件里包含了**还没被验证**的苹果,因为草稿本来就是"假设前面全对"往下铺的(这正是连乘=联合概率的另一面)。望远镜核对:第 2 位累积 prob = 2/3 × 1/2 = 1/3 = count(`我 爱 苹果 汁`)÷count(`我 爱`) = 1/3 ✓。草稿 = [苹果, 汁],score = 0.67+0.33 = 1.0(期望收 1 个)。
  换个视角看,这棵树就是一个**只会背 prompt 的 n-gram 语言模型**:每一步都条件在"全部已知前文"上给下一个 token 打分——和真 LM 的 P(next | 前文) 形状一模一样,只是 LM 用神经网络算,树用数数。
- **连乘有个漂亮的望远镜性质**:中间项全部约掉,累积置信度 = count(匹配串 + 整段草稿) ÷ count(匹配串)。数值例:`A B` 出现 4 次,其中 2 次接 `C`、那 2 次里 1 次再接 `D` → 两步比值 2/4、1/2,累积 0.5、0.25(= 1/4,正是 ABCD 次数 ÷ AB 次数);score = 0.5+0.25 = 0.75。
- **m 大时的实际形态**:一段 24-gram 在 prompt 里几乎必然只出现一次 → 匹配节点往下 count 全是 1 → 每个比值 1/1 = 1.0,**置信度恒 1.0,score = 草稿长度**——这就是"独苗路径 `min_token_prob` 掐不到"的另一面。带分数的比值(0.5、0.2……)只在 m 小、上下文多处出现时才出现。
- **一个结构边角**:m=24 时其实**猜不了**——树深封顶 24,匹配 24 个就站在窗口尽头,脚下没有孩子(源码里枚举干脆写成 `match_len < pattern长度`,连试都不试)。一般地"匹配 + 草稿 ≤ 24",配上预算 = m×1.0,单步草稿的结构上限是 min(m, 24−m) ≤ **12**(m=12 时最大);长段照抄区里 score 择优会自动选中 m≈12 附近的匹配。注意这个"和约束"来自**树深**(max_tree_depth),不是 `num_speculative_tokens`——后者只数草稿本身、不含命中的 m,且默认=树深,所以这根保险丝默认永远绷不紧。
  **为什么"和的上限"恰好严格等于树深?几何解释**:匹配的 m 个 token 和草稿的 k 个 token 连起来,正是**一条从根出发、长 m+k 的根路径**(匹配走到深度 m,草稿接着往下走)——而这棵树里任何根路径最长就是 max_tree_depth(每条窗口只喂了 24 个)。所以 m+k ≤ 24 是铁律,对单路径草稿和树状草稿的每条分支都成立。但"≤24"是**不可能超过**的上限;真要**打满** 24 还得三道闸都放行:①预算够(m×factor ≥ 24−m,factor=1 时即 m ≥ 12);②置信度沿途不跌穿 0.1;③那条路径在树里真的修到了 24 深(对应的出现位置得离序列结尾足够远,靠结尾的窗口是被截短的)。

**追问二:分叉处平手(苹果、香蕉都是 0.5)选哪个?**

选**兄弟链表的头**(`head_child`)。链表按 count 降序维护,count 相等时的先后是**实现细节**(先插入/先升到这个 count 的排在前面)——本例会选先插入的 苹果。这不是有原则的选择,也不需要是:两个 0.5 的赌注**期望收益完全相同**,选"错"了也只是草稿被拒、输出分毫不差(无损),所以代码不纠结,O(1) 拿链表头就走。真想"两个都猜"的机制其实存在——树状草稿(`use_tree_spec`,Draft 结构里的 `parents` 字段就是为它准备的,把苹果、香蕉两条分支都塞进草稿让验证挑),但 vLLM 0.18 没暴露这个开关,恒单路径。

**追问三:"置信度沿草稿连乘"到底是什么意思?**

先看清出草稿的动作:它不是一次判断,而是**从匹配节点出发、沿树往下走一串**,每往下走一格就往草稿里多写一个 token。"连乘"说的是陪着这次行走的一个变量 `prob` 的一生:

- 出发时 `prob = 1.0`;
- 走到**分叉口**(当前节点有多个孩子):选最高频的孩子,`prob ×= 该孩子count ÷ 当前节点count`(选中路线的"得票率");
- 走在**独苗路段**(只有一个孩子,count 相同):`prob` 原样平移(等于乘 1);
- 每写进一个草稿 token,就把**此刻的 prob** 记在它头上;
- 一旦 `prob < min_token_prob(0.1)` → 收手,草稿到此为止。

**为什么是乘,而不是每步各看各的?** 因为验证是按前缀斩断的(§四第 4 步):草稿第 3 个 token 想被接受,**前提是第 1、2 个都对**。所以"第 k 个 token 真能派上用场"的概率 = P(第1个对) × P(第2个对|第1个对) × … × P(第k个对|前面全对)——而每步的比值恰好就是对应条件概率的频率估计,**连乘 = 这串草稿到第 k 位全对的联合概率**。这也是望远镜性质的来源:中间项约掉,第 k 位的累积 prob = count(匹配串+前 k 个草稿) ÷ count(匹配串)(+ 是拼接;分子分母都直接读自树节点,见追问一的数字例子)。

数值走一遍(设匹配 `我 爱`,count 16;沿途节点 count 依次 8、8、4、2):

| 走到 | 分叉情况 | 本步比值 | 累积 prob | 动作 |
|---|---|--:|--:|---|
| 我爱(16) → 苹果(8) | 分叉 | 8/16 | 0.5 | 写入 苹果(0.5) |
| 苹果(8) → ,(8) | 独苗(count 没掉) | 8/8 | 0.5 | 写入 ,(0.5) |
| ,(8) → 你(4) | 分叉 | 4/8 | 0.25 | 写入 你(0.25) |
| 你(4) → 说(2) | 分叉 | 2/4 | 0.125 | 写入 说(0.125) |
| 说(2) → 过(1) | 分叉 | 1/2 | 0.0625 | **< 0.1,收手** |

两条顺手的规律/自检(本表初版的数字就是靠 ② 抓出错的):
① **count 沿路径单调不增**(孩子的窗口是父亲窗口的子集),而且**一旦掉到 1,后面全是独苗、比值恒 1/1、不再扣血**——count=1 的节点最多只有一个 count=1 的孩子,不可能再分叉。所以"扣血"只发生在 count 还大于 1 的前半段,越往深处走路径越"独"。
② **望远镜自检**:任何一行的累积 prob × 匹配节点的 count,必须等于"匹配串拼上前 k 个草稿"的出现次数——**是个整数**。第 4 行:0.125 × 16 = 2 ✓("我爱苹果,你说"出现 2 次)。若某行乘出来不是整数,说明这组数字在真实的树上不可能存在。

于是 `min_token_prob=0.1` 的真实含义清楚了:**"按 prompt 里的频率估,这份草稿到这一位还全对的概率至少要有 10%,否则后面的位置大概率是白占验证名额"**。也能看懂两个旧结论:①抄唯一出现的原文(独苗路径)全程乘 1、prob 恒 1.0,这参数永远掐不到它——report 快的根源;②分叉越散扣血越狠,出现 40 次、续写各不相同的"Dragon Age: Origins"们一两步就跌穿门槛。

最后,**score = 草稿各 token 的累积 prob 之和**,它不是随手的设计:第 k 位的累积 prob ≈ "第 k 个 token 被接受的概率",求和恰好 ≈ **这份草稿期望被接受的 token 数**(上表 score = 0.5+0.5+0.25+0.125 = 1.375,即"这份 4-token 草稿预计能收 ~1.4 个")。所以"score 择优" = 在各个 match_len 的候选草稿里,**选期望收益最大的那份**。

几个真实系统里的对应放大:预算公式 `min(24, m × max_spec_factor)`;深度 24 的树上,匹配 + 草稿总长 ≤ 24,整段照抄时草稿能滚到十几二十个 token(完整的滚雪球/微改写例子见 [18-vllm-suffix-decoding.md](18-vllm-suffix-decoding.md) §四的手算表)。

## 五、多请求:本地树 + 全局树(广义在这里用上)

vLLM 里同时存在两种树(源码 `cache.py`):

- **本地树:每个在跑的请求一棵**,里面只有一条序列 = 该请求的 prompt + 已生成(所以其实用不到"广义");请求结束整棵删掉。
- **全局树:整个引擎只有一棵**,装的是**最近 1 万个请求各自的输出**(prompt 不进全局树)——这才是真正的广义后缀 trie:每个请求的输出是一条独立序列(有自己的 seq_id),各自贡献 ≤24 窗口,**count 跨序列累加**。

广义的两个好处,拿 §三那棵树现场演示。假设另一个请求 B 的输出是 `他 爱 苹果`,把它作为第二条序列喂进同一棵树(窗口:`[他 爱 苹果] [爱 苹果] [苹果]`):

```
根下的变化:  爱(2) → 爱(3)      它的孩子: 苹果(1)→(2), 香蕉(1)
             苹果(1) → 苹果(2)   新增: 他(1) ─ 爱(1) ─ 苹果(1)
```

现在再有人拿 `爱` 来问"后面接什么",投票变成 **苹果 2/3 ≈ 0.67 vs 香蕉 1/3**——B 的经历改变了对 A 类问题的猜测,**这就是"抄别人/抄历史"的机制**(也是实验里"热树"现象的根源:树背过的答案,重测时命中虚高)。而当 B 被挤出缓存(FIFO),`remove(seq_id=B)` 把 B 贡献的窗口逐一撤销(count 减回去、独有的分支剪掉),A 的部分毫发无损——**多条序列不是拼接成一条**(拼接会在接缝处造出"A 结尾+B 开头"的假子串),而是共树分账。

## 六、放回 vLLM 全景(一段话)

每个 decode step:模型采样出新 token → 追加进本地树和全局树(树随输出增长,"抄自己前文"也能命中)→ 取结尾 ≤24 当 pattern,本地树、全局树**各猜一份草稿,score 高者胜** → 草稿排进下一步,一次前向验证,接受的最长正确前缀 + 1 个 bonus/纠正 token。参数、调用链、每行源码在哪,见 [18-vllm-suffix-decoding.md](18-vllm-suffix-decoding.md);为什么"重叠大不等于加速",见 [why-high-overlap-no-speedup.md](suffix-spec-decode/docs/why-high-overlap-no-speedup.md)。

## 八、进阶:教科书里的"后缀树"和 SAM,跟这棵树什么关系

> 这一节回答"它和标准结构的区别",跳过不影响用 vLLM。

**教科书后缀树 = 后缀 trie 的极致压缩。** 不限深,而是把所有独苗链压成边、边只存 (起,止) 引用,节点数压到 O(n)(叶子=后缀,内部节点=出现≥2次且续写有分歧的子串)。banana 的例子($ 是防止"后缀藏在半路"的终结符):

```
根 ├── "banana$"
   ├── "a" ──┬── "na" ──┬── "na$"      ← 后缀 anana$
   │         │          └── "$"        ← 后缀 ana$
   │         └── "$"                   ← 后缀 a$
   └── "na" ──┬── "na$"                ← 后缀 nana$
              └── "$"                  ← 后缀 na$
```

O(n) 时间建它要 **Ukkonen 算法**(靠 suffix link 跳边,出名难写)。强在"**不限长度**的任意 pattern 查询、最长重复/公共子串"这类分析问题。

**SAM(后缀自动机)= 另一条路线,不是树是 DAG。** 思想:给所有子串按 **endpos(出现位置的结尾集合)**分等价类,行为相同的子串并成一个状态。banana 里 `na` 和 `ana` 的 endpos 都是 {4,6} → 同一个状态;状态数 ≤2n-1,O(n) 在线构造且好写。它的看家本领是**流式匹配**:对另一个文本逐字符走,始终维护"当前位置结尾的、在 S 中出现过的最长后缀"——我们 overlap 分析的 `build_matchlen` 用的正是它(SAM 的 (v,l) 状态机)。深层关系:SAM 的 suffix link 树 = 反串的后缀树,两者一体两面。

**为什么 vLLM 哪个都不用?** 后缀树和 SAM 都是为"**一条固定的串**,建一次、只读查询"设计的;vLLM 的需求恰好全在它们的软肋上:

| 需求 | 教科书后缀树 | SAM | vLLM 的限深 trie |
|---|---|---|---|
| 限深 24 | 难(结构性侵入) | 无意义 | 弹队头,一行代码 |
| 多序列共树 | 广义后缀树(拼接+终结符,繁) | 广义 SAM(难) | 天然(count 累加) |
| 删除一条序列 | 难 | 几乎不可能 | `remove`:按 endpoints 撤销 |
| 实时频次 | 叶子计数需预处理 | cnt 需拓扑序 | count 建树时白送 |
| 逐 token 在线追加 | Ukkonen 可(难写) | 可 | 天然 |

pattern 恒 ≤24、序列一直在增删、要频次——**这组需求下,"朴素"就是最优解**。这也是读工业代码的一个常见教训:名字叫 SuffixTree,不等于教科书里的 suffix tree;先看需求再对结构。

**追问:SAM 有 suffix link、空间 O(n),不是比 O(24n) 的 trie 更优秀吗?** 四点拆开:

1. **空间优势基本是错觉。** SAM 的 "≤2n 状态" 是相对**不限深后缀 trie 的 O(n²)** 说的;限深已经把 n² 砍成 O(24n),再加路径压缩(窗口共享前缀、节点只存对原序列的引用)实际也是 ~O(n) 量级。两边真正的内存大头都一样:每个节点/状态的哈希表 + 原序列本身。
2. **count 才是命门。** speculate 每步都要"这个 context 后面接过什么、各几次"的**精确、实时**频次。trie 里这是建树白送的(每 token 只有 ≤24 个活跃指针各 +1,局部、O(24));SAM 的 cnt 标准做法是**冻结后按 len 拓扑序传播**——想在线维护,每追加一个 token 得沿 suffix link 链整条 +1(链长可达 O(n)),clone 时还要拆分计数,是重型机械。注意**连最友好的场景都栽在这**:本地树是单串、只在尾部追加——恰好是 SAM 在线构造的主场,可没有在线 cnt,speculate 还是没法投票。
3. **删除无解。** 全局树要 FIFO 驱逐单条序列:trie 逐窗口 count−1、剪枝即可;SAM 删一条串意味着 endpos 等价类要**重新拆分合并**,没有实用的减量算法。何况广义(多串)SAM 的构造本身就出名地微妙(clone 边界条件,流传的模板一半有坑),再叠加在线 cnt 和删除,等于自研一个研究课题。
4. **suffix link 能买到的太少。** link 的价值是把"每步从根重新匹配 pattern"的 O(24²) 摊成 O(1)(增量维护 (v,l),沿 link 链还能顺手枚举各 match_len)。但 O(24²) 本来就是几百次指针跳转、微秒级,在 ~30ms 一步的 decode 里是零头;而 link 换不来第 2、3 条里的任何一个。**真正的税大头是关异步调度(≈18%),树再省也省不到账面上。**

一句话:SAM 的优秀属于"**一条冻结的串、任意长 pattern、只读查询**"的赛道——我们离线 overlap 分析的 `build_matchlen` 用它,用对了地方;vLLM 的赛道是"**边长、边查、边删的多串 ≤24-gram 频次表**",这里 trie 的每一分朴素都花在刀刃上。
