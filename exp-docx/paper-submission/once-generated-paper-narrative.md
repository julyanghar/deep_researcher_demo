# Once Generated, Many Times Served —— 论文叙事稿

> **定位**：按论文行文顺序、用连贯叙述把故事讲完的**叙事稿**（中文），供成稿时直接翻译/扩写。骨架版（特征/缺口/弱点清单）见 [Once Generated.md](Once%20Generated.md)，零基础通俗版见 [once-generated-explained.md](once-generated-explained.md)。
> **数字口径**：无标注的数字均为已实测（100 题 trace，1,912 次生成调用，出处见骨架版）；`[待测]` 为预注册未兑现的实验。

## 〇、各节写法蓝图（怎么写；下面的正文是照此写出的示范）

### Abstract：六句结构，每个形容词配一个数字

1. **设定+痛点**（1 句）：负载是什么、为什么贵（调用数/token 量/分钟级延迟）；
2. **刻画贡献**（1-2 句）：首个 trace 级刻画 + 定义性特征命名（verbatim content recycling）+ 一两个最惊人的数字（90.7%、1.3 万字符/次）；
3. **结构洞察**（1-2 句）：两条正交轴（量异质→必须路由；形态逐字→字面机制够用）+ 口径教训（r=−0.23）——这是把"观察"升格为"设计依据"的桥，abstract 里不能省；
4. **机制**（1-2 句）：写侧收什么钱、读侧收什么钱、两侧怎么耦合（税对冲一句话）；
5. **结果**（1 句）：只放已兑现数字（−47.5%、2.37×、质量无损），未测的标 `[待测]` 或不进 abstract；
6. **负结果**（1 句）：划适用边界=可信度来源，这是本文风格的一部分。

坑：❌ 把期货写成成果；❌ "first" 不带限定词（要写成"首个 **trace 级内容**刻画"，related work 里能防住的措辞才敢放 abstract）。

### Introduction：五段式，每段一个职责、段末一个钩子

| 段 | 职责 | 必落的数字 |
|---|---|---|
| P1 | 设定与成本：谁在乎这个负载 | 5–30 分钟/次；~19 次调用；2 万 token prompt；墙钟三分解 |
| P2 | 观察+**一句话立反派** | "内容创作一次、计费三次"；90.7%、13,432 字符 |
| P3 | 刻画贡献（全文最长段，承重墙） | 两轴各自的数字（双峰 8–88%；22.9% vs 7.7%）+ 口径教训（−0.23 vs 0.90） |
| P4 | 设计：一个性质两侧收钱 + **为什么必须在一篇论文里**（税对冲） | 61.2%、2.37×、−51~67%、+28%/+14% |
| P5 | 结果 + 贡献列表（4 条，每条一行，期货标注） | −47.5%、质量 0.370 vs 0.366 |

坑：❌ P3 写成数字罗列——每个观察后面必须跟"这决定了什么设计"；❌ P4 只说"我们做了 A 和 B"而不回答"为什么 A+B 必须同框"（审稿人最常见的拒稿理由是 stapling）。

### Method：observation-driven，两章一体（§3 刻画 + §4 设计）

- **§2 背景只放两件事**：调用图（再循环的结构来源）+ decode 占 95%（全部取舍的锚点）。不写 tutorial，不解释什么是 KV cache（审稿人都懂）。
- **§3 刻画**：每小节 = 一个观察 + 段末一句"这决定了 §4 的哪个设计"。纪律：**刻画节的每个数字都必须在 §4 被消费一次，消费不掉的删**。负结果（3.5）放正文不放附录——它们是边界不是瑕疵。
- **§4 设计**：每小节开头一段人话讲机制，紧跟"它为什么由 §3 的哪条轴支撑"，再给关键工程发现（五常数、锁竞争）。**4.3 耦合必须单列成小节**——它是反 stapling 的论证核心。机制推导细节（五常数拟合、blend 位置修正算法）全放附录，正文只留结论与直觉。
- 坑：❌ 把 §3 和 §4 写成两篇小论文（靠"这决定了…"句式焊死）；❌ 机制新颖性措辞过宽（特化 head 的可认领交集=agent 真实轨迹 × post-hoc EAGLE 系 × 特化训练，一个限定词都不能少）。

### Experiments：主表 + 诚实拆账 + 税对冲图 + 预注册

1. **主表**：端到端三配置（vanilla/投机/投机+分节）× 列（e2e 墙钟、报告段、TTFT、质量分）——**加速和质量永远同表出现**；
2. **消融=诚实拆账表**：每行一个机制（suffix / enforce-eager / 分节 / blend / 路由），标注每项收益的**性质**（"eager 是系统修复非算法""分节收益来自并发、命中率反降 61.2→46.2"）——审稿人自己拆账不如你替他拆好；
3. **税对冲单独一张图**：TTFT 三线（vanilla / +eager / +eager+blend），这张图独自承担"两侧必须合体"的论证；
4. **口径与测量纪律小节**：batch=1 回放 vs 并发口径、热树污染、TP worker 重复计数——把踩过的坑写成方法论，反而加分；
5. **预注册实验**：判据先行（特化 head 15/20/35%；路由并发 oracle >5%），失败也报告——写明这句话本身。
- 坑：❌ 把 +28% 混进"我们的方法"不拆账；❌ 质量评委不写选择理由（要写：kimi-k2.5 自漂移 0.018 vs qwen-flash 0.236，后者不可用）。

### Related Work / Limitations（一句话原则）

- RW：按"层"组织（调度层→内容层）而不是按论文列表；每个最近邻给**一句话可辩差异**（Plato=位置相关前缀累加 vs 我们位置无关；RelayCaching=机制叙事 vs 我们刻画驱动+质量验证+税对冲）。
- Limitations：抢在审稿人前面自曝（单框架、batch=1、prompt 工程质疑），每条都配"我们做了什么/预注册了什么"。

## Abstract

Deep research agent——搜索、阅读、并最终写出带引用长报告的多智能体系统——正在成为大模型最重的推理负载之一：一次任务动辄数十次 LLM 调用、数十万 token 吞吐、数分钟到半小时的端到端延迟。我们在 **LLM 调用层**对一条生产形态的 deep research 流水线做了首个**内容级** trace 刻画（100 个研究任务、1,912 次生成调用），发现其定义性特征是**逐字内容再循环（verbatim content recycling）**：一段内容在流水线中只被生成一次，随后却被系统按"全新内容"反复计费——下游调用把它整段读入（prefill 重算 KV），又把它大面积逐字誊写进自己的输出（decode 逐 token 重新生成）。报告类调用平均 90.7% 的输出字符逐字来自其输入，平均每次照抄 1.3 万字符；而这份照抄，现有 serving 栈既不识别、也不利用。

刻画进一步揭示了再循环的两条正交结构：**量的异质性**——照抄比例在调用间剧烈分化（报告类均值 90.7% 近乎誊写；小结类逐题均值从 8.0% 摆到 88.2%，部分近乎原创），意味着任何单一加速机制都无法通吃，必须按重叠率分派策略；**形态的逐字性**——内容一旦跨调用复现，几乎全部以逐字形式出现（逐字口径 22.9% vs 语义改写口径仅 7.7%），意味着轻量的字面匹配机制即可接住全部可复用部分，无需语义级近似。我们还纠正了一个容易误导后续工作的口径陷阱：字符级覆盖率与实际加速**负相关**（r=−0.23），真正预测收益的是连续可收割 token 量。

基于刻画，我们构建了首个面向 deep research 负载、同时在 decode 与 prefill 两侧利用再循环的 serving 系统：**写侧**，以后缀匹配投机解码收割照抄段（并以逐步成本模型给出草稿深度的下注经济学），以 agent 自身轨迹特化训练的 draft head 补足改写段 `[待测]`，按调用重叠率分派策略；**读侧**，将上游调用 decode 时已算出的 KV 以内容寻址方式直接供下游 prefill 复用，其收益恰好缴回写侧加速引入的 TTFT 税，使两侧构成互补整体而非独立技巧的叠放。在 40 题 DRBench 上，写侧机制已将端到端墙钟从 9,293s 降至 4,879s（−47.5%），报告段解码 2.3×+（两次独立 run 2.32–2.37×），质量盲评无损；完整系统的合体评估 `[待测]`。我们同时报告一组诚实的负结果——通用 draft head 因文体失配全面失效、字符覆盖率的误导性、以及兄弟调用互抄不可利用——为"复用式加速在哪适用、在哪不适用"划出边界。

## 1 Introduction

**（P1：设定与成本。）** 以 OpenAI Deep Research 为代表的深度调研智能体，把"回答一个问题"扩展成"完成一次调研"：系统自主规划子问题、并行检索与阅读、逐步汇总，最终产出数千字带引用的研究报告。这类系统的内部结构像一个编辑部：规划者（supervisor）拆解问题，多个研究者（researcher）并行检索并各自写出小结，写作者（writer）汇总全部小结写成终稿。代价是推理量的爆炸：官方口径单次查询 5–30 分钟；我们实测的流水线上，一次任务平均 19 次生成调用，写作者单次调用的输入即达约 2 万 token；端到端墙钟中，小结生成占 32.8%、报告生成占 27.8%、检索等待占 31.8%——推理（而非检索）占了约六成。对这样的负载，serving 层的每一分优化都直接兑换成用户可感的分钟级延迟。

**（P2：观察与反派。）** 我们的出发点是一个被现有 serving 栈完全忽略的结构性事实：**deep research 的输出不是从零创作的，而是沿调用图对系统内已有内容的再循环**。检索片段被研究者逐字摘入小结；小结被写作者整段读入，又被逐字誊写进报告。同一段 token 的一生要被系统计费三次以上：生成它付一次 decode 全价，下游读它付一次 prefill 全价，下游重抄它再付一次 decode 全价。而现有推理引擎把每次调用当作独立请求、把每个输出 token 当作从零生成——**内容只被创作了一次，账单却按"每次都是新的"来收**。这就是本文标题的含义，也是本文要消灭的浪费。

**（P3：刻画贡献——两条轴与一个口径教训。）** 要利用再循环，先要量化它。我们对 100 个任务、1,912 次生成调用做了首个**在 LLM 调用层、以内容为对象**的 deep research trace 刻画（2026-07-11 四路对抗核查后的安全措辞——三类先例必须显式划界：已有的 trace 级 agent 刻画覆盖 coding、ReAct 与记忆系统三种对象但全部止于系统层指标，其中 TraceLab 明确丢弃文本内容；已有的内容重叠测量位于**别的层**——搜索查询日志层、最终报告质量层与单轮 RAG prompt 层；照抄率度量本身承自摘要 extractiveness 与 copy coverage/density 传统，我们首次将其用于刻画生产 deep research 流水线的逐调用负载，并与推理成本结构联合分析）。刻画的第一层结论是再循环体量惊人且随任务深度增长：报告类调用的输出平均 90.7% 的字符逐字来自输入（把"照抄"的门槛提高到连续 30 字以上，仍有 74.5%；最长单段逐字誊写 694 字）；系统从单轮升级为多轮迭代后，照抄率进一步从 86.9% 升至 90.7%。第二层结论更关键，它决定了系统该怎么设计——再循环有两条**正交**的结构轴。其一，**量分调用剧烈异质**：小结类调用的照抄率呈双峰分布、逐题从 8.0% 摆到 88.2%——有的调用在大段摘录，有的在做真正的压缩转述与观点归纳；这意味着"要不要用字面复用、用多深"必须按调用分派，任何全局开关都会在负载的一半上亏钱。其二，**形态一旦发生则逐字**：把同题小结两两对照，逐字口径的重合达 22.9%，而把"换个说法"也计入的语义口径反而只有 7.7%——跨调用的重复内容里几乎不存在"同义改写"，要么原样照抄、要么全新创作；这意味着后缀树、内容哈希这类**精确匹配**机制即可接住全部可复用部分，无需引入 embedding 检索式的近似匹配。此外我们报告一个会误导后续工作的口径陷阱：直觉上最自然的字符级覆盖率（Coverage@8）与小结段实际加速**负相关**（r=−0.23）——它把 8–30 字的碎片也计入"照抄"，而碎片对复用式加速无肉可吃；真正与收益正相关的是**连续可收割 token 量**（小结段 r=0.55，合并口径 r=0.90）。我们据此主张：刻画与优化复用式加速时，应以可收割量为一等指标。

**（P4：设计——一个性质，两侧收钱。）** 刻画直接给出了系统设计。同一个再循环性质，在推理的两个阶段各留下一份可收割的浪费，我们两侧都收：**写侧（decode）**，照抄段的下一个 token 大概率已经躺在上下文里——我们用后缀匹配投机解码把它们成段收割（报告段命中率 61.2%、解码 2.37×），并基于逐步成本的五常数模型（七种配置预测残差为零）给出**下注经济学**：单个草稿位置成本 0.62ms、盈亏线接受率 2.4%，高重叠段敢发深注、低重叠段浅注或不注；对字面机制无肉可吃的改写段，我们在 agent 自身轨迹上特化训练 draft head 作为第二臂 `[待测]`——现成通用 head 已被实测证伪（接受率 10.7–20.3%，20 题无一胜过后缀匹配，且败因被定位为文体而非语言：接受率与中文占比相关 −0.60），这一负结果恰好构成"必须特化"的动机。策略分派以调用重叠率为信号：类型标签给出先验（报告≈抄、小结≈半抄），类内的双峰异质性则需要每调用的重叠估计 `[待测：并发口径 oracle 上界]`。**读侧（prefill）**，被下游读入的小结，其 KV 在上游 decode 时已经算过一遍——我们把生成内容的 KV 按内容哈希归档，下游 prefill 命中逐字相同内容时直接复用、仅对位置敏感处选择性重算（大 prompt 档 prefill 时间 −51~67%）。与依赖"固定顺序前缀累加"的前缀缓存不同，内容寻址使复用**位置无关**：小结在报告 prompt 中的位置与其生成时完全不同，前缀缓存对此无能为力，而这正是最近的多智能体 serving 工作（如 Plato）明言做不到的情形。两侧不是两个独立技巧的叠放：写侧最强的单项优化（消除驱动层锁竞争，端到端 +28%）以 TTFT +14% 为代价，而读侧的 prefill 节省恰好把这笔税缴回——**decode 加速付 prefill 税，prefill 复用缴回税**，两侧合体才构成收支闭环。

**（P5：结果与贡献。）** 在 40 题 DRBench 上，写侧机制已将端到端从 9,293s 压至 4,879s（−47.5%），报告质量盲评无损（RACE，kimi-k2.5 裁判，0.370 vs 0.366）；KV 复用质量 40 题盲评无害；完整系统合体与并发口径评估为预注册实验 `[待测]`。本文贡献：
1. **刻画**：deep research 负载在 LLM 调用层的首个 trace 级内容刻画——逐字再循环的体量、两条正交结构轴（量异质/形态逐字）、随深度增长的趋势，以及"字符覆盖率误导、可收割量才预测收益"的口径教训；
2. **写侧机制**：重叠率驱动的投机解码——后缀匹配主臂 + 成本模型定价的下注深度 + agent 轨迹特化 draft head 第二臂 `[待测]`，附五常数逐步成本模型与驱动层锁竞争的发现与修复（+28%）；
3. **读侧机制**：内容寻址、位置无关的 decode-to-prefill KV 复用，及其与写侧的税对冲耦合；
4. **诚实负结果**：通用 draft head 的文体失效、字符覆盖率的误导性、兄弟调用互抄之不可利用、改写段的接受率天花板——为复用式加速划出适用边界。

## 2 背景：deep research 的推理成本结构

**2.1 流水线与调用图。** 一次 deep research 任务是一张有向的调用图：supervisor 产出检索计划（query_plan），多个 researcher 并行执行"检索→阅读→写小结"（本系统平均每任务 ~18 次小结调用），writer 读入全部小结产出终稿；多轮系统还会按覆盖缺口追加检索轮次。图上每个节点是一次独立的 LLM 调用，节点间以**文本**为介质传递中间产物——正是这个"以文本为介质"的设计，造成了内容在图上的反复流动：上游节点的输出（decode 产物）原封不动成为下游节点的输入（prefill 对象）。

**2.2 成本在哪里。** LLM 推理分两个阶段：prefill 并行读入 prompt 并为每个 token 计算 KV（决定首 token 延迟 TTFT），decode 逐 token 自回归生成（每 token 一次完整前向）。本负载中 decode 占 GPU 计算约 95%——写作者单次 prompt 约 2 万 token 看似庞大，但报告本身要逐 token 写出数千 token，后者才是大头。这个比例是全文设计取舍的锚点：**优化重心必须在 decode 侧；prefill 侧优化单独看撬不动大盘（端到端仅 −8~10%），它的价值必须在与 decode 侧的耦合中体现**（§4.3）。

## 3 负载刻画：逐字内容再循环

本节回答三个问题：再循环有多大（§3.1）、它的结构是什么（§3.2–3.3）、以及怎样的度量才能指导优化（§3.4）。所有测量基于 100 个任务的完整 trace（1,912 次生成调用），指标为 Coverage@k——输出中位于"与输入逐字相同的 ≥k 字连续片段"内的字符占比。

**3.1 体量：输出大面积逐字来自输入，且随任务深度加深。** 报告类调用（n=100）Coverage@8 均值 90.7%、中位 93.1%，全部调用超过 50%；门槛提至 ≥30 字连续片段仍有 74.5%，最长单段逐字誊写 694 字，平均每次调用照抄 13,432 字符（最多 31,122）。小结类调用（n=1,812）均值 53.8%。逐字誊写不是流水线的意外副产品，而是任务的内生需求：报告必须忠实转述检索证据与小结结论，引用越规范、誊写越多——系统从单轮升级为多轮迭代后，报告照抄率从 86.9% 升到 90.7%，照抄超 90% 的报告占比从 36% 升到 71%。**agent 越"deep"，再循环越强**——这决定了本文的优化随负载复杂度增值而非贬值。

**3.2 第一条轴——量：照抄比例分调用剧烈异质。** 均值掩盖了结构。小结类调用的 Coverage 分布呈双峰（80–90% 区间是最大桶），逐题均值从 8.0% 到 88.2%：检索摘录型的小结在大段誊写原文，分析归纳型的小结在做真正的压缩转述——小结整体对输入的 token 级长段照抄（cov24）仅 0.4%，说明其"照抄"多为碎片、成段誊写稀少。这条轴直接否决了"全局开一个加速开关"的设计：同一机制在照抄型调用上是收割机、在改写型调用上几乎空转（我们实测：后缀投机在报告段 2.29–2.37×，在小结段仅 0.93–1.07×——冷树回放到修复后热树 e2e 的口径区间——基本持平）。**策略必须按调用的重叠率分派**——这是 §4.1 路由设计的直接依据。

**3.3 第二条轴——形态：复现一旦发生，是逐字的。** 把同题的小结两两对照（18,975 对），逐字口径（out_contain8）均值 22.9%、37.3% 的配对超过 30%；而把"换个说法的相同内容"也计入的语义口径（out_jac5）反而只有 7.7%。跨调用的重复几乎全部以原样照抄的形态出现，"同义改写型重复"在本负载中近乎不存在。这条轴决定了**机制选型**：精确匹配（后缀树、内容哈希）即可接住全部可复用内容，无需语义向量、近似检索这类更重且引入错误率的机制。反面证据同样吻合：让写作者分节改写报告，字面复用立刻被破坏（后缀投机命中率 48.4%→38.9%）——改写会杀死复用，所以利用应在 serving 层进行，而非寄望于应用层改写（§7 进一步讨论"为什么不改 prompt"）。

**3.4 口径教训：覆盖率好看 ≠ 有收益。** 若刻画止步于 §3.1，会得出危险的错误结论："小结照抄过半，也该加速"。实际上小结的 53.8% 大半由 8–30 字的碎片贡献（@30 口径骤降至 9.5%，最长片段均值仅 51 字），而碎片对复用式加速无肉可吃——每段碎片都需重新建立匹配、草稿尚未养长片段已断。后果是可测量的：字符级 Coverage@8 与小结段实际加速相关 **r=−0.23（负）**，而 token 级连续可收割量与加速正相关（小结段 r=0.55/Spearman 0.79，合并口径 r=0.90），打平点约 19%。我们据此把**可收割 token 量**立为刻画与调度的一等指标，Coverage 仅作内容画像的辅助口径。这一教训本身即是刻画贡献的一部分：它解释了后缀投机在报告/小结两段 61.2% vs 28.8% 的命中分裂，也为后续工作提供了指标选型的避坑指南。

**3.5 边界（负结果）**。刻画同样告诉我们哪里**没有**可收割的结构：并行兄弟小结间确有内容冗余（同 researcher 配对逐字重合 34.6%），但其 token 级长段口径仅 0.69%、投机接受率对兄弟数完全不敏感（29.2%→29.5%）——**兄弟互抄不可作为草稿源**，冗余只能在检索与 prefill 侧利用。改写段的接受率天花板由内容的信息论性质决定，机制无法突破（树 oracle 上界仅 +11%）。这些边界在 §4 的设计中被显式尊重。

## 4 系统设计

设计原则一句话：**同一个再循环性质，在写侧收"重抄"的钱，在读侧收"重读"的钱，用重叠率把钱分对地方。**

**4.1 写侧：重叠率驱动的投机解码。** 主臂是后缀匹配投机：以已生成内容的结尾为模式，在 prompt 与已生成文本上建后缀索引，匹配处的后继 token 作为草稿，target 模型单次前向批量验证——验证保证输出与自回归逐字相同（无损），照抄段一步可收割多 token。它在照抄段的效率来自 §3.3：复现是逐字的，精确匹配足够。两项系统贡献使其在生产口径下站稳。其一，**逐步成本的五常数模型**：每步耗时 = graph 基线 24.0ms + eager 附加 0.5 + 投机管线 1.9 + 混合边界税 8.1 + 草稿激活 5.8 + 0.62ms×草稿位置数，七种配置预测残差为零。模型把"发多深的草稿"从启发式变成算术：单位置 0.62ms、盈亏线接受率 2.4%，报告尾部段仍有 5–9% 的接受率——**敢发深注**；小结的大多数解码步无肉可收（整体命中率仅 28.8%），应浅注或不注。其二，**混合边界税的定位与消除**：8.1ms 的大头是 CUDA graph 段与 eager 发射交错触发的驱动私有锁等待（cuLaunchKernel 内 4.5ms/步）；`enforce-eager` 一个参数消除之，端到端中位 +28%，代价 TTFT +14%——这笔税由读侧缴回（§4.3）。对字面机制无肉可吃的改写段，第二臂是**在 agent 自身轨迹上特化训练的 draft head** `[待测]`：现成通用 head 的失败（§1）不是"不够好"而是文体级失配，而 agent 负载天然自产 on-policy 训练数据（target 模型自己的历史输出）；预注册判据为接受率 15%（保本）/20%（胜过主臂的小结段贡献）/35%（独立机制贡献）。策略分派信号分两层：调用类型标签免费给出先验；类内双峰（§3.2）需要每调用重叠估计，其价值上界在 batch=1 口径下有限（完美路由仅将小结段 0.93×→1.05×），我们预注册在并发口径下重算上界、超过 5% 才引入学习型估计器——并发下劣质草稿付真钱（接受率 4.9% 的 ngram 草稿在并发下把小结段拖到 0.75×），分派在那里从优化变成止损。

**4.2 读侧：内容寻址的 decode-to-prefill KV 复用。** 观察很简单：模型 decode 生成小结时，已为每个生成 token 算过 KV；下游 writer prefill 同一段文本时，现行系统将其全部重算。我们在生成结束时把生成段 KV 按**内容哈希**切块归档（请求级 opt-in，生成段独立成块，取侧零改动自然命中），下游 prefill 命中逐字相同内容时直接装载，仅对位置编码敏感处做偏移与选择性重算。**位置无关**是与前缀缓存的本质区别：前缀缓存要求复用内容以相同顺序出现在 prompt 头部，而 deep research 的 writer 以任意顺序、任意子集读取小结——这正是依赖固定前缀累加的方案（Plato 明言）做不到的情形，也是 §3.3"复现是逐字的"使内容哈希可行的直接应用。实测：大 prompt 档（5–12k token）prefill −51~67%（writer 读入 3.044s→0.998s）；小档因固定搬运开销净亏，故挂 prompt 长度门控（crossover 4–6k，一个 if，我们如实称之为 if）。质量：40 题盲评复用与否无差。

**4.3 耦合：TTFT 税对冲。** 单看端到端，读侧只值 −8~10%（decode 占 95%，§2.2）——若止步于此，它只是一个可有可无的配菜，系统也将被质疑为"两个技巧的装订"。耦合点在于：写侧最大的单项收益（enforce-eager，+28%）以 TTFT +14% 为代价，对交互式负载这是可感的回退；而读侧在大 prompt 档的 prefill 节省恰好覆盖这笔税。**写侧欠的 TTFT 由读侧偿还**——去掉任何一侧，收支闭环都不成立。这是"必须在一个系统里同时做两侧"的论证核心，消融实验按此设计 `[待测：suffix+eager+blend 合体，报 e2e/TTFT/质量三线]`。

## 5 评估

**已兑现**（40 题 DRBench，Qwen3-32B，TP4）：
- 端到端：vanilla 9,293s → 投机 6,540s（−29.6%）→ 投机+报告分节 4,879s（**−47.5%**）；报告段解码 2.32×（本 run 口径；六格重跑口径 2.37×）；
- 质量：报告 RACE 盲评（kimi-k2.5 裁判）0.370 vs 0.366，无损；KV 复用 40 题盲评无害（复用臂 28.2–30.3% vs 不复用 24.8%）；
- 归因诚实拆账：−47.5% 中，投机贡献主要落在报告段（−52%），分节的增量收益来自并发而非命中率（命中率反降 61.2%→46.2%）；enforce-eager 的 +28% 是系统修复而非算法——消融表按机制逐项拆开呈报。

**预注册待兑现** `[待测]`：三件合体 e2e 与 TTFT 税对冲验证；batch>1 并发口径（吞吐/尾延迟，及并发下策略分派的止损价值）；特化 head 按 15/20/35% 判据；第二框架（GPT Researcher）与第二 base 模型上复测刻画核心表——回应"单框架 artifact"质疑；固定检索重放下的 KV 复用质量终裁。

## 6 相关工作

**负载刻画**：trace 级 agent 刻画已有三家先例——TraceLab（coding agent，且明确脱敏丢弃文本、只留字符数）、Agentic AI Workload Characteristics（ReAct 式 agent，含 decode-dominated 与上下文增长结论）、Agent Memory（记忆系统），全部止于系统层指标；deep research 相邻对象上的内容重叠测量则在别的层——Agentic Search in the Wild 在**搜索查询日志层**测得 token 级词项重叠 CTAR≈54% 及其随轮次演化（但其自认对 LLM 侧不可见）、Understanding DeepResearch via Reports 在**最终报告质量层**测段落间冗余、Byte-Exact Deduplication 在**单轮 RAG prompt 层**测字节级重复；AgentInfer 在 deep research 对象上测过 KV 命中率随上下文塌陷（92%→15%）。逐调用照抄率的**度量**承自摘要 extractiveness（Grusky 2018 起）与 copy coverage/density。本文占据的交集——**LLM 调用层 × 内容级照抄/冗余统计 × 与 serving 成本结构联合分析**——上述工作均未进入；新颖性押在交集与联合分析上，不押任何单项。

**多智能体推理优化**（Parrot、Autellix、Teola、Ayo 等）聚焦调度与编排层——请求间依赖、算子流水、KV 生命周期；本文工作在其下一层，利用的是**内容**级结构，与调度层正交可叠加。**Plato** 与本文最近：同样利用 agent 输出图做并行与 KV 复用，但其 KV 复用严格依赖固定顺序的前缀累加（论文自陈位置相关、无法重排子集）；本文的内容寻址复用位置无关，且增加了 token 级投机收割，两者一个做"读"的编排、一个做"读写"的内容复用。**投机解码**：SuffixDecoding 证明了 agentic 输出自重复可作草稿源，本文将其置于刻画驱动的路由框架内并给出成本模型定价的深度控制；SpecDec++/BanditSpec 学习"草稿发多长/选哪个 drafter"，其信号是运行时接受反馈，本文的信号是负载内容结构（重叠率），二者互补。**draft 特化训练**：DistillSpec/OSD/Aurora 在通用流量或 RL rollout 上（继续）训练 drafter；据我们所查，在 **agent 自身真实轨迹**上特化 post-hoc EAGLE 系 head 尚无已发表工作（PayPal 的部署报告将其明确列为 future work）——本文的特化臂 `[待测]` 占据该交集，无论正负结果均报告。**KV 复用**：CacheBlend/KVCOMM/DroidSpeak 聚焦 prefill 侧跨请求复用；decode 产物 KV 供下游 prefill 复用的机制叙事最近由 RelayCaching（preprint）提出，本文的差异在于刻画驱动（正向口径量化可复用量 `[待测]`）、质量验证、以及与写侧的税对冲耦合。**RAG 加速**：LLMA 早已观察到 RAG 输出照抄检索文档；本文在多智能体、多跳、长报告的负载上给出其定量画像与两侧联合利用。

## 7 局限与讨论

**为什么不在应用层消灭照抄？** 我们实测了这条路：去冗余 prompt 使小结输出 −18.7%、报告 −28.9%——照抄中确有可消除的铺垫；但被砍掉的是骨架与复述，**事实与引用完整保留**（照抄的主体是必须忠实转述的检索证据，是引用规范的内生需求而非啰嗦）；且分节改写实验显示，消照抄的改写同时杀死复用（命中率 −10pp）。应用层减肥与 serving 层提速是互补而非替代，且后者不依赖对 agent 内部的控制权。

**外部效度。** 全部刻画来自单一流水线、单一 base 模型、中英混合语料；照抄率的绝对值可能受 prompt 模板影响。第二框架/第二模型复测在预注册清单中 `[待测]`；我们同时论证再循环的根源（以文本为介质的调用图 + 引用忠实性）是这类系统的结构性质而非实现细节。

**口径。** 投机数字的主体为 batch=1 回放口径；并发口径（serving 论文的标准口径）为预注册实验。我们把测量纪律（热树污染、TP worker 重复计数等已踩过的坑）作为附录呈报。

**改写段天花板。** 特化 head 若达不到预注册判据，改写段在本框架内仍无解——我们将如实报告"域适配能移动接受率但不足以翻盘"的定量负结果；此时系统退化为"后缀主臂 + 分派开关"，刻画与两侧耦合的贡献不受影响（抗塌结构）。

---

## 8 结论

Deep research agent 的输出是对其自身中间产物的逐字再循环，而现有 serving 栈对此视而不见——内容被生成一次，却被反复按全新内容计费。我们量化了这一性质（量异质、形态逐字、随深度增长），据此在 decode 侧以重叠率驱动的投机收割重抄、在 prefill 侧以内容寻址复用重读，两侧以 TTFT 税对冲互补成环，端到端近乎腰斩且质量无损。更一般地，我们认为**内容级结构**是 agent serving 在调度层之下的下一个优化层：凡是以文本为介质、以引用忠实为约束的多调用系统，"once generated, many times served"的浪费与机会都同时存在。
