# sim_suffix.py 零基础教程:重演一场考试,算出 suffix 投机解码的理想上界

> **缘起**:实验测出 suffix 投机解码让 report 加速 1.76x、summary 反而略慢([summary.md](summary.md)),我们想知道**机制层面每一步发生了什么**、以及"就算所有开销全免,summary 最多能快多少"。真 server 回答不了这两个问题,于是有了这个 ~100 行的模拟器。本文从零讲清它的原理、每个数字怎么来、怎么用。
> **读法**:§一~§四 顺序读(概念梯子,每节只引入一个新想法);§五 手把手算一遍;赶时间只看 TL;DR 和 §六"怎么用";§八"进阶"跳过不影响理解。
> **文档分工**:后缀树数据结构本身 → [19 号教程](../../19-suffix-tree-explained.md);vLLM 里这套机制的真实调用链 → [18 号文档](../../18-vllm-suffix-decoding.md);本文只管**模拟器**。三篇用的是同一个玩具例子。

## TL;DR

sim_suffix.py 把"suffix 投机解码"这套猜词规则,**在纯 Python 里对 80 条真实请求逐 token 重演一遍**:prompt 是真的,"模型会说什么"用事先录好的真实输出当答案纸,唯独把所有时间开销当作零。得到的是**理想上界**:summary 就算这样也只有 1.33x,而 report 有 2.62x——"summary 调参救不了"从此钉死在机制层面,和实测(0.955x / 1.76x)方向一致。

---

## 一、背景 30 秒:decode 为什么慢,投机解码怎么救

大模型生成文本是一个字一个字往外蹦的:每吐 1 个 token,GPU 都要把**整个模型的权重从显存完整读一遍**——这是 decode 慢的根源,吐 1000 个 token 就要读 1000 遍。

投机解码的省钱思路:**先猜一串(比如 5 个 token),然后让模型一次前向把这 5 个一起检查,猜对的全部收下**。关键在这句话,记住它:

> **桥句 1:验证一串草稿和生成一个 token,成本几乎一样——都只读一遍权重。**

所以猜对 5 个 = 原来要读 5 遍权重,现在读 1 遍,净省 4 遍。猜错了呢?模型照常给出正确的下一个 token(它反正算了),**输出一个字都不会错、不会少**——亏的只是白检查了一次。

## 二、suffix 的"猜法":从 prompt 里抄

猜可以用小模型猜(要训练、要跑网络),suffix 方法更简单粗暴——**查表**:

> **桥句 2:看输出结尾这一小串,在 prompt 里(上次出现时)后面跟的是什么,就猜什么。**

为什么这招在我们场景灵:RESEARCH_SUMMARY / FINAL_REPORT 这类调用的输出**大段照搬输入 prompt**。模型抄 prompt 抄得越连续,"结尾这串上次后面跟什么"就猜得越长越准。(这套查表在 vLLM 里用后缀树实现,树本身怎么工作见 [19 号教程](../../19-suffix-tree-explained.md),此处不需要。)

## 三、为什么需要模拟器:真 server 回答不了的两个问题

实验跑完,我们手里有每条请求的**总加速比**,但有两个问题它回答不了:

1. **看不见内部**。想知道:哪一步出了草稿?草稿多长?被接受几个?"锁定"一段照搬花了几步?——生产代码里 propose 的内部过程没有逐步日志,加日志改 server 还会污染计时。
2. **拆不开"机制赚多少"和"开销花多少"**。开 suffix 有一笔每步都交的固定开销(≈18%,见 [18 号文档 §五](../../18-vllm-suffix-decoding.md));实测的 0.955x 是"赚的减掉花的",我们想单独知道**赚的那部分的上限**——"就算开销全免,summary 的照搬形状最多值多少?"这直接决定"调参还有没有救"。

所以需要一个**免费重演的沙盘**:把猜词规则单独拎出来,在真实数据上重放,不计任何时间成本。

## 四、沙盘的三个替身(以及为什么可以这么替)

模拟器用三个"替身"替掉真系统的三个部件:

**替身 ①:真模型 → 一张"答案纸"。** 实验时我们用 temp=0(贪心)从 server 录下了每条请求的完整输出文本([replay_outputs.jsonl](../data/replay_outputs.jsonl))。关键是:

> **桥句 3:temp=0 时,"模型下一个要说的词"是确定的,已经全部写在答案纸上。**

于是"验证草稿"不再需要跑模型——**拿草稿和答案纸对答案就行**:草稿从头连对几个,就是真系统会接受几个。(这一步严格成立仅限贪心;采样温度 >0 时验证是概率性的,本模拟器就不适用了。)

**替身 ②:后缀树 → 倒排表 + 逐步过滤。** 树的本质只是"把每个小串在 prompt 里的**所有出现位置**预先归好类"。模拟器不建树,现查:先建一张**倒排表**(每个 token → 它在 prompt 中的所有位置),要多长的串就在这张表的结果上逐步过滤(§五演示)。查出来的东西和树一模一样,只是慢一点——反正沙盘不计时。

**替身 ③:所有时间开销 → 零。** 不算验证的算力、不算查表的 CPU、不算真系统关异步调度的代价。所以模拟结果是**理想上界**:真实只会更差,不会更好。**上界的价值在"判死刑"**——上界都不够高,说明再怎么调参也救不回来。

## 五、手把手:在玩具例子上重演一步

用和 [18 号文档 §四](../../18-vllm-suffix-decoding.md) **同一个**玩具例子(那边走机制全程 8 步,这边把显微镜对准算法内部)。**教学简化:一个词当一个 token**(真实是 Qwen tokenizer 的子词);其余参数用真实值(往回看/树深 24、预算系数 1.0、置信度门槛 0.1)。

**题目卷(prompt,21 个 token,写下标方便对照):**

```
缓存₀ 分为₁ 本地₂ 缓存₃ 和₄ 全局₅ 缓存₆ 。₇ 本地₈ 缓存₉ 只₁₀ 存₁₁ 当前₁₂ 请求₁₃ ,₁₄ 全局₁₅ 缓存₁₆ 存₁₇ 历史₁₈ 请求₁₉ 。₂₀
```

**答案纸(模型 temp0 会输出的 15 个 token,照抄第二句但多加一个"而"):**

```
其中 本地 缓存 只 存 当前 请求 , 而 全局 缓存 存 历史 请求 。
```

### 5.1 开工前:建倒排表(替身②的前半)

扫一遍题目卷,记下每个 token 出现在哪些位置(出现 1 次的略):

| token | 出现位置 |
|---|---|
| 缓存 | 0, 3, 6, 9, 16 |
| 本地 | 2, 8 |
| 全局 | 5, 15 |
| 存 | 11, 17 |
| 请求 | 13, 19 |
| 。 | 7, 20 |

### 5.2 单步显微镜:已生成"其中 本地",这一步内部发生什么

此刻序列 = 题目卷 + `其中 本地`,取**尾巴**(末尾 ≤24 个 token;本例整个序列才 23 个,尾巴就是全部)。然后**从短到长枚举匹配长度 m**:

| m | 查什么 | 怎么查 | 结果(occ = 匹配到的位置集合) |
|--:|---|---|---|
| 1 | 尾巴最后 1 个:`本地` | 直接查倒排表 | occ = {2, 8} |
| 2 | 最后 2 个:`其中 本地` | 在 {2,8} 里留下"前一位是`其中`"的:位置 2 前面是`分为₁`✗,位置 8 前面是`。₇`✗ | occ = {} → **停止枚举** |

> 表格记法说明:**occ 存的是"匹配串最后一个 token 落在题目卷的哪个下标"**,不是匹配串本身。m=2 的过滤规则就是"从 m=1 的结果往前多对一位"——这个逐步过滤,等价于在后缀树里从根往下多走一步。

m=1 配上了,接着**出草稿**(三件事按顺序):

1. **定预算**(敢猜几个):预算 = min(24, m × 1.0) = min(24, 1) = **1 个**。匹配才 1 个 token,证据太弱,只敢猜 1 个——哪怕下面看到把握是十足的。
2. **投票**(猜哪个):看 occ 里每个位置的**下一个** token——位置 2 后面是`缓存₃`,位置 8 后面是`缓存₉`。票数:`缓存` 2 票 / 共 2 票。**置信度 = 2/2 = 1.0**(两次出现后面跟的完全一致,没有歧义)。1.0 ≥ 门槛 0.1,写进草稿。
3. **计总分**:score = 草稿各 token 置信度之和 = 1.0。(枚举的每个 m 都会产出一份草稿,score 最高者当选;本步只有 m=1 一份。)

最后**对答案**:草稿 `缓存` vs 答案纸下一个词`缓存`——对了,连对 1 个(acc=1)。按投机解码规则本步净吐 acc+1 = 2 个:`缓存` + 免费搭车的`只`(那个 +1 是模型前向反正会算出的下一个词,详见 [18 号文档 §三-7](../../18-vllm-suffix-decoding.md))。

### 5.3 再看两步(枚举择优 & 歧义投票)

**枚举 + score 择优**(已生成到"…本地 缓存 只"那步):

| m | 匹配串 | occ | 预算 | 草稿(置信度) | score |
|--:|---|---|--:|---|--:|
| 1 | `只` | {10} | 1 | `存`(1.0) | 1.0 |
| 2 | `缓存 只` | {10} | 2 | `存 当前`(全 1.0) | 2.0 |
| 3 | `本地 缓存 只` | {10} | 3 | `存 当前 请求`(全 1.0) | 3.0 |
| 4 | `其中 本地 缓存 只` | {} | — | (停止枚举) | — |

三份草稿,score 3.0 最高 → **m=3 当选**,猜 3 个、全对、净吐 4 个。注意规律:匹配越长 → 预算越大 → 草稿越长 → score 越高,所以**通常最长匹配胜出**;但"通常"不是"必然"——若最长匹配正好落在题目卷结尾(后面没词可猜),它的 score 反而低,短匹配会当选。这就是实现里枚举所有 m 而不是只取最长的原因。

**歧义时的投票**(假想某步尾巴只配上`缓存`,m=1):occ = {0,3,6,9,16},五个位置的下一个词是`分为`/`和`/`。`/`只`/`存`——各 1 票。最高票占比 1/5 = **0.2**:置信度 0.2 ≥ 0.1,勉强敢猜 1 个,但这是把握两成的赌注。**同一小串在 prompt 出现越多、续写越杂,反而越猜不动**;只出现一次的串(比如 URL)才是置信度 1.0 的独苗。(平票怎么办:实现取先出现的那个。真树按票数排序、平票看插入顺序——都是"谁先谁上",不影响任何结论。)

### 5.4 自检法(读者可以自己抓包)

- **每步净吐 = acc+1**:trace 文件每行引号里的文本恰好是 acc+1 个 token(唯一例外:最后一步的 +1 可能撞上输出结尾——本例 8 步的 Σ(acc+1)=16,比答案纸的 15 多出的 1 就是末步没发出去的搭车 token)。
- **置信度的分母必是票数总和**:任何一步的置信度都是"最高票/总票数"的连乘,拿 occ 数一数票就能验算。
- 本文和 18 号 §四 的所有数字都是同一套算法跑出来的,不是手编——想复查,§六 的 ③ 一条命令就能重演。

## 六、怎么用

**先认人**(四个角色):

- [`sim_suffix.py`](../../../../modify-code-runs/suffix-spec-decode/mechanism-sim/sim_suffix.py) 里的 `simulate()` = **车间**,一条请求逐步重演,产出每步四元组;`report()` = **记分员**,把一条的账汇总成人话;
- `sim_all.py` = **流水线**,80 条全量过车间 → `sim_all.json`;
- `trace.py` = **放大镜**,逐步打印每一步吐了什么文本;
- 数据:[workload.jsonl](../data/workload.jsonl) = 题目卷(`messages` 字段),[replay_outputs.jsonl](../data/replay_outputs.jsonl) = 答案纸(`text` 字段),按行号对齐(0-59 是 summary,60-79 是 report);tokenizer(Qwen3-32B,本地路径 `/data/yilin/huggingface/Qwen3-32B`)负责把文字切成 token。

```
workload.jsonl ──┐                       ┌→ report() 摘要      (sim_suffix.py 37 61)
                 ├→ tokenizer → simulate() ┼→ sim_all.json 汇总  (sim_all.py)
replay_outputs ──┘        (车间,逐步重演)  └→ trace_iXX.txt 逐步 (trace.py 53)
```

```bash
cd /home/yilin/modify-code-runs/suffix-spec-decode/mechanism-sim
python3 sim_suffix.py 37 61     # ① 单条摘要(传请求下标,可多个)
python3 sim_all.py              # ② 全量 80 条 → sim_all.json
python3 trace.py 53 > t.txt     # ③ 逐步 trace
```

**① 的输出逐行解读**(以 i37 为例):

```
输出 token 数(=baseline 步数): 257     ← 不开投机就要读 257 遍权重
suffix 步数: 174  (步数压缩 1.48x)      ← 理想上界:开销全免时最多快这么多
其中 空草稿步(纯税): 21 (12%)           ← 连草稿都出不了的步(没匹配/置信度全灭)
有草稿但全拒: 107                       ← 白对了一次答案的步(猜了,一个没中)
接受 token 总数: 83                     ← 赢的总额(每接受 1 个=省读 1 遍权重)
每步接受数分布: 0:128 1:25 2:13 ...     ← 形状指纹:全是小口 → 碎片式照搬
接受>=10 的步数: 0                      ← 有无"大口收割"是 summary/report 的分水岭
```

**② `sim_all.json` 字段**:`i`/`tag`(请求号/阶段)、`n_out`(输出 token 数 = baseline 步数)、`n_steps`(模拟步数)、`comp`(= n_out/n_steps,理想压缩)、`empty`(空草稿步)、`drafted`/`accepted`(草稿/接受总数,比值 = 接受率)、`big`(接受 ≥10 的步数)。

**③ trace 行格式**:`step k pos=P match=M draft=D acc=A | '文本'`——第 k 步、输出位置 P、匹配长 M、草稿长 D、接受 A 个,文本 = 本步净吐的 A+1 个 token。真实请求的 trace 长什么样、怎么读出"锁定→滚雪球→斩断",见 [why-high-overlap-no-speedup.md](why-high-overlap-no-speedup.md) §三/§四。

**换自己的数据**:仿照两个 jsonl 的字段(`messages`[{content}] / `text`)按行对齐;或 `from sim_suffix import simulate` 直接传两串 token id。**改机制参数**:改文件头四个常量 `MAX_DEPTH / MAX_SPEC / FACTOR / MIN_PROB`(对应 vLLM 的 max_tree_depth / num_speculative_tokens / max_spec_factor / min_token_prob)。

维护记录:2026-07-02 数据目录重组后已修——三个脚本的数据路径指向 `data/`,`sim_suffix.py` 补了 `if __name__ == "__main__"` 保护(此前被 import 时会白加载两遍数据)。

## 七、追问区(预判的坑)

- **凭什么拿录好的输出当验证标准?** 桥句 3:temp=0 时模型下一个词是确定的,答案纸=模型的 argmax,对答案=真验证。**温度 >0 一律不适用**。
- **倒排表+过滤凭什么等价于后缀树?** 树只是把"每个小串的所有出现位置"预先归类(桥句见 §四替身②);occ 的逐步过滤 = 在树里从根往下逐 token 走,终点集合相同。差别只在速度,而沙盘不计时。
- **两份草稿 score 平手选谁?** 实现用 `>=` 更新最优 → 后枚举的(更长的 m)胜出,与真实现一致。投票平票见 §5.3 括号。
- **第 0 步还没输出,尾巴是什么?** 是 prompt 自己的结尾。所以第 0 步就可能出草稿(用 prompt 结尾配 prompt 内部),模型第一个词往往是自己的话,这份草稿通常被拒——18 号 §四的 step0 就是这形态。
- **"步数压缩"凭什么当时间上界?** 每步的主成本都是"读一遍权重"(桥句 1),步数 ∝ 时间;模拟又免掉了全部额外开销,所以压缩比是时间加速的上界。对照真实还要打折:实测固定开销 ≈18%,summary 的 1.33x 上界打完折只剩 ≈1.09x。
- **草稿全拒亏 token 吗?** 不亏:每步至少净吐 1 个(acc+1 的那个 +1)。亏的是白做的验证算力和真系统里每步的固定开销。
- **模拟里 acc=20 的大步,真系统真吃得下吗?** 不完全:真树有"匹配+草稿 ≤ 树深 24"的结构上限(见 §八差异表末行),超长收割会拆成两步——模拟在这点上偏乐观。

## 八、进阶(跳过不影响理解)

### 8.1 与真系统的差异(诚实小字——决定你能拿它的数字做什么)

| 简化 | 方向 | 影响 |
|---|---|---|
| "树"只有 prompt:没模拟**已生成输出入树**(真系统无条件追加)和**全局树** | 偏保守 | 自我重复的照搬、跨请求命中都测不到;冷树+串行下影响小 |
| 没模拟全局树的**坏草稿**(跨请求相似前缀出高分错误草稿) | 偏乐观 | 真系统的拒稿率比模拟高 |
| 答案纸和计时 run 的输出**不逐字一致**(vLLM temp0 非逐位确定) | 双向 | 单条对不上,统计上代表([overlap-replay.md](overlap-replay.md) 表1 旁证) |
| **验证零成本、零税**:不算被拒草稿的验证开销、不算关异步调度+查树的 ~18% 固定税 | 偏乐观 | 所以输出是"理想上界",不是加速预测 |
| 用 tokenizer 对输出文本重新分词,当作当时生成的 token 序列 | 近似 | 干净文本下 encode(decode(ids))≈ids,边界处偶有出入 |
| 没建模"匹配+草稿 ≤ 树深24"的结构上限(模拟沿 prompt 原文延续,草稿可到 24) | 偏乐观 | 真实树上单步草稿 ≤ min(m, 24−m) ≤ 12(factor=1);长段照抄时模拟里 acc≥13 的大步,真实系统会拆成两步吃 |

**正确用法**:看**形状**(每步接受分布、大步有无、空草稿步占比、锁定-雪球-斩断的 trace 形态)和**聚合/对比**(summary 1.33x vs report 2.62x;与实测逐条相关 summary 0.48 / report 0.87)。**错误用法**:拿单条模拟值预测单条实测(i37 模拟上界 1.48x、实测 0.757x——短输出实测噪声大,方向一致但绝对值对不上)。

### 8.3 复杂度与耗时

每步 O(|occ(1)| + 草稿长×|occ|),大头是常见 token 的倒排桶。实测:1 万 token prompt + 257 token 输出秒级;4186 token 的 report 分钟级;80 条全量约 20 分钟。

## 九、术语表(忘了名词回来查)

| 术语 | 一句话 | 代码里叫 |
|---|---|---|
| 尾巴 / pattern | 当前序列末尾 ≤24 个 token,用来查表的钥匙 | `seq_tail` |
| 倒排表 | token → 它在题目卷里的所有位置 | `index` |

| 草稿 | 猜出来待验证的一串 token | `draft` |
| 置信度 | 按出现频率估的把握,分叉处连乘,<0.1 停 | `prob`(门槛 `MIN_PROB`) |
