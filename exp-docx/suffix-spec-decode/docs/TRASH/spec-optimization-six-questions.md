# 投机解码优化六问:每个创新点的动机与收益来源(小白版)

> 缘起:针对 deep research agent 解题过程的进一步优化,提出六个问题:①只验分数最高的草稿是否浪费?能否并行验证多个候选?②能否兼容异步调度?③能否预测"这一步/这道题"适不适合投机?④模型什么时候想抄原文、什么时候想总结,能否从内部状态判断?⑤选稿算法能否优化?⑥还有什么方案。
> **读法**:§零 先备五个概念(读过 [18 号](../../18-vllm-suffix-decoding.md)/[19 号](../../19-suffix-tree-explained.md)可跳);之后每个创新点固定三段——**哪里疼(动机)→ 钱从哪来(收益)→ 要花什么(代价与判定)**。全文比喻沿用 18/19 号那套:一步=开一次印刷机,后缀树=只会背题库的猜题人,验证=批改,固定税=入场费,草稿=赌注。
> 关联:方案分层与已判决项见 [optimization-proposal.md](optimization-proposal.md)(L0 采用/L3 否决/L2 搁置);机制背景见 [why-high-overlap-no-speedup.md](why-high-overlap-no-speedup.md)。

## 零、先备五个概念(30 秒)

1. **一步 = 开一次印刷机。** 模型每吐 1 个 token,GPU 都要把整个模型权重从显存完整读一遍——这就是 decode 的成本单位。**一切加速 = 减少开机次数。**
2. **投机 = 先猜后批。** 猜题人(后缀树,只会背"题库"= prompt 和历史输出)先猜一串草稿;模型开一次机把整串一起批改,对的前缀全收——开一次机吐出多个 token。猜错也不影响输出(无损),只是白批了一次。
3. **两笔钱,别混。** **入场费(固定税 ≈18%)**:开了投机后**每步都交、跟猜没猜中无关**——大头是"关异步调度"(CPU 备菜不能再藏在 GPU 炒菜的影子里),小头是每步查树。**赌注(草稿)**:batch=1 时验证搭"反正要开机"的便车、几乎免费;并发大了车上没空位,才开始要真钱。
4. **harvest(可收割占比)**:一条输出里"能连续逐字从 prompt 抄出来"的 token 占比。它单调预测加速,打平点 ~15-20%。
5. **基线已更新**:v2 端到端实测,全局开 suffix 默认参数 = **墙钟 −29%,summary x1.00 不亏**、report 大赚([l0-l3-v2-results.md](l0-l3-v2-results.md))。所以本文所有创新回答的是"**怎么再挤**",不是"怎么止损"。

## TL;DR(六问判定表)

| 问 | 判定 | 一句话理由 |
|---|---|---|
| ① 并行多候选(树状投机) | **搁置,等上游** | 天花板量出来只有 +11%:输的时候多半不是"押错马",是"模型不想在题库的马里选" |
| ② 兼容异步调度 | **上游工程,自己不碰** | 别人靠"锅里顺手出稿+提前留座",suffix 的草稿是饭后 CPU 想的、长短无常,留座留不对 |
| ③ 预测适不适合投机 | 逐步=微利;逐题=搁置 | 税的大头是"开店时就定的",单笔生意躲不掉;**该预测的是并发红线** |
| ④ 抄 vs 总结的信号 | match_len 免费可用;探针=可做的研究;attention 不可行 | 猜题人每步免费报告"在抄吗";FlashAttention 从不把注意力分数写出来 |
| ⑤ 打分算法 | 现状已近最优;改**代价感知**+**草稿越墙** | 免费自助餐的拿法不适合按重量收费的日子;report 被 24 层树墙腰斩 |
| ⑥ 新方向 | **EAGLE-3 对照评测**;力气转向 report 侧与 prefill 侧 | 学徒能猜题库外的话+不用关异步,恰好补猜题人两个死穴 |

## 一、并行多候选:动机成立,天花板量出来很低

**现状一句话**:猜题人每步内部其实产生了**很多份**候选草稿(每个匹配长度一份、本地/全局树各一份),但只把分数最高的一份交给模型批改;平手取更长的匹配。

### 创新点 1a:分数平手时,两份都验?

- **哪里疼**:直觉上"平手 = 两份一样好,扔掉一份可惜"。
- **为什么其实不赚**:平手的两份草稿,通常是**同一段原文的不同证据**(本地树查到的和全局树查到的是同一片话;匹配 2 个和匹配 3 个续写出的也是同一段)——内容相同,验两份 = 同一张答卷交两遍,纯浪费。
- **判定**:现状(平手取更长)没毛病,不动。

### 创新点 1b:分叉处多押几支(树状投机)

- **哪里疼(动机,完全成立)**:分叉处苹果/香蕉各 0.5,现在只押苹果——押错这步就白干。而验证是搭便车的:batch=1 下多验一支**几乎不加钱**。"第二注几乎免费,为什么不下?"
- **钱从哪来(如果做)**:原来押错的那些步,从"净吐 1 个"变成"命中另一支、净吐 2 个以上"。
- **但天花板刚量过,很低。** 用模拟器造了个"天眼"版本(`sim_tree_oracle.py`):假设分叉处**所有**支路都被并行验证(宽度无限)——这是一切多候选方案的绝对上限。"宽度无限"的准确含义,三点:①这里的"候选"**不是现有草稿池里的那十几份**(那是窄版天眼),而是**从匹配点出发、题库子树里每一条走得通的路径**都算一份——包括贪心走廊从不产出的支路(匹配 `我爱` 下的子树 苹果→汁/派、香蕉→好吃,三条路径全打包验证,占 5 个草稿 token);②"无限"= 每个分叉都**不裁剪、全带**,且往下递归(被题库真实分叉数封顶,非数学无穷);③它只能当天花板,因为验证成本 = 树节点数,宽度翻倍成本翻倍——并发下这些是真金白银的 token 预算,真实 tree spec 必须裁到每分叉 top 2~3 支,捞到的只会比 oracle 更少:

| | 线性(现状:只验最高分一条) | 树 oracle(无限宽并行) |
|---|--:|--:|
| summary 60 条·理想压缩 | 1.33x | **1.50x**(步数只多省 11.3%;单条中位 +12%) |
| i61 report | 3.32x | 3.67x(+10%) |

- **为什么这么低**:逐步 trace 早就给过答案——summary 草稿被拒时,模型接下来要说的多半是**它自己的过渡语**("而"/"improve"这类),这些词**在题库里根本不存在**。不是押错了马,是模型压根不想在这几匹马里选——马加得再多也没用。剩下真正"押错分叉"的少数步,才是那 11.3%。
- **互证(窄版天眼,`sim_multi_candidate.py`)——和树 oracle 的区别要钉清**:两个实验里"猜"的原料完全一样(同一棵题库树、同一套匹配),oracle 都**不发明题库外的 token**;区别只在"事后诸葛亮"安插在流水线的哪个环节:

  | | 窄版天眼 | 树 oracle |
  |---|---|---|
  | 草稿集合 | **与现状相同**(每个匹配长度一份贪心走廊稿,分叉处仍只走最高票) | **加宽**:分叉处所有支路都带上(宽度无限) |
  | 天眼开在哪 | **选稿**:不用 score,事后挑"实际命中最多的那份**已有**草稿" | **验证宽度**:真实续写在题库**任何一条**路径里就算接受 |
  | 回答什么 | score 选稿环节丢了多少分? | 验证端若吃树状草稿,最多多赚多少? |
  | 结果 | +1~2pp(score 选错的步仅 0~3%) | +11.3% 步 |
  | 对应现实方案 | 改选稿算法(不动验证端) | 树状投机(tree attention,引擎手术) |

  玩具三方对照(匹配 `我 爱`,分叉 苹果0.5/香蕉0.5;现状产出的各份草稿**全是苹果开头**,只是长短不同):模型要说**香蕉** → 现状拒;**窄版天眼照样拒**(已有候选里翻不出香蕉开头的);树 oracle 命中——这类步就是 +11.3% 的来源。模型要说**而**(题库里没有)→ 三者全拒——这类步占大头,所以连无限宽也只有 +11%。
  两个实验是"从两侧夹"(窄版 ⊂ 树版):窄版证明**选稿环节几乎无损**(score ≈ 事后最优,瓶颈不在挑哪份);树版给出**加宽验证的全部价值**;真正的失败模式"模型要说的话不在题库里",任何 oracle 都救不了。
- **要花什么**:模型的批改流水线现在只会批"一列直线"的答卷;树状答卷要让每个位置知道"我的前文是哪条支路"——需要 tree attention(自定义注意力掩码)+ 批改老师(rejection sampler)改造 + 调度记账,**引擎级手术**。arctic C++ 留了门(`_speculate_tree` + `Draft.parents`),但 vLLM 0.18 没接线。
- **判定**:+11% 的无限宽上界(真实 2~3 支更少、税照交、并发下赌注不再免费)配不上引擎级改造。**搁置,等上游。**

## 二、兼容异步调度:钱最多的方向,但那是上游的活

- **哪里疼(动机,这是六问里钱最多的)**:入场费的大头就是它。回顾人话:正常时 CPU 给**下一步**备菜(排班、构建输入)和 GPU 炒**当前**这道菜是同时进行的,每步墙钟 ≈ 炒菜时间;开了 suffix,备菜必须等炒完——因为下一步的菜单取决于"这一步客人吃了几口(接受几个)"+"CPU 饭后才想出来的草稿"。每步硬多几毫秒,积起来 ≈18%。**若兼容成功,这笔税直接退回来。**
- **为什么 EAGLE 那些方法可以**:两个条件它们都有——①草稿是"锅里顺手做出来的"(在 GPU 前向内部产生,不用等饭后);②调度员按"每人最多带 K 个菜"**提前留座**(padded speculation),不用知道实际几个。
- **suffix 为什么卡死**:草稿是饭后 CPU 想的(条件①没了);长度 0~12 忽长忽短(条件②留座留不对):留 24 个座,常常只坐 1 人——并发下座位就是 token 预算,浪费不起;留 4 个座,又把 report 的长草稿掐死——**恰好掐在赚钱的地方**。"先乐观留座、错了回滚"是研究级引擎改造。
- **判定与替代**:上游(vLLM)的活,自己不碰。自己能动的:①税的小头(查树)——树深扫描(§六);②绕过而非解决——换"锅里出稿"的学徒(§六 EAGLE);③接受现状(端到端已净赚 29%)。

## 三、预测"适不适合投机":先看清税的结构,再决定预测什么

- **哪里疼(动机)**:"不适合投机的步/题,别交税"——听起来天经地义。
- **为什么收益比直觉小**:税分两半。大头"关异步调度"是**开店时就定的、全店统一**——不管这一步猜不猜,税都在;逐步预测能省的只有"查树"那点小钱。而且 suffix 天然自带逐步刹车(没匹配 = 空草稿 = 不下注),预测器能改进的空间很薄。逐题躲税则需要**开两家店**(开/不开投机的两个 server 池,因为 vLLM 的投机配置是 server 级),而 summary 已经不亏——为省不了几个钱的事把服务架构翻倍,不值。
- **真正值得预测的**:**并发红线**。便车逻辑——车上人多了就没便宜可搭;投机收益随并发缩水,什么档位转负还没压测过。预测"现在多忙、该收多大的手",接 §五(a) 的动态赌注。
- **判定**:逐步=微利可选(接受率滑窗,连续 N 步零接受暂停猜题);逐题路由=维持搁置(前提被 v2 推翻);并发红线压测=值得做。

## 四、模型什么时候抄、什么时候总结:信号从便宜到贵

- **哪里疼(动机)**:如果能提前知道"这题/这段会大段抄",就能提前调整赌注、路由、甚至改 prompt。
- 五种信号,按获取成本排序:

1. **任务类型(最便宜、已实证 r=0.55)**:检索/罗列型(要资源/名录/数据,输出≈把来源摆出来,URL 密度高)→ 抄;分析/综述型(重组多源观点)→ 总结。再加压缩比:4 万字进、4 千字出 → 必然是总结。report 阶段天然抄(任务本身就是把 summary 组织进终稿)。
2. **输出内部位置**:同一份输出里,小标题/过渡句 = 自己的话,列表项/术语串/引用 = 抄——逐步 trace(i37/i53)里肉眼可见。
3. **match_len 轨迹(免费的在线信号,容易被忽略)**:想知道"模型此刻在抄吗",第一反应是去看模型脑子——**其实猜题人每步都在免费报告这个信号**:匹配长度连续增长 = 正在抄,归零 = 回到自己的话。做在线自适应(动态调赌注/暂停猜题)用它就够,**不用碰模型内部**。
   两个容易混的层次要分开:**"命中的 token 进入下一步匹配"是机制本身早就有的**(接受的 token 成为序列结尾 → 下一步 pattern 以它们结尾 → 继续抄则匹配变长,即 18 号 §四 的"雪球";注意匹配没有跨步状态,每步都从树根重新枚举,不是断点续传)。而这里的建议是另一件事:**机制只在"本步之内"消费一次 match_len(定预算),跨步的轨迹目前被扔掉**——没人看这条曲线。
   同理,"命中多就多投、没命中就不投"在**单步层面也已内置**(match 长→预算大,没 match→空草稿不下注),但它**无记忆**:连续 200 步一无所获,第 201 步照样把整套查询做完、查完才知道又是空——查询费一步没省。建议的增量正在于此:做成**滞回开关**——连续 N 步低迷(match≈0/接受 0)→ **连 propose 都跳过**(这才省到钱);恢复靠便宜触发器(每 K 步抽查一次,或见到 URL/引号/列表编号立刻恢复);轨迹爬升时反向更激进(调大 factor、并发下保额度)。"命中"信号可用 match_len(查树得来,"有肉的迹象")也可用每步接受数(验证得来,"上注赢没赢",更直接且免费),两个都看最好。轨迹长什么样:i53(分析型)`1,1,2,1,0,1…` 低位打摆 = 基本不在抄;i61(抄 URL)`1,2,5,11,23` 单调爬升 = 大丰收。建议 = 加个便宜控制器盯它:连续 N 步 match≈0 → 调小 factor/暂停 propose(省 CPU 税);爬升 → 放开下注;并发紧张时优先收"轨迹低迷"请求的投机额度。"免费" = `Draft.match_len` 本来就是每步算好返回的字段,零新增成本——对比 attention(拿不到)和探针(要训),猜题人的工作日志里已经写着答案,只是今天没人翻。
4. **hidden state 探针(可做的研究实验)**:动机是把预测提前到**模型刚读完题那一刻**(比 match_len 更早,才能用于路由/预算分配)。为什么可能行:模型读完题时"打算怎么答"(罗列还是综述)应该已经编码在它此刻的内部向量里——拿一个几百参数的线性小分类器去读这个向量,预测这条请求的 harvest。为什么便宜:**标签免费**(1812 条旧日志离线跑 overlap 脚本批量打标,不花 GPU)、特征免费(prefill 反正要跑)、训练笔记本级。先验证"可预测性"再谈部署。
5. **attention 分布(理论成立、在线拿不到)**:照抄行为确实有 attention 指纹(注意力像手指按着原文逐行移动——文献里的 induction heads / retrieval heads)。但现代注意力内核(FlashAttention/paged attention)为了快,**从头到尾不把注意力分数写进显存**——想要分数就得重写内核、整体变慢,本末倒置。别走这条路。

- **清醒剂**:预测出来必须能**改变某个动作**才有价值。路由已低 ROI、动态调参用信号 3 就够——所以本问目前是理解性/长线课题,不在关键路径上。

## 五、打分算法:先辩护,再讲两个真改进

**先辩护现状**:score = 草稿各 token 累积置信度之和 ≈ **这份草稿期望能被批对几个**。在"赌注免费"(batch=1)的世界里,"期望批对的多"就是唯一该最大化的量——现状已经是对的目标函数,不是低垂果实(上面窄版天眼的 +1~2pp 也证明了这点)。改进空间在两个假设失效的地方:

### 创新点 5a:代价感知投机(赌注免费的假设在并发下失效)

- **哪里疼**:现在的打分默认"验证搭便车、赌注免费"。batch=1 时是真的;但生产部署是**题内 9 路并发**——车上有人了,草稿开始占别人的座(token 预算、算力)。打个比方:**免费自助餐时多拿不亏;改成按重量收费,还按免费的拿法拿,就开始亏**。
- **钱从哪来**:把目标从"E[批对数]"改成"E[批对数] − λ×草稿长",λ = "现在车上多挤"的换算系数(随 batch 大小升)。效果:忙时自动少下注(草稿变短、门槛变高),闲时放开下——每一步的赌注都调到当时的性价比最优,防止并发高峰期投机反噬吞吐。
- **要花什么**:引擎小改(按负载动态缩 `max_spec_factor`/抬 `min_token_prob`)+ 一轮并发压测标定 λ。六问里最务实的算法改动。

### 创新点 5b:草稿"越墙"(树深 24 的结构墙腰斩了 report)

- **哪里疼**:report 整段照抄时,理论上一步能吞 24 个 token,实际被卡在 ~12。原因是一堵**结构墙**:树只有 24 层,匹配用掉 m 层,草稿只能再往下走 24−m 层(匹配+草稿 ≤ 24)——配上"预算 = m"的规则,单步最多 min(m, 24−m) ≤ 12。**最赚钱的场景(整段照抄)恰好被腰斩。**
- **钱从哪来**:树的叶子节点本来就记着"我是原文第几页第几行"(`ref_seq/ref_idx` 引用,19 号 §三讲过的路径压缩账本)。草稿走到树的尽头、且路径是独苗(count=1,说明就是在抄某一段唯一的原文)时——**翻开原文接着抄就行了**。不用把树建深(建深 = 每个 token 的建树成本全面上涨,等于给所有请求加税);"越墙"只在"已确认大丰收"的步发生,日常零成本。report 的单步草稿从 ~12 抬向 24。
- **要花什么**:改 arctic C++ 几十行 + 先用模拟器加"越墙"变体量上界,值了再动手。report 是 decode 时间大头——这里的每 1% 比 summary 的 1% 值钱。

### 创新点 5c:给全局树的票打折(高票 ≠ 适合你)

- **哪里疼**:全局树的票是**一万个别的请求**投的——"这串后面接 X"在别人的语境里对,不代表在你这道题里对。现在本地/全局两份草稿分数硬碰硬比,可能让"外来的高票"压过"本地的真相",押了外来稿被拒。
- **钱从哪来**:用线上日志统计两个来源各自的**历史命中率**,给全局票乘个折扣,让分数回归"真实赢率"→ 少押错。
- **风险(先量再动)**:v2 端到端里全局树恰恰是 summary 翻正的功臣(同题 18 份 summary 互相喂)——折扣下狠了会把红利折没。先从日志量出两源真实命中率,再决定折多少。

## 六、新方向与路线图

### 创新点 6a:EAGLE-3 对照评测(换个不用背题库的猜题人)

- **哪里疼**:suffix 猜题人有两个天生死穴。①**题库里没有的它一个字都猜不了**——summary 里 ~46% 是模型自己组织的话,这些步全在交纯税;②**雇它就必须关异步调度**(18% 入场费,§二)。
- **EAGLE 是什么(人话)**:给大模型配一个**小学徒**——一个很小的神经网络,挂在大模型身上,看着大模型此刻的"脑内状态"(hidden state)猜接下来几个词。它不靠题库,靠**学过大模型的说话习惯**——所以转述、组织语言这种题库外的内容它也能猜(业界典型 2-3x);而且它在 GPU 锅里顺手干活(§二的条件①②都满足),**异步调度不用关**。两个死穴同时消失。
- **钱从哪来**:summary 的纯税区(~46%)变成可加速区 + 退掉 18% 入场费。
- **要花什么/为什么是"评测"不是"换"**:①要 Qwen3-32B 的学徒头——先查社区有没有现成的,没有就要自己训(真成本所在)+ 少量显存;②suffix 的独门绝技(整段照抄一步吞十几个、热全局树跨请求记忆)学徒未必赶得上——report 侧可能反而退步。所以是**同 workload 对照评测**,谁端到端强用谁。

### 创新点 6b:blend × suffix 兼容测试(两刀切在不同段)

- **动机**:一次请求的时间 = 读题(prefill)+ 写作(decode)。suffix 只省写作;LMCache blend(KV 复用)省读题。**两刀切在不同段,理论上互不抢 = 可叠加**——report 这种"长 prompt + 长输出"两头都吃。
- **为什么要测**:从没同时开过。blend 会改写 KV,suffix 的树照常工作吗?connector 和 speculative-config 同开稳不稳?一次 server 启动 + 20 条请求就能验——**成本最低、悬赏最久的一项**。

### 创新点 6c:树深扫描(税的小头,一次重启的实验)

- **动机**:树深 24 是每 token 建树成本的乘数(24 个书签各走一步,19 号 §三)。砍到 12:查树的税约折半;代价:长匹配封顶 12,report 的长草稿受损。净方向量了才知道——**唯一没扫过的参数,六问里最便宜的实验**。

### 路线图(按优先级)

1. **blend × suffix 兼容测试**(6b,20 条即验);
2. **并发压测找红线 + 动态赌注**(5a);
3. **模拟器量"越墙"上界 → 决定改不改 arctic C++**(5b);
4. **EAGLE-3 对照评测**(6a);
5. **树深扫描**(6c);
6. **搁置**:树状投机(①,上界 +11% 不值引擎手术)、逐题路由(③,前提翻案)、attention 在线信号(④,拿不到分数)。

**优先级总纲**:summary 已不亏、且它的 decode 占比本来就小——力气花在 **report 侧**(越墙、树深)和 **prefill 侧**(blend),别继续抠 summary。
