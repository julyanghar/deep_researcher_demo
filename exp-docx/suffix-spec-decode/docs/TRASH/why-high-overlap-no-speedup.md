# 为什么重叠很大、suffix 却没效果:两条真实请求的逐 token 拆解

> 缘起:[questions-log.md](questions-log.md) Q9 已经给了定性答案——"overlap 量的是照搬总面积,suffix 吃的是单段连续长度;summary 是碎纸屑,report 是整页复印"。本文把这个比喻**钉到代码行和真实请求上**:按 [../18-vllm-suffix-decoding.md](../../18-vllm-suffix-decoding.md) 拆出的源码机制写了一个逐步模拟器,把本实验 80 条真实请求逐 token 重放一遍,让"碎片吃不到肉"从比喻变成**能逐步数出来的数字**。

## 一、先看一对"重叠一样大、命运相反"的真实请求

从本实验 80 条里挑出的一对(字符级重叠面积几乎相同):

| | i53(RESEARCH_SUMMARY)| i61(FINAL_REPORT)|
|---|--:|--:|
| 内容 | MDA 游戏设计框架的**分析型摘要** | 复述来源+URL 的**长报告** |
| 字符级 Coverage@8(重叠"面积")| **85.9%** | **85.0%** ← 几乎一样 |
| 字符级 Coverage@30(≥30字长段)| 11.1% | 76.6% |
| 最长照搬段(字符)| 79 | 206 |
| **最长照搬段(token)** | **10** | **140** |
| 可收割 token 占比(harvest)| 9.5% | 71.1% |
| **实测加速(长度归一 char/s 比)** | **0.78x(慢 22%)** | **1.88x** |

(口径注记:未做长度归一的 decode 时间直接比是 0.62x vs 2.91x——但两次 temp0 运行的输出**长度本就不同**(i53 的 suffix run 多写了 26%),时间直接比会把"这次答案恰好写长/写短"也算进快慢,对短请求噪声很大。本文统一用长度归一口径,与 overlap-replay/题级差异报告一致;详见 §六 勘误条。)

按"重叠面积"看,这两条应该同样受益——实际差了 **2.4 倍**、一负一正。再补一条更狠的:i37(UAV 控制综述)字符级 cov8 高达 **94%**、连 cov30 都有 52.7%,长度归一实测也只有 **0.89x**,翻不了正(这条为什么"说得通",§六 末有专门拆解)。

**结论先行:重叠的"面积"不决定 suffix 的收益,重叠的"形状"——单段连续 token 长度——才决定。** 下面用代码说清楚为什么形状是唯一重要的,再用逐步 trace 给你看这两条请求内部每一步发生了什么。

## 二、决定命运的三条代码规则(+一笔进场费)

完整调用链见 [../18-vllm-suffix-decoding.md](../../18-vllm-suffix-decoding.md),这里只拎出和"形状"直接相关的三条:

**规则 1:锁定靠结尾逐 token 匹配,一段照搬要重新锁一次。**
每步拿**当前序列末尾 ≤24 个 token** 当 pattern([suffix_decoding.py:77-78](../../../../anaconda3/envs/lmcache/lib/python3.12/site-packages/vllm/v1/spec_decode/suffix_decoding.py#L77-L78)),去 prompt 后缀树里配([`_match_context`,suffix_tree.cc:800](../../../../anaconda3/envs/lmcache/lib/python3.12/site-packages/arctic_inference/csrc/suffix_decoding/suffix_tree.cc#L800));"配上 m 个"指**末尾最后 m 个 token 这个后缀在 prompt 某处出现过**(m=1..24 逐个试,配不上即停)。模型**刚要开始抄某句**时,结尾还是它自己的过渡语——要么配不上,要么只配上 1~2 个 token 且 prompt 里到处都是这 1~2 个 token(下文续写五花八门,猜了也是瞎猜)。
"这一小串在 prompt 多处出现"**不是挑一处抄,而是所有出现处一起投票**:树节点记着每种续写出现过几次,置信度 = 子节点 count ÷ 父节点 count,沿草稿**连乘**,跌破 `min_token_prob=0.1` 就不再往下写。i53 的 prompt 里 "Dragon Age: Origins" 出现约 40 次、后面跟的各不相同(", examining"/" as an MDA"/" include magic"/" is its"/" has several"……),分叉一散,连乘一两步就跌穿门槛——**出现越多、歧义越大,反而越猜不动;唯一出现的串**(如一条 URL)沿途无分叉、置信度恒 1.0,才能一口气猜到预算上限。**必须真抄进去几个 token,匹配才"锁定"**。这笔锁定成本,抄 38 个碎片段就要交 38 次,抄 1 个整段只交 1 次。

**规则 2:胆量∝证据,草稿长度 = 已匹配长度 × 1.0。**
预算公式 `max_tokens = min(24, match_len × max_spec_factor)`([suffix_tree.cc:606-609](../../../../anaconda3/envs/lmcache/lib/python3.12/site-packages/arctic_inference/csrc/suffix_decoding/suffix_tree.cc#L606-L609));实际草稿长度 = min(这个预算, 置信度跌破 0.1 或走到路径尽头的位置)。刚锁定时 match_len=3 → 只敢猜 3 个;猜中了,下一步 match_len≈7 → 敢猜 7 个……**草稿像滚雪球,要滚四五步才能滚到顶格 24**。段太短,雪球还没滚大这段就抄完了——注意账要这么算:**"净赚 1~2 个"只发生在碎片段内部的那几步;段外的大多数步一无所获还照样交税**(i53:388 步里 313 步接受 0 个)。所以不是"每步都小赚",而是"少数步小赚、多数步纯亏"。

**规则 3:验证按前缀收割,微改写当场斩断。**
草稿和目标模型的 argmax 逐位置比,第一个不等处斩断、其后全弃([rejection_sampler.py:388-404](../../../../anaconda3/envs/lmcache/lib/python3.12/site-packages/vllm/v1/sample/rejection_sampler.py#L388-L404))。summary 抄的时候爱小改(换个连接词、缩掉半句)——字符级统计里这只是把一个长段记成两个短段、**面积照样计入 coverage**;但对 suffix 是草稿被拒 + 回到规则 1 重新锁定。
真例(i53,就是 §三 trace 的 step307-309):prompt 原文是 "…such as Dragon Age: Origins, to analyze **its narrative design**…",模型写成 "…such as Dragon Age: Origins, to analyze **and improve game design elements** like combat…"——前 7 个 token(": Origins, to analyze and")顺着原文顺利收割;下一步模型要说 " improve",树按 prompt 原文方向出的草稿第一位就和它不等 → **整份草稿全拒,白验证一次**;此后结尾 "…analyze and improve" 在 prompt 里不存在,匹配归零、重新走一遍锁定。字符级统计里 "such as Dragon Age: Origins, to analyze " 这 40 多个字符**全部计入 coverage**,token 级却只赚了 7 个——"看起来在抄"和"逐 token 一字不差地抄"之间的折损就是这样发生的。

**进场费:** 只要开 suffix,vLLM 就**关掉 async scheduling**([vllm.py:719-731](../../../../anaconda3/envs/lmcache/lib/python3.12/site-packages/vllm/config/vllm.py#L719-L731))+ 每步维护/查询后缀树。为什么关异步这么贵:正常时 CPU 为下一步排班/构建输入张量可以**和 GPU 算当前步重叠**,每步墙钟 ≈ GPU 时间,CPU 开销"藏"在 GPU 计算后面;而 suffix 的草稿是**采样之后在 CPU 上算出来的**、且每步每请求净吐几个 token(1~25)验证完才知道,下一步的 batch 内容必须等本步全部结束才能定 → CPU 段从"藏在 GPU 后面"变成"串在 GPU 前面",每步多出几毫秒。实测这笔固定税 **≈18%**(几乎无照搬的请求恒 0.81x,见 [overlap-replay.md](overlap-replay.md)),每步都收、与命中无关。

这四条合起来就是判词:**suffix 是"按段收费的复印机"——每段先交锁定费,段越长越划算;碎纸屑复印一张亏一张;没得复印的步数还照交机器租金。**

## 三、慢例 i53:"86% 重叠"在 token 级根本不存在

i53 的输出长这样(开头):

> Recent developments in procedural content generation, player behavior modeling, and AI integration have influenced practical applications within the MDA (Mechanics, Dynamics, Aesthetics) game design framework in several ways: 1. **MDA Framework Overview**: The MDA framework provides a structured ap…

典型的**分析型转述**:术语全部来自 prompt("procedural content generation"、"player behavior modeling"……随便一个短语都 ≥8 字符、都能在 prompt 里找到,所以字符级 cov8 高达 86%),但**句子骨架是模型自己的**——它在重组、压缩、换连接词,不是在整段复印。

按源码机制模拟(验证零成本的理想上界,方法见 §六),i53 全程 483 个输出 token 的账本:

| 指标 | 数值 | 说明 |
|---|--:|---|
| baseline 步数(=输出 token 数)| 483 | 一步一个 token |
| suffix 步数 | 388 | **只省 95 步,理想上界 1.24x** |
| 一个 token 都没接受的步 | 313(81%)| 其中 80 步连草稿都出不了 |
| 每步接受数分布 | 0:313 1:61 2:10 3:3 5:1 | **全场最大的一步只接受 5 个** |
| 接受 ≥10 的"大步" | **0** | 整条请求没有一次大口收割 |
| 草稿总数 → 接受 | 502 → 95(19%)| 八成草稿是白验证 |

逐步 trace 看它"最好的一段"什么样(全场唯一一次 acc=5,在抄游戏名 "Dragon Age: Origins" 附近):

```
step304 pos=372 match=1 draft=1 acc=0 | ' games'
step305 pos=373 match=1 draft=1 acc=0 | ','
step306 pos=374 match=2 draft=2 acc=0 | ' such'
step307 pos=375 match=2 draft=2 acc=2 | ' as Dragon Age'          ← 锁定了
step308 pos=378 match=5 draft=5 acc=5 | ': Origins, to analyze and' ← 雪球刚滚起来
step309 pos=384 match=2 draft=2 acc=0 | ' improve'                 ← 模型换自己的话,斩断
step310 pos=385 match=0 draft=0 acc=0 | ' game'
step311 pos=386 match=1 draft=1 acc=1 | ' design elements'
```

(`match`=锁定的匹配长度,`draft`=草稿长度,`acc`=被接受数,`|` 后是该步实际吐出的文本 = acc+1 个 token。)

三条规则全在这 8 行里显形:step304-306 是**规则 1 的锁定费**(match 只有 1~2,猜 1~2 个还全错);step307-308 锁上了、**规则 2 的雪球刚从 2 滚到 5**;step309 模型开始说自己的话,**规则 3 当场斩断**,回到重新锁定。这一段"照搬"总共收割 7 个 token,就是 i53 全场的巅峰。

**为什么字符级统计说它 86% 重叠?** 因为 cov8 问的是"输出里这串 ≥8 字符的字,在 prompt 里出现过吗"——术语当然出现过。但 suffix 需要的是"**结尾这一串和 prompt 某处逐 token 一致,并且下文还继续一致**"。术语嵌在模型自己的句法里,token 级连续匹配只有 1~3 个,一碰到模型自己的连接词就断。**86% 的面积,在 token 级的连续段里只剩 9.5% 可收割(harvest)**。

算总账:理想上界 1.24x,交完 ~18% 进场费 ≈ 1.0x 打平——这还是"每步验证白送"的假设;长度归一实测 0.78x(单条短请求还有 temp0 噪声和 propose 的 CPU 开销,见 §六)。**怎么算都无肉。**

i37 同理更极端:cov8=94%、cov30=52.7%,看着像"抄了一大半",但 174 步里**没有一步接受 ≥10**,最好的一步是抄专有名词连串 " Disturbance Observer (DOB)"(match=11 draft=11 却只 acc=6——prompt 原文后面跟的和模型要说的不一样,规则 3 斩断)。它的照搬全是"编号列表里每句嵌几个术语"的形态,trace 里满屏 `match=1~2 draft=1~2 acc=0`。

## 四、快例 i61:同样 85% 的重叠,形状完全不同

i61 是长报告,4186 个输出 token,大段复述它 prompt 里的 summary(报告的活儿本质就是"把各 researcher 的 summary 组织进最终稿")。模拟账本:

| 指标 | i61 | 对照 i53 |
|---|--:|--:|
| 步数压缩(理想上界)| 4186→1261,**3.32x** | 1.24x |
| 接受 ≥10 的大步 | **93 次**(其中 13 次顶格 24)| 0 |
| 草稿 → 接受 | 4968 → 2925(**59%**)| 19% |

看一个典型窗口——抄 prompt 里的一条 PDF 链接:

```
step256 pos=536 match= 1 draft= 1 acc= 0 | 'http'
step257 pos=537 match= 0 draft= 0 acc= 0 | '://'
step258 pos=538 match= 1 draft= 1 acc= 0 | 'wb'
step259 pos=539 match= 2 draft= 2 acc= 2 | 'xb.xml-j'                      ← 锁定
step260 pos=542 match= 5 draft= 5 acc= 5 | 'ournal.net/cn/article/pdf'     ← 5
step261 pos=548 match=11 draft=11 acc=11 | '/preview/10.14183/j'           ← 11
step262 pos=560 match=23 draft=23 acc=20 | '.cnki.1005-6122.JMW25095.pdf]' ← 20
step263 pos=581 match= 1 draft= 1 acc= 0 | '。'
```

**同样先交三步锁定费**(step256-258,和 i53 一模一样)——但因为这段照搬够长,雪球滚起来了:2→5→11→23,**7 步吐完 ~45 个 token**。锁定费摊在 45 个 token 上不痛不痒;i53 的锁定费摊在 7 个 token 上,摊不平。

同一台机器、同一套机制、几乎同样的"重叠面积",差别只有一个:**i61 的重叠是连续几十上百 token 的整段(最长 140 token),i53 的重叠是嵌在自己句法里的 1~10 token 术语碎片。**

## 六、模拟器怎么做的(+诚实小字)

脚本 `sim_suffix.py`(存于 `/home/yilin/modify-code-runs/suffix-spec-decode/mechanism-sim/`,含 80 条汇总 `sim_all.json` 和 i37/i53/i61 全量逐步 trace):

- **忠实复刻源码的部分**:pattern 取末尾 ≤24 token;枚举 match_len=1..24、短的配不上即 break;预算 = min(24, match_len×1.0);沿最高频续写走、分叉处置信度连乘、<0.1 停;各 match_len 按 score 择优;验证 = 草稿与真实输出的最长公共前缀 m 被接受、本步净吐 m+1。与 [../18-vllm-suffix-decoding.md](../../18-vllm-suffix-decoding.md) §3 的调用链逐条对应。
- **简化**:①树只用 prompt 建(忽略"已生成输出也入树"和跨请求全局树——前者略低估自我重复的命中,后者在冷树串行下影响小);②拿 capture 到的 temp0 输出当 ground truth,它和计时 run 的实际输出不逐字一致(vLLM temp0 非逐位确定,见 [overlap-replay.md](overlap-replay.md) 诚实小字)。
- **所以模拟数字是"结构指纹",不是绝对值预测**:看形状(每步接受分布、大步有无、锁定费次数)很可靠,拿单条模拟值对单条实测会有出入——比如 i37 模拟上界 1.48x、长度归一实测 0.89x,短输出(257 token)的单条实测本身噪声就大;它所在 harvest 桶(harvest≈27%)的长度归一均值是 1.08x,和模拟方向一致。聚合与相关性(§五)才是校准的正确读法。
- **勘误(2026-07-02,加速比口径)**:[per_call_overlap.csv](../data/per_call_overlap.csv) 的 `speedup` 列是 **decode 时间直接比**(baseline decode_s ÷ suffix decode_s),**没有做长度归一**;本文初版从该列引用了 i37=0.757、i53=0.622、i61=2.906,均已改为长度归一 char/s 比(0.89 / 0.78 / 1.88)。为什么归一才对:两次 temp0 运行的输出**内容和长度都不同**(80 条中长度相对差中位 7%、P90 33%;极端如 i23,baseline 写了 1218 字符、suffix 写了 9242 字符,时间直接比 0.34x、归一后 2.56x——完全是两码事),时间直接比把"这次恰好写多写少"也算进了快慢。**桶级结论对口径不敏感**(harvest 四分位的剂量-反应两种算法几乎重合:0.82/0.97/1.08/1.90 vs 0.81/0.96/1.06/1.94),但**单条数字必须用归一口径引用**。
- **专门拆一遍 i37(§一里"更狠"的那条,为什么其实说得通)**:它的反直觉感来自把三层折损混在一起看。第一折,字符 94% ≠ token 可收割——cov30=52.7% 听着吓人,但 30 字符只有 ~8-10 个 token,i37 全场**最长**照搬段 75 字符 = 17 个 token,tok_cov24=0,harvest 27%;机制模拟 257 步只省 83 步、没有一步接受 ≥10 → **上界 1.48x**,这是 94% 重叠的真实身价。第二折,~18% 固定税 → 净期望 ≈1.1~1.2x,即"约打平、略正"。第三折是测量:长度归一实测 0.89x,与"约打平"相差的最后一点是 1.3K 字符、6 秒 decode 的单条噪声(它所在桶均值 1.08x)。所以不是机制说不通,是"0.757 翻不了正"这句话里一半是口径水分、一半是单条噪声——机制预期本来就只是打平。

## 七、一句话收拢

**判断 suffix 值不值得开,别问"输出里有多少内容来自输入"(coverage 面积——它对 summary 加速的相关是 −0.23,反向!见 [per-request-speedup-variance.md](explore-idea/data-source/per-request-speedup-variance.md);反向的另一半原因是语言混淆:字符 cov8 与每 token 字符数相关 +0.88,几乎在测语言而非照搬,见 [questions-log.md](questions-log.md) Q19),要问"输出里有多长的连续 token 串逐字来自输入"(token 级 harvest,打平点 ~15-20%)。** 因为机制上 suffix 的每一分收益都来自"锁定后的连续收割",而每段照搬都要预交锁定费、每步解码都要交进场费——面积再大,切成碎片就摊不平这两笔钱。

## 附录:80 条全量表(token 级重叠 × 加速比,按 harvest 升序)

> 应 Q18 的追问补全:每条请求的 token 级重叠指标与长度归一加速比。机器可读版含未归一口径:[per_call_overlap_norm.csv](../data/per_call_overlap_norm.csv)(`spd`=长度归一,`spd_raw`=decode 时间直接比;原 per_call_overlap.csv 的 speedup 列即后者,勘误见 §六)。
> **harvest 列怎么读**(Q20):它是**光看文本形状估出来的"白捡区"占比**——数"前面已连续照搬 ≥3 个 token、且下一个还在延续"的位置数 ÷ 输出总 token 数([analyze_replay_overlap.py:46](../../../../modify-code-runs/suffix-spec-decode/analyze_replay_overlap.py#L46)),算它时**没有跑任何草稿/验证**。它不是"草稿实际生效的 token 占比",但和模拟器的实际接受占比几乎重合(80 条相关 **0.99**);单条会偏:"锁 3 个才开张"的近似**偏严**——短匹配若续写独一份照样命中(i53:harvest 9.5% vs 模拟实际接受 19.7%);长段照搬时**非常准**(i61:71.0% vs 69.9%)。

**先读结论(表是证据)**:
- **summary 加速比不高的根本原因就是 token 级连续重叠低**:60 条 summary 的 harvest 中位仅 14.3%,低于打平带(15-20%);**harvest≤20% 的 42 条中位 0.86x,>20% 的 18 条中位 1.07x**——同一阶段内,过线就翻正。
- 说"根本原因"要带上另一半:低重叠只解释"**赚得少**",把"赚得少"变成"净亏"的是那笔 ~18% 固定税(每步都交)。若税为零,harvest 再低也只是不赚(≈1.0x),不会变慢。
- 表按 harvest 排序后**任务标签是交错的**:harvest 33-52% 的 5 条 summary(i34/i9/i27/i10/i17,1.13-1.52x)排在最低的 report(i77,36.5%,1.27x)前后——再次证明决定因素是 harvest 不是任务类型;report 恒快只因它的 harvest 全部 ≥36%(中位 62.6%,加速中位 1.89x)。
- 表内唯一的大离群 i23(harvest 18.3% 却 2.56x):两次运行输出 1218 vs 9242 字符,单条不可当证据(见 §六勘误)。
