## TL;DR

- GPU 不会自己干活,每个计算任务(kernel)都要 CPU 通过**驱动**下单。`cuLaunchKernel` = **逐道报菜**(一次下一道);`cuGraphLaunch` = **递套餐单**(预先录好几百道菜,一次递一张单)。
- 套餐单在录制时**份量定死** → vLLM 预录一排固定尺寸(1,2,4,8,16,24,…,512 个 token)= **"桶"**;实际批量落在两桶之间就**向上凑整**(pad)。"桶形状"= 套餐的固定份量规格。
- 开 suffix 投机后,注意力形状每步变 → 被踢出套餐 → 一步之内**套餐单和逐道报菜交错 ~128 次**(64 层×2)。**收银台(驱动)在两种下单方式之间反复切换时,内部一把锁把报菜的服务员挂起**——实测这笔等待 **4.5ms/步**,是混合模式税 8.1ms 的主体。
- 治法不是修锁(锁在闭源驱动里),是**别换班**:全套餐(不开投机)或全逐道(`--enforce-eager`)。

## §1 GPU 不会自己干活:kernel 与"下单"

先立一个事实:**GPU 是个只会埋头做菜的后厨,自己不会看菜单**。CPU 上的程序每要 GPU 算一件事(一次矩阵乘、一次 norm、一次 softmax……),都得给它下一张单。这一件事在 CUDA 里叫一个 **kernel**(一道菜)。

一步 decode(吐一个 token)要算:64 层 × 每层(norm、QKV 投影、注意力、输出投影、FFN 两个大矩阵乘……)≈ **几百道菜**。谁来下这几百张单?CPU 上有个**发射线程**(服务员),挨个下。

**桥句(全文反复用):GPU 做菜快不快是一回事,单下得顺不顺是另一回事——本文讲的税全在"下单"环节,不在"做菜"环节。**

## §2 cuLaunchKernel:逐道报菜

`cuLaunchKernel` 是 CUDA **驱动**(NVIDIA 提供的闭源库 `libcuda.so`,相当于收银台)暴露的下单接口:服务员每喊一次,收银台记一笔账、把一道菜的工单塞进后厨的传送带。**一次一道**,这种逐道下单的模式叫 **eager**(现做现点)。

单次手续费不贵(正常几 μs),而且 decode 时后厨本来就在等冰箱取料(显存带宽瓶颈,GPU 大部分时间在等权重数据),服务员只要下单速度跟得上,**手续费会被后厨的等待掩盖**——这就是为什么**全 eager 其实不慢**:实测每步只比全套餐慢 0.5ms(24.5 vs 24.0ms,五常数表的②)。

## §3 cuGraphLaunch:递套餐单(CUDA graph)

但"跟得上"有条件。如果 CPU 忙(Python 解释器、调度逻辑挤占服务员时间),几百道菜逐个喊就可能喊不赢。CUDA 给的解法:**CUDA graph**——把一步 decode 的几百次下单**预先录制**成一张"套餐单"(录制 = capture),之后每步只需 `cuGraphLaunch` **递一次单**,后厨照单连续做完,全程不用服务员再开口。

vLLM 默认给 decode 用它(server 启动时那几分钟 "Capturing CUDA graphs" 就是在录套餐)。纯套餐模式 = 五常数表的①,24.0ms/步,最快。

## §4 套餐的代价:份量定死 → "桶形状"

套餐单是**录像**:里面每道菜的份量(张量形状——这一步处理几个 token、几条请求)在录制时**写死**了。运行时来的批量必须和录的完全一样才能直接回放。

批量每步都可能不同怎么办?vLLM 预先录**一排不同份量的套餐**:1、2、4、8、16、24、32、…、512 个 token(server 日志 `cudagraph_capture_sizes` 里那串数)。这一排固定规格就是**"桶"(bucket)**,**"桶形状" = 每个预录套餐对应的固定张量形状**。运行时实际批量 N 落在两桶之间 → **向上取整到最近的桶**,差额用假 token 填上(**pad,凑份量**)——凑的部分照算照付钱,但换来"有现成套餐可用"。

> **"桶抖动"顺带就懂了**:suffix 草稿长度每步变 → 每步落进**不同的桶**,pad 浪费忽多忽少、套餐来回换——这就是税表里"C(k24 混合)比 C0 还多 8.5ms 桶抖动"的意思;也解释了为什么 **pad 参数在混合模式赚(把形状钉在一个桶上,+2ms)、全 eager 下倒亏**(没有桶,pad 纯属多做菜)。

## §5 为什么会出现"混合模式"(一顿饭两种下单方式)

到这里是两个纯模式:全套餐(快)、全逐道(几乎不慢)。**混合模式是被投机逼出来的**:

1. 开投机后,**注意力**这道菜的形状没法预录——每个请求这一步验证几个草稿 token(1+k,k 每步变),注意力的输入形状跟着变,套餐录不了;
2. vLLM 的折中:**把注意力从套餐里挖出来现场点,其余(norm/QKV/FFN)仍用套餐**——编译时按 `splitting_ops`(注意力算子)把整图切开,每层被切成"注意力前半段"和"注意力后半段"两个图段;
3. 于是一步 decode 变成:**递图段套餐单 → 逐道报注意力的菜 → 再递下一个图段 → 再报菜 →……64 层下来 ~128 次换班**。

## §6 收银台里那把锁:换班为什么贵

现在回答主问题。`cuLaunchKernel`(报菜)和 `cuGraphLaunch`(递套餐单)**是同一个收银台(驱动)的两个窗口**,共享收银台内部的账本(设备状态、命令队列)。两种下单方式**交错**出现时,收银台为了账本一致,内部会**串行化**:一种方式的提交还没走完流程,另一种方式的服务员就得**挂起等待**——具体动作是驱动把线程停在一个**条件变量**上(系统调用 `pthread_cond_timedwait`,"在椅子上坐等叫号")。

**证据(逐事件记录,非猜测)**:nsys 的 OSRT 追踪记下了每次等待的调用栈——混合模式(C0)下,发射线程 **50,184 次、共 13.41s** 的 cond 等待,栈都穿过 `cuLaunchKernel`(= 在报菜函数**内部**被挂起等锁);**全 eager 对照同一查询 0 次**。换算(÷4 个 TP worker、÷765 步)≈ **4.5ms/步**。复现命令与物证文件见姊妹篇 [§三b](mixed-mode-tax-explained.md)。

**为什么只能认命**:锁在 `libcuda.so` 里,闭源,"谁持锁、为什么这样设计"不可见——但**不必见**:可操作结论已经完备,**别让两种下单方式交错**(全套餐或全逐道),锁等待就归零。这正是 `--enforce-eager` 修复的全部原理。

均摊一下有个直觉:混合税 8.1ms ÷ ~128 次换班 ≈ **63μs/次**(其中锁等待 ~35μs/次)——单次不起眼,一步 128 次、每步都交,就是 8.1ms。(⚠️ 这是**均摊口径**;逐事件的真实分布是"每步 ~16 次撞上、单次中位 272μs",见追问区 Q8 修订。)

## §7 手把手:一步 decode 的换班时序(单步显微镜)

**先认人**:服务员 = CPU 发射线程;收银台 = CUDA 驱动(libcuda);后厨 = GPU;套餐单 = 预录的图段;报菜 = cuLaunchKernel。

教学用 **3 层**演示(真实 **64 层**;费用取均摊值 63μs/次,真实逐次有 37.6μs 重尾抖动):

| 时刻 | 服务员动作 | 下单方式 | 边界费(换班) |
|---|---|---|---|
| t0 | 递"嵌入+第1层前半"图段 | cuGraphLaunch | — |
| t1 | 报第 1 层注意力的菜(若干道) | cuLaunchKernel | **换班① ~63μs**(部分时间挂在锁上) |
| t2 | 递"第1层后半+第2层前半"图段 | cuGraphLaunch | **换班② ~63μs** |
| t3 | 报第 2 层注意力 | cuLaunchKernel | 换班③ |
| t4 | 递"第2层后半+第3层前半"图段 | cuGraphLaunch | 换班④ |
| t5 | 报第 3 层注意力 | cuLaunchKernel | 换班⑤ |
| t6 | 递"第3层后半+输出头"图段 | cuGraphLaunch | 换班⑥ |

- 3 层 = 6 次换班;**真实 64 层 ≈ 128 次 × 63μs ≈ 8.1ms/步** ✓(和税表④对上);
- 与此同时后厨(GPU)在正常做菜(读权重 ~24ms/步)——**税不是后厨慢了,是服务员在收银台被挂起**,挂久了后厨传送带出现空档;
- 对照:**纯套餐**一步只有 t0 一次递单,零换班;**全逐道**一步几百次报菜但从不换方式,锁等待为 0,手续费被后厨等待掩盖(仅 +0.5ms)。

## 追问区(预判的坑)

**Q1:为什么全 eager 几百次下单反而不慢,混合才 128 次换班却慢?**
下单**次数**不是税,**换方式**才是。eager 的每次报菜手续费几 μs 且被 GPU 等显存的时间掩盖;混合的每次换班要过驱动的串行化关卡(锁),35-63μs 一次且**不可掩盖**(服务员被挂起,后面的单全堵住)。

**Q2:`cuGraphLaunch` 递单本身贵吗?**
比一次报菜贵、但比逐道报完一整段便宜。混合模式下每步 ~65 次递单共 **1.4ms**(税表里"图段启动"那笔);纯套餐模式一步只递 1 次,可忽略。

**Q3:锁到底是谁?能不能修?**
在闭源驱动里,栈只能看到 `cond_timedwait ← libcuda 内部帧 ← cuLaunchKernel`,再深没有符号。修不了也不必修——绕开(别交错)即可,这是工程判断不是妥协:换驱动版本/等 NVIDIA 优化都不如一个 `--enforce-eager` 可靠。

**Q4:桶和 pad 到底什么关系?**
桶 = 预录套餐的固定份量;pad = 实际批量凑到最近的桶。**只有用套餐(graph)时 pad 才有意义**;全 eager 没有桶,pad 就是纯多做菜(实测倒亏 3.5ms,见姊妹篇 §五)。

**Q5:ngram 投机也混合,为什么不慢?**
它**同样交**混合税(每步 31.4 ≈ C0 的 34.0),但每步接受的 token 多,摊薄后净赚——"交税"和"亏钱"是两回事(姊妹篇 §四第 4 步)。

**Q6:eagle3 也投机,它交这笔税吗?**
不交这把锁的税——eagle3 的草稿数固定(num_spec=k 恒定),verify 形状稳定,能留在套餐体系里(cudagraph FULL_AND_PIECEWISE 正常捕获)。它的问题是另一本账(drafter 前向费,见 [../eagle-idea.md](../eagle-idea.md) 加速实测)。

**Q7(读者追问):"串行化等待"为什么费时?eager 不也是串行的吗?——eager: `cuLaunchKernel×N` 连发,混合: `cuGraphLaunch+cuLaunchKernel` 交替,而且递套餐单比逐道报菜省,混合按理应该比 eager 更快才对?**

你的两串式子写得完全对,坑在**"串行"是两个不同的意思**:

- **eager 的串行 = 顺序下单但从不停顿**。每次 `cuLaunchKernel` 是**异步**的:喊一嗓子(几 μs)立刻返回,继续喊下一嗓子——服务员一直在干活,单子在后厨传送带上排起长队,GPU 永远有活干。手续费虽然 ×几百次,但**全程与 GPU 做菜重叠**(GPU 反正在等显存),所以净代价只有 +0.5ms。
- **混合的串行 = 服务员被强制睡觉**。每次换班,驱动把发射线程**挂起**在条件变量上(`pthread_cond_timedwait` = 线程进内核态睡眠,等叫醒还要过操作系统调度器)。睡着的服务员**什么单都下不了**——传送带断供,GPU 排队的活干完就干等(气泡)。这 35-63μs/次**无法与做菜重叠**,和 eager 那种"边喊边做"本质不同。

再算你说的"混合应该更快"那笔账,数字放进去就翻了:

| | 省下的 | 多付的 |
|---|---|---|
| 混合 vs eager | 几百次报菜手续费 → 但这笔本来就被 GPU 等显存**掩盖**,实际只省 ~0.5ms | 128 次换班 × 35-63μs = **4.5-8.1ms,不可掩盖** |

**省的是一笔"本来就不用真付"的钱(已被重叠掩盖),多付的是一笔"必须干等"的钱**——所以混合比 eager 慢 ~8ms/步。"递套餐单比逐道报菜省"只在**纯套餐**模式下兑现(一步只递 1 次、零换班,24.0ms 全场最快);一旦要和报菜交替,换班罚款把省的全吞了。实测三点闭环:纯套餐 24.0 < 全 eager 24.5 < 混合 34.0(C0)。

补一个实锤细节:那笔换班罚款在 nsys 里的样子,就是混合模式下的 `cuLaunchKernel` 调用**平均变慢到 24.8μs、重尾 37.6μs**(正常几 μs)——慢的不是"发射工作"变多,是**发射函数内部在等锁**(栈:`cond_timedwait ← libcuda ← cuLaunchKernel`);全 eager 下同样的 `cuLaunchKernel` 没这笔等待。

**Q8(读者追问):63μs 是花在两个 API 的"切换"上?"挂起"是不是指 cuGraphLaunch 发起后、等它算完、再发 cuLaunchKernel,这一等 63μs?**

**不是等图段算完**。先钉前提:两个 API 都是**异步**的——都只是把活塞进 GPU 排队队列就返回,谁都不等 GPU 算完;正常时 CPU 一路排单、GPU 在后面追着干。混合模式坏掉的是:**紧跟在 cuGraphLaunch 之后的那次 `cuLaunchKernel`,函数本身变慢了**(正常 ~5μs 返回 → 混合下平均 24.8μs、重尾 37.6μs)——多出的时间是线程在**函数内部**撞上驱动锁、被操作系统睡眠。

- **"挂起"的精确含义**:`pthread_cond_timedwait` = 线程对 OS 说"睡我,有人叫号再喊醒"→ 线程下 CPU 核 → 条件满足后被**同进程的驱动内部线程 `pthread_cond_broadcast` 叫醒**(OSRT 实证:每 worker 等待 1.1-1.36 万次 vs 广播 1.17-1.43 万次,计数 1:1、线程 ID 不同)→ 把单下完返回。
- **逐事件真相(2026-07-09 修订,推翻本节旧反证)**:等待**不是**每次边界都发生——每步每 worker 只 **~16 次**,但单次**中位 272μs**(p90 346μs),16×272μs ≈ 4.4ms/步 ✓。272μs 与图段执行时长(~325μs)**同量级** → "等在途图块被 GPU 消费"类**流控机制不能排除**(旧文写"35μs≪325μs 故不是等图段"用的是均摊数,逐事件层不成立,收回)。能排除的是"每图段必等完成"(那样 65 段×325μs≈21ms/步,实测只 4.4ms)。**"63μs/次换班"是均摊口径**(8.1ms÷128 边界),逐事件是"少数边界撞上流控、每次等约一个图段的排空时间"。
- **完整时序(一次换班)**:

```
CPU(服务员): [递图段i]──[报注意力菜: 函数内睡35μs💤→下单]──[递图段i+1]──…
              ↑异步,μs级返回   ↑"挂起"发生在这里(函数内部)
GPU(后厨):  …[执行图段i-1][执行图段i][ 气泡~130μs ][执行注意力]…
                                       ↑CPU 停顿累积→队列断粮→真空转
```

- **两头账对上**:CPU 侧 128 次 × 63μs ≈ 8.1ms;GPU 侧 ~130μs/层真气泡 × 64 ≈ 8.3ms——同一笔税的两个侧面(CPU 停顿是因,GPU 断粮是果)。

**Q9(读者追问):所以 eager 是 CPU 连续发、发了就丢给 GPU;混合是 cuGraphLaunch 发完后"挂起、等一段时间"再发 cuLaunchKernel?驱动为什么要这样?**

先纠正时序:CPU **没有主动等**——发完 `cuGraphLaunch`(快速返回)后**立刻**调用了 `cuLaunchKernel`,是**这次调用陷在驱动函数内部**35μs 才返回(线程被驱动挂起)。从外面看像"隔了一段才发",实际是"发了,但卡在函数里"。

**驱动为什么要挡**(机制推断,符合全部观测的最简解释;闭源无法直证):两个 API 底下是两种提交方式——
- 逐 kernel = **往小票上续写**:驱动为 stream 维护一条命令环形缓冲(pushbuffer),每次 launch 追加一小段,连续追加是无争抢快路径(~5μs);
- graph = **插入一整本预印菜单**:录制时已烤成整块命令缓冲,launch 时把整块挂进设备队列——另一种记账方式。

同一 stream 上交错时,驱动要保证"小票→整本→新小票"顺序与上下文状态一致。结合逐事件分布(每步仅 ~16 次等待、单次中位 272μs ≈ 一个图段的执行时长,见 Q8 修订),**最贴合数据的机制是流控**:追加路径(报菜)偶尔要**等在途的图块被 GPU 消费、腾出提交通道/令牌**才能继续——等待次数少、单次时长 ≈ 图段排空时间。**纯模式都不交税的原因**:连续续写不占图块通道、连续插整本驱动内部自流水——只有"换方式"制造这种通道争用(解释三个实测事实:税仅混合有、eager 为 0、纯 graph 为 0)。**为什么睡眠不自旋**:等待窗口上百 μs,自旋烧 CPU 不划算,驱动 park 线程;睡着期间零提交 → 不可掩盖。

**什么时候被唤醒**:同 worker 进程里**另一个驱动内部线程**在通道腾出后 `pthread_cond_broadcast` 叫号;`timedwait` 自带超时兜底。**事件级配对实锤(2026-07-10 补)**:抽样 8 个稳态等待,**每个结束前 3.7-10.1μs 同进程必有一条 broadcast**(约 6μs = 叫号→调度器捞回→函数返回的唤醒延迟);且配对**稳定一对一**——每个 worker 的发射线程(TID 如 1003856)有专属的驱动服务线程叫醒它(TID 如 1003995,比发射线程大 ~120-190,libcuda 自起)。GUI 里看:broadcast 是 ~2μs 细刺,要在专属服务线程的 OSRT 泳道、等待结束前 ~6μs 处、放大到 ~20μs 视野才可见——500μs 视野下宽度仅 1/250,肉眼找不到属正常。

实锤/推断分级:等待在 launch 函数内部(栈)✅、仅交错时发生 ✅、唤醒者=同进程驱动线程(计数配对)✅、等待时长分布 ✅;"流控/通道令牌"具体机制 🔶 推断(闭源)——可操作结论(别交错)不依赖推断部分,已被 Ceager 实测验证。

|---|---|
| kernel | GPU 上的一个计算任务(一道菜) |
| eager | 逐 kernel 现场下单的执行模式(逐道报菜) |
| `cuLaunchKernel` | 驱动的逐道下单接口 |

| `pthread_cond_timedwait` | pthread 条件变量等待:原子地"放锁+睡觉",直到别的线程 signal/broadcast 叫号或超时自醒;睡着期间不占 CPU、零提交。时间线上的"块"= nsys 记录的这次调用[进入,返回]区间,块长=睡了多久;**嵌在 cuLaunchKernel 块正下方**=睡在发射函数内部(有害),出现在闲置线程上(等活)则无害 |
