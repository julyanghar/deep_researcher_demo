# suffix 与 vLLM 原生解码的每步耗时解剖(A vs C0 vs Ceager)

> **目标一句话**:suffix 投机相对原生 decode,每步多花多少、每一毫秒花在哪。
> **为什么只比这三个**:A = 原生锚点;C0 = suffix 的默认执行形态(graph+eager 混合)、空草稿隔离纯机制费;Ceager = suffix 的全 eager 形态、带真草稿。C(k24 混合)不在主表——它 = C0 + 8.5ms 桶抖动(变长草稿×混合的交互项,见 [how-we-found-the-fixed-tax.md](how-we-found-the-fixed-tax.md) §7.3),属于"配置自找的",不是 suffix 与原生的本质差。
> **口径**:TP4-40 replay、batch=1、summary 段、每步 ms(n_out=步数已核实,见 how-we-found 追问区);A/C0 与 ctx 无关,Ceager 含 ctx 斜线(注明)。数据截至 7/5 深夜;标"待测"两格由夜间链(C0piece / C0eager)回填。

## 一、三配置定义

| | 投机 | 草稿 | 执行方式 | 每步实测 |
|---|---|---|---|--:|
| **A** | 关 | — | 纯 CUDA graph(整模型一张图) | **24.0ms**(恒定) |
| **C0** | suffix 开 | 恒为空(max_spec_factor=0) | graph+eager 混合(FFN 在图、注意力 eager——spec 配置的默认形态) | **34.0ms**(恒定,ctx 无关) |
| **Ceager** | suffix 开 | k24 变长(平均 ~2.6 个/步) | 全 eager(`--enforce-eager`) | 短 ctx **~30.5** + ctx 斜线 → 中位 33.9 |

## 三、逐项成分解析

**① 基础前向 23.5ms——物理底价,不可优化。** decode 是显存带宽瓶颈:每步把 32B 权重(TP4 每卡 1/4)从显存读一遍。同模型同卡,三配置一样。

**② CPU 发射/启动(真工作,0.5~2.5ms)——三配置差 ≤2ms,不是主角。** A 每步只说一句"回放整图"(428μs)+零星 kernel;C0 每层"启动一段 FFN 图(21μs)+手发 3-4 个注意力 kernel"×64;Ceager 手发 ~760 个但单件恢复 3μs 流水速度。发射**数量**从来不是税(Fea 发 350 万/25s 也不慢,§5.4 老结论)。

**③ 驱动锁等待 4.5ms——C0 独有的大病,本表最大单项。** 发射线程停在 `cuLaunchKernel` 内部、CUDA 驱动私有条件变量上(调用栈实锤:`pthread_cond_timedwait ← libcuda ← cuLaunchKernel`;每步 ~36 次、共 4.5ms)。机制:**graph 段提交与 eager 发射交错时,驱动内部强制串行化**,发射线程被挂起。全 eager 对照精确为 0.00s——**不交错就不收费**。所谓"慢发射 37.6μs 重尾"的本质是等锁不是干活。治法:全 eager(已实测)或全图(需改 vLLM 让 spec 注意力入图)。

**④ launch 外 cond 等待 1.2~1.9ms——两家都有的小账。** 关键线程在发射之外停在条件变量上(GIL 切换/跨线程队列交接类)。具体调用栈归因由 step1(C0eager 全开采样)给出。优化空间存疑(GIL 类基本没得治)。

**⑤ propose+RPC 0.2~0.35ms——suffix 的"聪明"很便宜。** 查后缀树本身极快(C0 空 speculate 仅 0.03ms);跨进程取草稿 0.15ms。早期怀疑的"降税三件套"(增量匹配/C++ 下沉)优化的就是这项——无肉可砍,已判死。

**⑥ sampler+metadata 0.9~1.26ms——投机公共机件。** rejection sampler(逐位验证)+ spec metadata(为每个位置排 logits 索引),CPU 侧 Python 逻辑。带草稿口径 1.26(0.82+0.44);零草稿裸值今晚 C0piece 直测(表格闭合预测它 ≈0.9)。注意这是**任何** vLLM 投机方法都有的账,不是 suffix 特有。

**⑦ 异步被关 0.5ms——结构性但便宜。** vLLM 白名单只给 GPU 张量化的投机开异步调度,suffix(CPU drafter)被强制关。batch=1 直测只值 0.5ms(D vs E;ngram 全 eager 下甚至为 0)。并发场景的价值待 e2e 复核,但量级判断:小。

**⑧+⑨ 草稿验证的按量费——eager 下"近似免费"的精确版。** 每个草稿位置搭权重读的便车,边际单价 ~0.4ms/位置(pad 强制 8 槽 +3.5ms 反推);短 ctx 平均 2.6 个位置 ≈0.75ms,看不见。**但单价随 ctx 涨**(每个位置要读一遍该请求的 KV):50k 字符的 summary 每步要多付 4~6ms(⑨斜线)——这是"长 prompt summary 亏损"的机制,治法是**策略**(ctx 感知/自适应关投机),不是系统改造。

## 四、结论与优化含义

1. **suffix − 原生的本质差**(去掉配置自找的桶抖动后)= ③④⑤⑥⑦⑧ 六项 ≈ **+6~10ms/步**,其中唯一的大项(③4.5ms)**已被全 eager 消掉**;
2. **当前推荐形态 Ceager 的剩余账单**:七笔小账(每笔 ≤2.2)+ ctx 斜线。小账里可再抠的只有 ⑥(投机公共机件,可向 GPU 下沉)和 ⑤⑦(合计 <1);**斜线要靠策略治**(长 ctx 少押注),这与"头几步 miss 关投机"是同一个自适应开关;
3. **盈亏公式**(为什么这些成本值得付):净收益 = 每步接受数 ÷ (1 + 每步附加费/24) − 1。Ceager 短 ctx 附加费 ~6.5 → 盈亏线 = 每步接受 1.27;冷树 summary 1.43 微赚,report 3+ 大赚,热树 e2e 全面翻正。
