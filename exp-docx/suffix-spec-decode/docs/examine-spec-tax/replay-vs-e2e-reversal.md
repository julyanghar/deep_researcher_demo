# 为什么 summary 投机"replay 低效、e2e 翻正"——别全甩锅给全局树

> **缘起**：一个反复被误读的说法——"replay 下 summary suffix 效率不高（0.83x），到 e2e 就翻正（1.02x），根本原因是全局树"。**这话对一半**：全局树是原因之一，但既不是"低效"的病根，也不是翻正的唯一原因。本文把两件被混在一起的事拆开。
> **一句话**：**"低效"的病根是混合边界税（执行模式），`--enforce-eager` 治它、和树无关、replay 冷树下就生效**；"同配置 replay→e2e 翻正"才是树参与的地方，且是"热树抬接受 + 并发摊薄"两者合力，树只占温和的一半。

## 一、先把两件事分开（这是关键）

"replay 0.83x → e2e 1.02x"其实是**两个不同的问题**被叠在一句话里：

| | 问题 A：为什么 summary 低效？ | 问题 B：为什么同配置反转？ |
|---|---|---|
| 比的是什么 | C(混合) vs 原生 A | 同一个 C，replay vs e2e |
| 根本原因 | **混合边界税 8.1ms**（执行模式） | 热树 + 并发（合力） |
| 和树有关吗 | **无关** | 部分有关 |
| 干净解法 | `--enforce-eager`（去边界税） | —— |

## 二、问题 A：低效的病根是混合边界税，不是树

**铁证**：推荐配置 **Ceager（suffix + `--enforce-eager`，去掉混合边界税）在 replay 冷树下就已经是 1.00x**——不需要 e2e、不需要热树，光换执行模式就翻正了。

所以"summary 投机低效"的病根是那 8.1ms 的混合边界税（graph+eager 逐层切换、主体是驱动锁等待），**治它的是 eager，跟全局树一点关系没有**。把低效归因于"树不够热"是找错了病。

（成本模型见 [fixed-tax-conclusions.md](fixed-tax-conclusions.md)：每步 = 基线 24 + eager 0.5 + spec 管线 1.9 + 混合边界 8.1 + 草稿费。Ceager 消掉 8.1。）

## 三、问题 B：同配置反转 = 热树 + 并发，两者合力

如果坚持用**旧混合配置 C**（不换 eager），它确实从 replay 0.83x 反转到 e2e 1.02x。这才是树参与的地方，但有**两个原因**：

### 原因① 热树的贡献：温和，而且不是"抬接受率"（2026-07-06 决定性实验修正）

**先纠正本文初稿的错**：初稿写"热树抬接受 +22%（每步 token 1.39→1.70）"——**两处都不准**。决定性实验（接受率 vs 完整兄弟数，同轨迹口径）实测：

| 完整兄弟数 | 接受**率** | 每步草稿 | 每步 token |
|---|--:|--:|--:|
| 0 | 29.2% | 2.06 | 1.60 |
| 3-5 | 31.5% | 2.42 | 1.76 |
| 10+（满预热） | 29.5% | 2.55 | 1.75 |

- **接受率对兄弟数完全平**（Δ+0.3pp）——热树**不抬接受率**；
- 兄弟只是让**草稿数**变多（+24%，料多→匹配长→预算大），这些多草稿在同样 29% 率上命中 → 每步 token 温和涨 **+9%**（1.60→1.75），**3-5 个兄弟即饱和**；
- 初稿的"1.39→1.70"里，兄弟只占 +9%，其余是工作负载差异（被我全算给了树）。

> **重要澄清**：更早那个"热树让接受率翻倍（26.4%→42.5%）"是 v2 手工切时间窗混进 report 的**测量假象**；干净轨迹口径 e2e summary 接受率 28.3%，与冷树 ~29% 几乎一样（[questions-log-2.md Q3](../questions-log-2.md)）。**树对接受率的贡献 ≈ 0，对每步 token 的贡献 +9%（经更多草稿）且早饱和。**

> **关全局树对照(cache_ab 干净轨迹口径复核 7/1，2026-07-06)**：同 204 条 workload、都 enforce-eager、断言 cache=0/10000 生效。**关/开全局树接受率几乎相同(OFF 30.1% vs ON 30.0%，Δ0.1pp)**；差别全在草稿量——**开树每步草稿 2.38 vs 1.77(+34%)→ 每步 token 1.71 vs 1.53 → char/s 87 vs 80(+9%)**。
> **订正**:7/1 旧说法"关全局树接受率反而更高(50%)、开树略高"是 **server 窗口口径的测量假象**(与 26.4→42.5 那个同类),干净轨迹口径下**接受率两边平**。**正确结论(不变):全局树抬的是草稿数量(投机音量)、不抬接受率**——与兄弟实验(率平、草稿涨)、census(28.3%)三方一致,`~30%` 接受率是内容天花板、树组织方式顶不动。物证:`~/modify-code-runs/suffix-spec-decode/tax-split/{res_cache_ON,res_cache_OFF}.jsonl` + tax_status.txt。

### 原因② 并发摊薄（另一半）

e2e 题内并发 3——每步的固定税是**每步常数**（不随 batch 涨），并发把它摊到多个请求的产出上，每 token 税 ÷ 并发数。nsys batch 扫描已证：C 与 Ceager 的每 token 差随 batch 从 8.1ms 缩到 4.5ms。这一半和树无关。

## 四、结论

- **想让 summary 投机高效** → 用 `--enforce-eager`（去混合边界税），**replay 冷树下就生效，和树无关**；
- **"旧配置到 e2e 会翻正"的现象** → **主要靠并发摊薄**；热树的贡献很小（每步 token +9%，经"更多草稿"非"更高率"，且 3-5 兄弟即饱和），**接受率对兄弟数完全平**（决定性实验，Δ+0.3pp）；
- 树的贡献被早期"接受率翻倍（26.4→42.5）"的说法严重夸大——那是测量假象，干净口径 summary 接受率 28.3% ≈ 冷树 29%。

**一句话**：低效怪的是执行模式（混合边界税），翻正主要靠并发摊薄 + 换 eager；全局树对接受率贡献 ≈ 0、对每步 token 贡献 +9% 且早饱和。把功劳全给全局树，既高估了树、又错过了真正的杠杆——`--enforce-eager`。
