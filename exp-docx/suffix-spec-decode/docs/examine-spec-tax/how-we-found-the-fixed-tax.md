# 固定税破案记:开投机后每步凭空多花的 10ms,到底花在哪?(零基础·修订版)

> **⚠️ 修订声明(2026-07-05)**:本文旧版把那 455μs"空隙"当成"GPU 纯空转",并推出"每步 29ms GPU 空转 = 固定税"——**这是错的**。补测 ngram 投机 + 两者的 eager 版本后发现:①那 455μs 里大头是 **CUDA graph 捕获的 FFN 在跑**(nsys 没记录而已),不是空转;②ngram 投机**同样有这个 gap 却比原生还快**,所以"有 gap"根本不是慢的根因。真正的固定税是 **graph+eager 混合模式的逐层边界管理**,而且它**只狠咬 suffix、不咬 ngram**。本文按订正后的结论重写,并把踩过的坑原样留在破案主线里(坑越明,结论越结实)。

> **缘起**:开 suffix 投机解码后,每个 decode step 会**凭空多花 ~10ms**——哪怕这一步一个草稿都没猜(空草稿),照样多花。这笔"税"每步都交、和猜没猜中无关,是 summary 不加速的根子。本文从零讲清:我怎么像破案一样,用一串实验把这 10ms 逼到"**graph↔eager 逐层边界管理**",以及为什么同样是混合模式,ngram 不用交这笔税。
> **读者起点**:你懂 LLM 推理基础(decode 一步出一个 token、attention、KV cache、decode 是显存带宽瓶颈),但**没做过 GPU 性能剖析**(不知道 kernel launch、CUDA graph、nsys 是什么)。这些本文都会从头讲。
> **读法**:§一~§四 是概念梯子(每节一个新词,顺序读);§五 是破案主线(嫌疑人排除表);§六 是"455μs 到底是什么"的定案与订正;§七 是核心——用 ngram vs suffix 的 2×2 对比锁定真凶;赶时间看 TL;DR + §七那张表。
> **分工**:实验部署/口径见 [tax-split-experiment-plan.md](tax-split-experiment-plan.md);结果速览见 [tax-split-progress-report.md](tax-split-progress-report.md);本文讲"怎么破的案、为什么信"。

## TL;DR(订正版)

decode 慢在**显存带宽**,GPU 算力大量闲着。**CUDA graph** 把一步内几百个 GPU 操作录成一次回放,GPU 连续跑不等 CPU;vLLM 默认给 decode 开着(原生 A 就用它,一整个模型录成一张大图)。**开投机后,注意力被踢出 graph 变成 eager,而 FFN/norm 仍留在 graph 里——形成"graph+eager 混合":每一层都要在"回放 FFN 图"和"手工发注意力"之间来回切一次。**

在 nsys 时间线上,这表现为注意力流上每层一个 **455μs 的空隙**。旧版误以为这是 GPU 空转——**其实这空隙里绝大部分是 graph 捕获的 FFN 在跑(nsys 默认不记录 graph 内部 kernel,所以看着像空的)**。真正多花的税,是**混合模式每层多出的那 ~100μs 边界管理开销**(≈每步 6-10ms),不是 455μs。

**为什么锁定是"混合边界管理"而不是别的**,靠两个对照:
- **全 eager 版(Ceager)**:把 FFN 也踢出 graph、全程手工发,消掉了 graph↔eager 边界——suffix 立刻从 0.83x 翻回 1.00x(summary)、1.85x 冲到 2.29x(report)。**所以慢的是"混合",不是"eager";eager 本身反而更快。**
- **ngram 投机**:结构上同样是 graph+eager 混合、同样每层有 ~379μs gap、发射同样慢(24.8μs,自家 eager 的 7.5 倍),**却比原生还快(1.03x)**——(二次修正)它**不是免税,是税照交、靠每步多吐 token 摊薄赚回**;suffix 额外多付两笔:变长长草稿搅 graph 桶(k24 +8.5ms/步,最大单项)+ 自家 CPU 机件 ~1.6ms(异步调度本身经 D/E 直测只值 0.5ms)。

一句话(修正版):**离开纯 graph 谁都要交 ~7-10ms/步的执行税(混合边界慢发射);投机靠每步多吐 token 摊薄。suffix k24 混合雪上加霜:变长草稿搅桶 +8.5、自家 CPU 机件 +1.6,摊不回来 → 0.83x;全 eager 消掉搅桶、发射恢复 3μs 快速 → 平手/大赚(完整账本见 §7.3)。**

## 一、先立靶子:"每步固定税"到底指什么

我们比三种服务器,同一批请求,量**每个 decode step 的平均耗时**(实测,summary 段,TP4):

| 配置 | 每步耗时 | 说明 |
|---|--:|---|
| **A** = 不开投机(baseline) | **24.0ms** | 正常 decode,一步出一个 token |
| **C** = 开 suffix 投机 | 42.5ms | 每步猜多个草稿,含"猜对省的"和"多验证的",混着 |
| **C0** = 开 suffix 但**永远空草稿** | **34.0ms** | 投机机制全开,但一个草稿都不猜、不验证 |

C0 是关键"对照组":投机机制全打开(所有开销都在),但**草稿数=0**,前向和 A 一模一样大(都只处理 1 个 token)。**可它每步还是比 A 多花 34−24 = 10ms。**

> **桥句 1:C0 空草稿仍多花 10ms —— 这 10ms 与"猜没猜、验没验"无关,纯粹是"投机机制开着"本身的代价。这就是"每步固定税",本文要破的案。**

(为什么用 C0 破案:C 混着"猜对省的时间"和"多验证的时间"算不清;C0 把两者归零,剩下的纯粹是机制开销。控制变量。)

## 二、第一个新词:kernel 和 "kernel launch"(CPU 派活给 GPU)

GPU 不自己决定算什么。**每一个 GPU 计算动作(一次矩阵乘、一次 softmax、一次搬运),都是 CPU 程序"派"给 GPU 的一个任务,叫一个 kernel。**

- **kernel**:GPU 上跑的一个计算函数(比如"算这一层注意力")。
- **kernel launch(发射)**:CPU 执行一句 `cudaLaunchKernel(...)`,意思"GPU,去跑这个 kernel"。这是一次 **CPU→GPU 派活**。

一个 32B 模型有 **64 层**,每层有注意力、FFN、norm 等算子,每个算子一到几个 kernel。所以**一步 decode,CPU 要发几百个 kernel**给 GPU。

> **桥句 2:发射是"CPU 说一句、GPU 才动一下"。如果 CPU 发得慢,GPU 就得干等。**

## 三、第二个新词:CUDA graph(把几百次派活录成一次回放)

decode 每步都发同一批 kernel、同样顺序(模型结构不变)。每步现场一个一个发,太浪费。

**CUDA graph:把"这几百个 kernel、按这个顺序发一遍"录成一段"图",之后每步只说一句"回放这段图",GPU 自己把几百个 kernel 连续跑完,CPU 不再逐个发。**

比方:
- **eager 模式(不用 graph)** = 你(CPU)站厨师(GPU)旁边做 64 道工序的菜,每道都现场喊一声,厨师做完一步停下等你喊。
- **graph 模式** = 你提前把 64 道工序录成整张工单交给厨师,厨师一口气做完,不回头等你。

> **桥句 3:graph 下 GPU 一步内几百个 kernel 背靠背连续跑(CPU 不插手);eager 下每个 kernel 都要 CPU 现场发、GPU 做完一个停下等下一个。**

vLLM 默认给 decode 开 CUDA graph(A 就用着,**整个模型录成一张大图**)。怀疑点自然落到:**开投机后,是不是 graph 用不上了?**

## 四、第三个新词:GPU 空转气泡(GPU 干等 CPU)

接上你已有的知识:**decode 是显存带宽瓶颈**——每步把 32B 权重从显存读一遍,GPU 算力大部分时间在等数据,真正算的时间很短。

所以正常一步里 GPU 本来就有大量"闲着等显存"的时间。**关键不是"GPU 闲不闲",而是"GPU 闲下来时,CPU 有没有及时把下一个活递过来"**:

- CPU 及时递(或 graph 已排好)→ GPU 等完显存立刻干下一个,不浪费;
- CPU 没及时递(eager 下 CPU 忙别的)→ GPU 等完显存**还得再干等 CPU 发指令**,这段额外干等 = **GPU 空转气泡(bubble)**。

> **桥句 4:气泡 = GPU 已准备好干下一个活,却在干等 CPU 发 kernel 的那段时间。**

## 四b、【本次新增的关键概念】两个"看不见":graph 内 kernel + 混合模式

这两点是订正旧版错误的钥匙,必须先讲清:

**(1) nsys 看不见 graph 内部的 kernel(默认)。** graph 是"一次回放几百个 kernel"。nsys 默认把"回放这张图"当**一个整体**记账,**不把图里那几百个 kernel 逐个记下来**(除非专门开 `--cuda-graph-trace=node`)。后果很坑:一个跑在 graph 里的 FFN,GPU 明明在算,但 nsys 的逐 kernel 表里**查无此人**——时间线上那段看起来像"没有 kernel = 空的",其实 GPU 忙得很。**旧版就栽在这:把"graph FFN 在跑但没被记录"错当成了"GPU 空转"。**

**(2) graph+eager 混合模式(piecewise graph)。** 开投机后 vLLM 不是把 graph 全关,而是**只把注意力这一段从图里挖出来、改成 eager 手工发**,FFN/norm 仍留在图里。于是每一层变成:

```
[回放这层的 FFN/norm 图(graph)] → [手工发这层的注意力(eager)] → [回放下层 FFN 图] → …
```

一层里要在"图回放"和"手工发"之间**切一次**,64 层就切 64 次。**这个来回切,就是本案真凶的藏身处。**(为什么投机非得把注意力改 eager:草稿长度每步变,注意力的张量形状跟着变,而 graph 要求形状固定,所以注意力只能退出 graph。上游机制见 [../../18-vllm-suffix-decoding.md](../../../18-vllm-suffix-decoding.md)。)

> **桥句 5:开投机 ≠ 关掉整张 graph,而是"FFN 留图里、注意力改手工"的混合。nsys 只看得见手工发的注意力,看不见图里的 FFN——这既是后面所有"空隙"的来源,也是最容易误判的陷阱。**

## 五、破案主线:差分实验,逐个排除嫌疑人

**方法叫"差分实验":造两个只差一件事的服务器,量它俩每步耗时之差 = 那件事的成本。** 一次只动一个变量。

### 5.1 嫌疑人排除表

| 嫌疑人 | 用什么实验查 | 结果 | 判决 |
|---|---|---|---|
| **异步调度**(CPU 提前为下一步备料) | A(异步开) vs B(异步关),都不投机 | 差 ~2% | 非投机下只值 ~0.5ms(但投机下另说,见 §七) |
| **propose**(查后缀树猜草稿的 CPU) | 给 propose 加计时器(插桩) | 0.16ms/步 | 太小,排除 |
| **验证/元数据**(rejection sampler 等) | 给这些环节加计时器 | 合计 ~1.4ms/步 | 太小,排除 |
| **CUDA graph 没用上** | F(不投机+强制关graph) vs A | 只慢 0.7ms | 非投机下 graph 只值 0.7ms,存疑(§七翻案) |
| **kernel 发射数太多** | nsys 数 A/C0 各发多少 | C0 是 A 的 18× | 曾误判"就是它",被 Fea 推翻 |
| **发射数量**(复核) | Fea(不投机+eager+异步关)的发射数 | Fea 是 C0 的 7.7×,却不慢 | **推翻**"发射多=税" |
| **注意力流上每层 455μs 空隙** | nsys 逐 kernel 时间线 | 每层一次 455μs | 旧版误判"空转=税",§六订正 |
| **graph+eager 混合边界管理** | Ceager/Deager(全eager)+ ngram 对照 | eager 让 suffix 翻正;ngram 混合却不慢 | **定案(§七)** |

### 5.2 先排除"CPU 环节"(插桩:给代码加秒表)

**插桩** = 在代码关键处塞 `t0=计时(); …; 累计+=计时()-t0`,env 开关控制,不开零影响。四个 CPU 环节加秒表,实测(21600 步累计,换算每步):

| CPU 环节 | 每步耗时 |
|---|--:|
| propose(查树猜草稿) | 0.16ms |
| rejection sampler(验证草稿) | 0.82ms |
| spec metadata(排验证位置) | 0.44ms |
| 取草稿的跨进程通信 | 0.15ms |
| **CPU 环节合计** | **~1.6ms** |

固定税 10ms,能加秒表的 CPU 代码只占 1.6ms。**剩下 ~8ms 不在任何能加秒表的 CPU 代码里**。这本身是线索:秒表量的是"CPU 在执行某段代码的时间",够不着的 = "GPU 在忙/空、而 CPU 那一刻没在跑我们插桩的代码"的时间。要看真相得换 nsys。

> kernel 是**异步**发射的:CPU 发完 `cudaLaunchKernel` 立刻返回去发下一个,不等 GPU 跑完。所以 GPU 时间线上发生什么,CPU 秒表根本不知道。

### 5.3 排除"graph 本身"的错误尝试:F 实验(以及它为什么误导)

嫌疑最大的是"开投机丢了 graph"。先量 graph 值多少:造 F = 不投机 + `--enforce-eager`(关 graph),对比 A:

- F vs A:每步只慢 **0.7ms**。

关 graph 才慢 0.7ms?那"丢 graph"最多值 0.7ms,解释不了 10ms。**这条线索一度让人以为 graph 无关。** 但这是陷阱:F 是"**不投机**下关整张 graph",而 C0 是"**投机**下只挖走注意力"——两者根本不是一回事(§七会看到,投机下 graph 的去留影响大得多)。

### 5.4 转折:nsys 数 kernel,先误判再被 Fea 推翻

**nsys(NVIDIA Nsight Systems)** 是 GPU 性能剖析器:录一段时间里 GPU 上每个 kernel 的起止、CPU 每次发射耗时、GPU 空转间隙。只录自己进程,别人进程看不见。

各录 A、C0 的 25 秒稳态 decode,数 kernel 发射数:

| | A(有graph) | C0(投机空草稿) |
|---|--:|--:|
| kernel 发射数 | 25,059 | **455,943(×18)** |

C0 发的是 A 的 **18 倍**,暴涨的几乎全是 3 个**注意力** kernel(`flash_fwd_splitkv`、`combine`、`reshape_and_cache`)——它们在 A 里 0 次(封在 graph 里没被记录)。**看着石锤:投机→注意力退 graph→逐个发→发射爆炸=税。**

**但这错了。** 造 **Fea = 不投机 + 强制eager + 异步关**(让 baseline 也逐个发,口径对齐 C0),数它:

| | A(graph) | Fea(eager baseline) | C0(投机) |
|---|--:|--:|--:|
| kernel 发射数 | 25k | **3,505,900(×140)** | 455,943(×18) |
| 每步耗时 | 24ms | ~25ms | 34ms |

**Fea 发 350 万个(是 C0 的 7.7 倍),每步只慢 0.7ms;C0 发得少得多,每步反而慢 10ms。**

> **桥句 6:发 kernel 的数量不是税。Fea 发得比 C0 多 7 倍还不慢——它的海量发射被"GPU 等显存"掩盖了(CPU 一边发,GPU 一边等显存,重叠);C0 的某些发射掩盖不住,变成真气泡。** 嫌疑从"发射数量"转向"为什么 C0 有些发射掩盖不住"。

## 六、"455μs 空隙"到底是什么:定案与订正

nsys 能导出**逐 kernel 明细**(`cuda_gpu_trace`):每个 kernel 几点起、几点止。相邻两个 kernel 之间有空隙,就疑似 GPU 空转。取一个 rank(device 0)的主计算流(stream 19)看。

### 6.1 现象:注意力流上,每层一个 455μs 空隙

把一层注意力附近拉出来(C0 实测):

```
相对ns        dur    kernel
…979629      2016    reshape_and_cache(写KV,上一层)
…982285     10624    flash_fwd_splitkv(注意力主体)
…993581      8192    flash_fwd_splitkv_combine
…003213       960    memcpy Device-to-Device
     ┌─────────────  【空隙 455μs,这条流上什么都没有】
…459825      2048    reshape_and_cache(下一层)      ← 456μs 后才出现
```

统计:C0 主计算流上,**每层 reshape_and_cache 之前都有一个 ~455μs 的空隙**(14438 次样本,中位 455μs)。旧版就是在这里下了"GPU 空转 455μs/层 × 64 层 ≈ 每步 29ms 空转 = 税"的结论。

### 6.2 陷阱一:空隙里"所有流覆盖 = 0",像极了空转

进一步查:那 455μs 空隙里,**这个 rank 的任意流**有没有别的 kernel 在跑?实测——**0 个 kernel,总覆盖 0μs**。看起来铁证空转了。

### 6.3 反证:trace 里根本没有 FFN kernel → 空隙是 FFN 在跑

但"覆盖 0"有**两种**解释,必须分清(这是旧版漏掉的一步):
- (a) GPU 真空转,啥也没干;
- (b) GPU 在跑 graph 里的 FFN,但 **nsys 没记录 graph 内部 kernel**(§四b (1)),所以逐 kernel 表里查无此人 → 看着像空的。

判据很硬:**看 trace 里到底有没有 FFN 类 kernel。**

| 配置 | trace 里出现的 kernel |
|---|---|
| A(原生·纯graph) | 只有 memcpy、argmax(采样)、nccl —— **没有任何注意力/FFN/norm 计算 kernel** |
| C0(投机·混合) | 有注意力(flash_fwd/reshape,eager) —— **但没有 rms_norm、silu、FFN 的 GEMM** |

A 里连注意力都看不见(全在大图里)。C0 里注意力看得见(它是 eager 挖出来的),但 **FFN 的矩阵乘、norm 一个都没有**——它们在 graph 里,没被记录。

**算术反证(彻底否掉"空转"):** C0 每步 34ms。主计算流上可见的注意力 kernel 每层才 ~22μs(reshape2+flash10.6+combine8.2+memcpy1),64 层合计**才 ~1.4ms**。如果 455μs×64≈29ms 真是空转,那 GPU 每步只干了 1.4ms 活——一个 32B 模型的一次前向**不可能只要 1.4ms**(原生 A 同样的 FFN 要实打实算 ~24ms)。**这 ~24ms 的 FFN 计算没在 trace 里,只可能藏在那些"空隙"里跑。** 所以那 455μs 里绝大部分是 FFN 在算,不是空转。

> **订正结论:注意力流上每层那 455μs,≈ 图捕获的 FFN/norm/proj 在执行(nsys 没记录)+ 一小段真气泡。旧版"每步 29ms GPU 空转"错在把 FFN 计算算成了空转。**

**直接影像验证(已完成,2026-07-05;当天二次审计修正过一次读数)**:开 `--cuda-graph-trace=node` 重录 C0 和 D(该选项让 graph 内部节点逐个被记录)。结果:C0 的 trace 里冒出 **34,210 个 gemm、8,553 个 silu**——正是之前隐身的 FFN kernel,它们把原来 455μs 的"空隙"填满,**窗口内 GPU 活跃率 91.5%**。"空隙里是 FFN 在跑"从算术推论升级成直接影像。

**但要诚实标注 node trace 的边界(二次审计发现)**:node 级追踪自带每节点开销,会把 kernel 拉长——插桩后 C0 每步"GPU 忙"30.6ms(比未插桩 A 的 24ms 还多),等于**把气泡转成了忙碌**,所以它**只能证明 gap 里跑的是谁(kernel 身份),不能定量气泡多大**(初报的"残余气泡 88μs/层"只有 n=1 样本,系误读,撤回)。**气泡定量以未插桩差分为准**:C0 比 A 每步多 ~10ms,可插桩 CPU 环节 1.6ms,剩 ~8ms = GPU 侧逐层气泡/低效 ≈ **~130μs/层**。物证:`tax-split/nodetrace_{C0node,Dnode}.nsys-rep`。

### 6.4 那真税到底多大?——每层 ~100μs,不是 455μs

既然空隙大头是 FFN,真税就得这么算:

- **原生 A**:每步 24ms ÷ 64 层 = **375μs/层**(含全部 attn+FFN+norm,一张大图无缝连续跑)。
- **C0**:每层 = 可见注意力 22μs + 空隙 455μs = **477μs/层**。
- **每层之差 = 477 − 375 = ~100μs/层**,×64 ≈ **每步 6-7ms**(与 C0 比 A 多的 10ms 同量级,差额是 spec metadata 等 CPU 小项)。

**这 ~100μs/层才是固定税**:同样的 FFN,原生用一张大图无缝跑完;C0 却要每层"回放 FFN 图 → 停下切到手工发注意力 → 再切回下层 FFN 图"。**切换本身的协调开销 + 切换处 GPU 的小气泡,每层多花 ~100μs。** 不是 455μs 空转,是 100μs 的混合边界税。

### 6.5 顺带订正旧版两个说法

- 旧版"C0 GPU 活跃仅 6%、94% 空转":那 6% 是**只在被记录的注意力 kernel 里**的时间;另外 94% **大头是没被记录的 graph FFN**(GPU 在忙),不是空转。"6% 活跃"是 trace 可见性造成的假象,不能读成"94% 闲着"。
- 旧版把气泡归因于"多插了个 memcpy":实测 C0 和 Fea 每步都有 64 个 D2D memcpy,数量一样,memcpy 不是差异来源。真差异是**混合模式的逐层切换**。

## 七、【核心】ngram 也有 gap 却不慢:2×2 对比锁定真凶

到这里只知道"税出在混合边界"。但**凭什么说是'混合'而不是'eager'或别的?** 而且你会问:ngram 投机不也是混合吗,它怎么样?把 {suffix, ngram} × {graph 混合, 全 eager} 四格全跑出来,真凶自现。

### 7.0 先认识对照组 ngram:同为"照抄"投机,机关全拆光

ngram_gpu = prompt-lookup 检索投机的 GPU 张量版:每步拿输出结尾的 n-gram 在本请求上下文里做 GPU 滑窗匹配,找到就抄它后面**固定 k=8** 个 token 当草稿。和 suffix 的关键差别(详表见 [tax-split-progress-report.md](tax-split-progress-report.md) §3.1b):

| | suffix | ngram_gpu |
|---|---|---|
| 草稿来源 | 本地树+**全局树**(跨请求互抄) | 只本请求上下文 |
| 草稿长度 | **变长** 0~k(胆量∝证据) | **定长 k=8**(不足补 -1) |
| 执行位置 | **CPU**、采样后 → **异步被强制关** | **GPU 前向内** → **异步保留**(白名单) |

记住两点后面要用:**①ngram 定长+张量化,所以进了异步调度白名单;suffix 变长+CPU,被强制关异步**——这正是 §7.3 真凶的来源;**②ngram 定长满发、不看证据,草稿质量差**——这在 e2e 并发下会反咬一口(§七c)。

### 7.1 四格吞吐表(char/s,越大越快;加速比对原生 A)

char/s = 输出字符数 ÷ decode 秒数,是不受 tokenizer 影响的可靠吞吐口径。TP4,summary 30 题、report 10 题:

| **Ceager** suffix k24 | 全 eager | 关 | **152 (1.00x)** | **298 (2.29x)** |
| **D** ngram k8 | graph 混合 | **开** | 157 (1.03x) | 279 (2.14x) |
| **Deager** ngram k8 | 全 eager | 开 | 155 (1.02x) | 291 (2.24x) |

| | C0 suffix 混合 | D ngram 混合 | Ceager suffix eager | Deager ngram eager |
|---|--:|--:|--:|--:|

### 7.2 三个直接结论

**结论一:慢的是"混合",不是"eager"。** 看 suffix 两行:**C(graph 混合)126 → Ceager(全 eager)152**,summary 从 0.83x 翻回 1.00x;report 241 → 298,1.85x 冲 2.29x。全 eager 把 FFN 也踢出 graph、**全程手工发,不再有 graph↔eager 边界**,每层 gap 从 455μs 塌到 44μs。**所以固定税就是那个边界;eager 本身不但不慢,反而是目前 suffix 最快的模式。** 这也回答了一个很自然的反问:"既然 eager 更好,suffix 全用 eager 不就行了?"——对,实测确实行(代价是失去大图 graph 的那点好处,但混合模式那点好处远抵不上边界税)。

**结论二:"有 gap"不是根因——ngram 同样混合、同样有 379μs gap,却比原生还快(1.03x)。** 这正是旧版逻辑的致命漏洞:如果"每层有个 gap"就是慢的原因,那 ngram 该一样慢才对。可 D(ngram 混合)157 ≈ Deager(ngram eager)155——**混合模式对 ngram 几乎无伤**。所以 gap 的**存在**不是税,gap 能不能被藏住才是。

**结论三:同 k=8、同混合,suffix 仍慢于 ngram → 排除"只是 k24 拖累"。** 有人会怀疑 C 慢只是因为 k24 草稿长、验证多。但把 suffix 也压到 k8(G0)对比 ngram k8(D):summary G0=141 < D=157。**同样 k=8、同样混合,suffix 还是比 ngram 慢**,证明差异是 suffix 特有的东西,不是草稿长度。

### 7.3 真凶账本(2026-07-05 晚二次修正:ngram 其实也交税,它靠摊薄赚回)

> **修正声明**:本节旧版写"ngram 异步开着、把每层 CPU 边界活藏掉了,同结构却不交税"——**测了 D 的发射速度后发现说重了**。实锤数据:D(ngram 混合)的 cudaLaunchKernel 单次 **24.8μs**(自家全 eager Deager 只要 3.3μs,慢 7.5 倍;C0 是 37.6μs)——**慢发射是混合模式的病,ngram 一样得**;折算每步 CPU 发射+启图 ≈9.1ms,和 C0 几乎一样。每步墙钟也印证:D 31.4ms ≈ C0 34.0ms,ngram 混合**同样付了大头执行税**,只比 suffix 少 ~2.6ms。

那 ngram 凭什么整体还快?修正后的完整账本(TP4-40 summary,batch=1,A=24.0ms/步):

| 配置 | 每步ms | 税的构成 | 怎么活下来 |
|---|--:|---|---|
| D ngram 混合 | 31.4 | 混合执行税 ~7.4 | **摊薄**:每步带 1.34 个接受 token → 23.4ms/token < A 的 24,税被接受赚回 |
| C0 suffix 空草稿 | 34.0 | 执行税 + **suffix 附加费 ~2.6 = 异步关 0.5(直测) + 插桩可见特有环节 ~0.3(propose+RPC) + 未归因 ~1.8(嫌疑:采样结果 GPU→CPU 回传同步,补测排队)** | 无草稿,纯亏 |
| C suffix k24 混合 | 42.5 | 上面全部 + **桶抖动 ~8.5**(变长长草稿步步换 graph 桶) | 接受 1.39/步,摊不回这么多 → 0.83x |
| Ceager suffix k24 全eager | 33.9 | 执行税 ~9.9(发射恢复 3μs 快速,但数量×10;**草稿免费**——无桶可搅) | 摊薄:1.39/步 → 24.4ms/token ≈ A → 平手,report 大赚 |

三个修正后的关键事实(第 1 条经 E 格与用户四次追问逐步修正):
1. **异步调度在 batch=1 只值 ~0.5ms**——直测:D(31.4) vs E(31.9);A vs B 同样 0.5ms。**suffix 附加费 2.6ms(C0−D)的其余部分归因(诚实版)**:插桩可见的 suffix 特有环节仅 ~0.3ms(propose 0.16+RPC 0.15;C0 的树查询实测才 0.03ms);~~"CPU 机件 1.6ms"~~ 之前引用的 1.6ms 是 **C(k24 带草稿)口径**,其中 sampler 0.82/metadata 0.44 是**草稿驱动的投机公共开销**(ngram 也有类似账),不能记 suffix 特有——该说法撤回。**余量 ~1.8ms 未逐项归因,头号嫌疑:每步采样结果的 GPU→CPU 回传+同步**(suffix 的 propose 吃 CPU list 必须做,ngram 全程 GPU 张量不用;从未直测,补测实验 C0+STEP_TIMING 已排队)。每层发射 ngram 照样慢(24.8μs),它没被"藏",是被**接受 token 摊薄**;
2. **suffix k24 混合的最大单项是"草稿×混合"交互项(+8.5ms)**。铁证(交互项模式):混合列随草稿长度涨(C0 34.0→G0 37.3→C 42.5),eager 列不随(CeagerK8 33.7≈Ceager 33.9)。**"桶抖动"是这笔钱的机制假说命名**(形状变化→换图桶/重建元数据/发射重尾),同族未排除候选:形状落到未捕获桶尺寸整步回退、分配器抖动、元数据重建随"变化幅度"走。**"贵在变、不在长"已有跨配置强证据(G0/G/C0 三角)**:同为混合,G0(变长,平均 2.4 位)比 C0 贵 3.3ms=1.4ms/位置,G(定长 8 位)只贵 1.3ms=**0.16ms/位置**——位置多 3 倍反而便宜 2.5 倍,形状恒定时混合下的草稿单价甚至低于 eager 的 0.4。终审方案(候选实验,未排)=C+SUFFIX_TRAJ 逐步回归 `Δt ~ β1·dr + β2·|Δdr| + β3·1[尺寸未捕获]`:β2 显著=搅桶实锤;β1 显著=贵在长度、搅桶证伪;β3 显著=病灶细化为"未捕获尺寸回退"(优化就变成"多录几个桶",比全 eager 还便宜)。**设计原理(大白话)**:三个嫌疑人对"哪步贵"的预言不同——长度派说"草稿多就贵",搅桶派说"和上一步不一样才贵",踩坑派说"落在没录图的尺寸才贵";配置级对比(G0/G/C0)三样一起变分不开,而轨迹把同一次运行拆成几千个自带对照的小实验:"dr=3 且没变"和"dr=3 刚从 0 跳来"的步天然成对,长度相同历史不同,当场分离;β1 另有照妖镜——eager 纯计算单价 0.4ms/位置,混合下 β1 若也 ≈0.4 则长度无混合附加费。eager 列"草稿近似免费"的精确版:草稿位置搭权重读的便车(batch=1 权重读一遍伺候所有位置),边际成本实测 **~0.4ms/位置**(CeagerK8pad 强制 8 槽 +3.5ms 反推);自然草稿平均 2-3 个/步 → 总量 ~1ms 看不见。三种情况破坏免费:混合(桶抖动)、并发(算力共享,dummy 付真钱)、长上下文(KV 读随 ctx 涨);
3. Deager(30.8)≈D(31.4)≈E(31.9)≈DeagerNA(30.8):ngram 的 {混合,eager}×{异步开,关} 2×2 已补全——**异步的 0.5ms 残值只存在于混合列,全 eager 列精确为 0**(30.8 vs 30.8,char/s 155=155)。eager 下发射快而匀、无慢可藏,"强开 suffix 异步"在推荐配置(全 eager)下理论收益为零,彻底销案。

一句大白话(修正版):**离开"纯 graph"谁都要交 ~7-10ms/步的执行税;投机活下来靠的是每步多吐 token 把税摊薄。suffix 混合雪上加霜的是变长草稿搅桶(+8.5,全 eager 一刀消掉)和自家 CPU 机件(+1.6,天生);异步调度在 batch=1 无关紧要(0.5ms)。ngram 命好:定长不搅桶、机件在 GPU 上白嫖,税照交、靠摊薄活得体面。**

> **诚实边界**:异步 0.5ms 是 batch=1 口径——**并发下异步的价值可能上升**(引擎调度与 GPU 重叠在批大时更值钱;7/4 e2e 里 A vs B 墙钟差 ~5%),e2e5 同天数据出齐后复核。桶抖动 8.5ms 的归因有 Ceager≈C0 直接对照,硬。"suffix 强开异步"候选实验的价值因 D/E 直测而大降,基本可砍。

**六修补充(7/5 深夜,逐条 vs ctx 分析)**:把 30 条 summary 逐条摆开看"每步差 vs prompt 长度",发现 **C0−A 是横平的常数 +10.0ms**(8k~51k 字符纹丝不动——"固定"二字首次被逐条坐实,它与一切计算量无关),而 **Ceager−A 是斜线**(短 ctx +6~8、长 ctx +11~14 = 基础费 ~7 + ctx×草稿按量费)。**出租车比方**:C0 是纯起步价(手续费——spec 采样路径/回传同步/引擎往返,每步同样几道章,不管文件多厚),Ceager 是起步价+里程费(每个草稿位置的验证要读一遍该请求的 KV,上下文越长押注越贵)。**两种病两种药**:起步价治法=改代码砍章(系统);里程费治法=策略躲(ctx 感知自适应投机——与"头几步 miss 关投机"同一开关,再加"上下文过长收手"条件)。这也把"长 prompt summary 亏损"的机制闭环:34-51k 的 summary 踩在里程费最贵区段、接受率又低,付了里程费还不中。**两者中位近等(34.0 vs 33.9)是巧合**,内部构成不同。据此 C0eager 预测修正为三分支:≈34→固定税全属 spec 管线;**≈30-31→拆成"执行模式无关 ~6-7 + 混合特有 ~3-4"**(边界拿回部分罪名);≈25-26→固定税基本是混合的锅。另:nsys 探针负载(高接受 report 型)下 Ceager≈Fea 每步打平(GPU 忙/闲逐类账本相减≈0)——spec 的 GPU 侧每步成本在该负载下≈0,进一步支持"残量在 CPU 侧管线或 ctx×草稿按量"。

**七修·残量定案(7/5 深夜,零新实验——挖 7/5 午间 cpusample 采样文件)**:未归因残量的身份链闭合——C0 每步 7.0ms 的"发射"动作里,**4.5ms 是发射线程停在 `cuLaunchKernel` 内部、CUDA 驱动私有条件变量上的等待**(OSRT 直测:关键线程的 pthread_cond_timedwait 74% 落在 launch 调用内部,调用栈 `cond_timedwait ← libcuda 内部帧 ← cuLaunchKernel` 实锤);**全 eager 对照(cpusample_Ceager)同项精确为 0.00s**。机制:graph 段提交与 eager 发射交错时驱动内部强制串行化,发射线程被挂起;"慢发射 37.6μs 重尾"的本质是**锁等待,不是发射工作**。更深一层"谁持锁"在 libcuda 闭源符号内,不可见也不必见——可操作结论完备:**别让 graph 与 eager 交错提交**(全 eager 或全图),与 Ceager 实测全面对齐。C0 固定税 10ms 由此预拆:锁等待 4.5 + 图段启动 1.4 + spec 管线(执行模式无关部分,C0eager 今晚给裸值) + 零头;C0eager 预测进一步收窄到 **~28-30**。

**终审格 C0eager(空草稿+全 eager,夜间链在跑)——为什么它能终审**:先澄清一个前提:**空草稿也走混合路径**——vLLM 的执行骨架是启动时按配置定死的(spec 配置在,注意力就被排除在 graph 外,FFN 录成 64 段小图),草稿=0 只是让 eager 注意力输入恰好 1 个 token,骨架不变;物证在 C0 自己的 trace 里(64 次/步 cudaGraphLaunch + eager 注意力 kernel + 455μs 结构 + 37.6μs 慢发射,混合全套特征齐)。C0eager 则把"边界"**物理拆除**(零图、零图段启动、零边界、发射恢复 3μs),但 spec 机件与 C0 完全相同。于是判决二选一:
- **C0eager ≈ 33-34(≈C0)**:边界拆光税没少 → 税出在两配置共有的 spec 步管线(回传同步/采样路径/memcpy/引擎往返),"混合边界=固定税"撤案,边界现象降级为"税在混合形态下的花法";
- **C0eager ≈ 26-30(掉下来)**:边界确实值钱,且 Ceager 的 +9.2 里草稿验证占大头("草稿在 eager 下免费"要修),边界叙事部分复活。

## 七b、税的完整账本:固定税之外还有"按量税",k 怎么选

破的案是**固定税**(每步都交、与草稿无关)。但 suffix 每步实际多花的钱还有第二个分量——**按量税**:每多一个草稿位置,前向就要多算一个位置的验证+多读一份 KV,草稿越长交得越多。两个分量分开治,才能解释下面这张递进表(TP4-40 summary,每步 ms,原生 A=24.0):

| 配置 | 每步ms | 总税 | 相邻省 | 归因 |
|---|--:|--:|--:|---|
| C suffix k24 变长 | 42.5 | +18.5 | — | |
| G0 suffix k8 变长 | 37.3 | +13.3 | **5.2ms** | **截短=按量税**(草稿短→验证位置少) |
| G suffix k8+pad 定长 | 35.3 | +11.3 | 2.0ms | padding 规整形状,小效应 |
| D ngram k8 | 31.4 | +7.4 | 3.9ms | (修正)非免税:同样交混合执行税,省的是 suffix 自家 CPU 机件 ~1.6ms,余下靠接受摊薄(§7.3) |

**为什么 k=24 不划算——接受早就饱和了**:全程每步接受长度(accepted+1)实测 C(k24)=2.33、G0(k8)=2.21、G(k8+pad)=2.12。k 从 8 抬到 24,每步只多接受 **+0.12 个 token(+5%)**,每步时间却贵 14%。这个 workload 的匹配段本来就短,k=8 已经装下几乎全部能接受的草稿,k=24 多出的上限全是白验证的按量税。

### 七b.1 k 和 padding:两个容易混淆的旋钮

表里 G0(k8) 和 G(k8+pad) 只差一个 padding,先把这两个旋钮掰开——**它们钉的是两个不同的东西**:

- **k(`num_speculative_tokens`)= 天花板**。suffix 每步按"匹配到多少证据"决定草稿长度:匹配 3 个就发 3 个、没匹配就发 0,k 只是不许超过的上限。所以 **k=8 下草稿长度仍然步步在变(0~8 之间)**。实测:G0 全程 draft=51,444、约 2.1 万步 → **平均每步草稿才 ~2.4 个,离天花板 8 远得很**。
- **padding(`SUFFIX_PAD_DRAFTS=8`,我们的插桩)= 定长器**。每步草稿不足 8 就用 dummy token(id=0)补到**恰好 8 个**。dummy 在验证时必被拒(和真 token 一起过 rejection sampler,错就砍),**无损、纯占位**。

两个铁证(TP4-40 server 日志累计):

| | G0(k8 变长) | G(k8+pad 定长) |
|---|--:|--:|
| draft 总数 | 51,444 | **168,504 = 21,063 步 × 8,恰好整除** |
| accepted 总数 | 25,364 | 25,087(几乎不变) |

draft 翻 3.3 倍、accepted 纹丝不动——**padding 发的全是必死的 dummy,只买"形状"不买"接受"**;整除关系直接证明每步严格 8 个。

**为什么 padding 才能保证规整、k 不行**:每步验证张量的形状 = 这一步的草稿位置数。k 只钉住了**天花板**,**地板还在动**(这步 2 个、下步 0 个、再下步 5 个)→ 形状步步变。padding 把地板顶到天花板(**恒等于 8**)→ 形状每步一样,这才叫规整。ngram 天生就是这么干的(定长 k=8、不足补占位)——**这正是它"张量化、规整"、能进异步调度白名单的结构原因;suffix 的变长草稿则是它被强制关异步的原因**(§7.0 的两点在这里汇合)。

**为什么"规整"对 CUDA graph 值钱——graph 是台复读机**:回到 §三,graph 的本质是**录一遍、以后原样照放**——录制时每个 kernel 的线程数、张量大小、显存地址**全部写死在图里**,回放时不看今天的输入,原封不动重发。所以形状必须和录制时一致:按"4 行输入"(1真+3草稿)录的图,喂 7 行进去,图里的 kernel 只算前 4 行,后 3 行没人碰——**形状对不上,图就不能用,没有商量余地**。vLLM 的现实妥协是**多录几张图(桶)**:给一批常见 token 数各录一张,运行时选最近的桶、把输入 pad 到桶大小。于是变长草稿的代价就清楚了——
- k=8 变长:这步 3 草稿选"4-token 桶",下步 0 草稿换"1-token 桶",再下步 6 草稿又换桶——**每步换桶**,选桶逻辑+配套元数据(验证位置索引/logits 索引/rejection sampler 张量)步步重建、尺寸步步不同;
- pad=8 定长:每步恒 9 token(1真+8草稿)→ **永远同一个桶**,同一张图、同尺寸元数据、同一块显存布局,步步复用。

**那 pad 帮的到底是谁?为什么只值 ~2ms/步**:spec 注意力**在配置层面就走 eager**,pad 不 pad 它都不回 graph(要改代码才行)。所以 pad 规整的是**配角**——还留在 graph 里的 FFN/norm(桶稳定不换)+ 验证/采样链路(张量形状稳定,不用步步重算尺寸重排索引);而**主角**(eager 注意力 + 混合边界税)根本不理它。配角的边际小利,实测就是那 ~2ms。

**"padding 是否只对 graph 有效"的判决实验(2×2,已完成 7/5)——预测精确命中,机制闭环**:

| pad 效应(k=8,TP4-40 replay,SUMMARY 每步ms) | graph 混合在场 | 全 eager |
|---|---|---|
| 不 pad | G0 = 37.3 | CeagerK8 = **33.7** |
| + pad | G = 35.3 | CeagerK8pad = 37.2 |
| **pad 的净效应** | **−2.0ms(赚)** | **+3.5ms(倒亏!)** |

(REPORT 列同型:混合下 pad 赚 2.0ms,eager 下倒亏 2.5ms。)

**判决**:pad 在混合下赚、在全 eager 下**倒亏**——dummy 槽的按量成本两边都在付,而"桶稳定"收益只有 graph 在场才存在。**"padding 的收益经由 graph 桶稳定"实锤。** 推论:全 eager 部署下 pad 是纯负资产,别加。

顺带闭环另一桩悬案:**eager 下 k 也无所谓了**——CeagerK8(k8)summary 151 ≈ Ceager(k24)152 char/s,report 293 ≈ 298。之前"k=8 甜点"的整套讲究(截短省按量、pad 补形状)全是**混合模式的伤药**;换到全 eager,草稿近乎免费,k 用默认即可。

**k×pad 全扫点(TP4-40,char/s 加速比,summary/report)**:

| | k8+pad8 | k16+pad16 | k24+pad24 | k24 变长 |
|---|--:|--:|--:|--:|
| SUMMARY | **0.97x** | 0.89x | 0.85x | 0.83x |
| REPORT | **2.11x** | 2.00x | 1.91x | 1.85x |

单调递减:**k 越大越差,pad 跟着放大也救不回**(padding 本身只值 ~2ms/步——它填的 dummy token 永远被拒,只买到"形状规整",买不到接受)。结论:**按量税的甜点 = k=8+pad;固定税(混合边界)另治(§九的两条路)。两刀都砍才追得平/超过 ngram。**

顺带钉一个数据勘误:早期 TP4-10 条曾测出"k8 比 k24 还差"(G0=56 < C=64 char/s),n=40 后方向稳定反转(G0=141 > C=126)——**10 条口径的单点噪声,以 40 条为准**。

## 七c、真实 e2e 下的第一次校验:破案结论的适用边界

以上全部是 **replay(batch=1、冷树)**口径。真实 agent(e2e,local 缓存、10 题、题内并发 3)7/4 跑了五格,两处结论**反转**、三处保持——反转处恰好暴露了 batch=1 口径的边界(完整表见 [tax-split-progress-report.md](tax-split-progress-report.md) §3.7):

| 配置 | SUM tok/s | REP tok/s | 每题墙钟中位 | e2e 接受率 |
|---|--:|--:|--:|--:|
| A 原生 | 25.3 | 40.6 | 352s | — |
| C suffix k24 | **25.9(≈平手)** | 81.7 (2.01x) | **285s(最快)** | **39.9%** |
| D ngram k8 | **19.0 (0.75x!)** | 90.0 (2.22x) | 318s | **4.9%** |

- **suffix 的 summary:replay 0.83x → e2e 平手。** e2e 的全局树是热的、还有兄弟互抄,接受率抬到 39.9%(replay 冷树约一半),接受收益盖过固定税。**replay 冷树口径系统性低估 suffix。**
- **ngram 的 summary:replay 1.03x → e2e 净亏 0.75x。** §七说"ngram 不交固定税",没错——但那是 batch=1 的固定税口径。e2e 里 ngram 发了 95 万 draft(C 的 4.3 倍,定长满发)只接受 4.9%:batch=1 时垃圾草稿的验证搭权重读的便车≈免费,**并发=3 时算力共享,垃圾草稿要付真钱——ngram 的"定长满发"从无害变成负资产(按量税爆炸)。**
- **保持的三条**:report 两家都大赢(方向同 replay);异步调度差异仍只有几个点;固定税结构不变。
- **每题墙钟 C 最快**(285s,比 A 快 19%)——report 段的 2x 直接压缩单题关键路径。

**悬念已落地(7/5 同天六格重跑,终版表见 [tax-split-progress-report.md](tax-split-progress-report.md) §3.7b)**:**Ceager 在 e2e 并发下不但保住、还是全场冠军**——墙钟 278s(比 A 快 14%)、SUM 26.4(唯一不亏)、REP 95.1(2.35x);唯一代价 TTFT +14%(纯 enforce-eager 的 prefill 账单,C 与 A 的 TTFT 相同)。两刀合一(CeagerK8pad)被否:pad 洪水把接受率打到 13%,两段均输。**§九方向一可直接上生产:`suffix + --enforce-eager`,k 默认、不 pad。**

## 八、之前对不上的现象,现在全闭合

1. **F(不投机关graph)只慢 0.7ms,C0(投机)却慢 10ms?** —— F 关的是**整个模型**的大图,退成的 eager 都是 memory-bound 算子、发射被 GPU 等显存掩盖;C0 是**混合**——每层在图与手工之间切换、且异步被关,切换处的 CPU 活露出来。不是"有没有 graph",是"混不混、藏不藏得住"。
2. **Fea 发 350 万 kernel 不慢,C0 发 45 万反而慢?** —— 数量不重要,重要的是发射/CPU 活能否被 GPU 的显存等待掩盖。Fea 全 eager 匀速发、能掩盖;C0 混合+异步关,每层切换处掩盖不住。
3. **为什么税和"猜没猜草稿"无关(空草稿也交)?** —— 混合边界是"注意力走 eager"这个执行结构带来的,每层每步都发生,和草稿数量无关。C0 草稿=0 照样逐层切换。
4. **ngram 明明也混合,凭什么不慢?** —— (二次修正)它其实**也交**混合执行税:发射同样慢(24.8μs)、每步 31.4ms≈C0 的 34ms。不慢靠两条:**每步带 ~1.34 个接受 token 把税摊薄**赚回;没有 suffix 那 ~1.6ms 的自家 CPU 机件。异步调度经 D/E 直测只值 0.5ms/步,不是主因(§7.3 修正账本)。

## 九、这案破了有什么用

优化标的从旧版那句(错的)"消除 455μs GPU 空转",订正成两个**有 kernel 级物证**的具体方向:

1. **让 suffix 的注意力也能进 graph / 或走全 eager 去掉混合边界。** 实测 Ceager(全 eager)已把 suffix 从 summary 0.83x 救到 1.00x、report 1.85x 救到 2.29x——**这是当前投入产出比最高的一步,基本无需改核心逻辑,只是 `--enforce-eager`**。
2. **让 suffix 能开异步调度**(hack assert + k8pad 定槽 + worker 侧同步喂内容)。(三次修正后基本判死)异步在 batch=1 经 D vs E **直测只值 ~0.5ms/步**,不值得为它做系统改造;唯一保留的问号是**并发下异步价值可能上升**(7/4 e2e 里 A vs B 墙钟差 ~5%),等 e2e5 同天数据复核——若并发下也小,这条路销案。

外加两条配套(§七b/§七c):**按量税用 k=8+pad 砍**(k 大是错的,pad 放大救不回);**方向一(全 eager)上生产前必须过 e2e 并发这关**——batch 变大后 graph 价值上升,Ceager 的 e2e 格已排队。

这是能写进论文的系统机制发现——**不是"我们觉得慢在 GPU 空转"(旧版,错),而是"慢在 graph+eager 混合的逐层边界 + 异步调度被关,ngram 同构不同命证明了这点"**。

## 追问区(预判的坑)

- **旧版说 29ms 空转,现在说没空转,到底哪个对?** 新的对。旧版把"graph FFN 在跑但 nsys 没记录"错当成空转了。真相:那 455μs 里大头是 FFN 计算,真税是每层多出的 ~100μs 混合边界开销(§六 6.3-6.4)。
- **更早的"suffix 固定税 ≈18%"还成立吗?** **数字复现,结论已死。** 同配置同口径(k24+混合+冷树+batch=1)下今天 TP4-40 复现 0.83x ✓;但完整会计是:**支出** +18.5ms/步 = 变化费 8.5(草稿×混合交互,确定)+ 固定税 10.0(内部构成待 C0eager 终审:切换固定费 x + spec 步管线 10−x,现有证据倾向 x≈0),**收入** = 每步 0.43 接受 token 的摊薄抵扣,**净额** = +24%/token ≈ 0.83x。它②只属于那一个配置组合——Ceager 同口径净税 0%(1.00x);③换 e2e 热树口径连旧配置自己都回 1.02x(支出没变,收入变多)。正确改写:"suffix 在最坏配置组合下净损 ~18%,其中确定可消的变化费占支出近半,一个启动参数即可消除"。
- **"每步耗时"严不严谨?每步吐 token 数不一样。** 比想象中更严谨——7/5 晚审计发现一个意外事实:**带草稿的投机运行里,vLLM 返回的 completion_tokens(即 res 里的 n_out)数的是"decode 步数"而不是 token 数**。铁证:同题 temp0 下 Ceager 与 A 输出逐字相同(i5,566 字符),A 报 n_out=307(=真 token 数),Ceager 只报 145——145 步 × ~2.1 token/步 ≈ 307。而 C0(空草稿,1 token/步)n_out 与 A 同量级 ✓。**所以全文的 `decode_s/(n_out−1)` 对所有配置都是"每步 ms",各行单位一致,账本成立**;且聚合闭合:Ceager 33.9ms/步 ÷ 1.43 token/步 = 23.7ms/token ≈ A 的 24.0 → char/s 平手 152=152。吞吐比较仍以 **char/s**(长度归一)为主。
- **455μs 里到底多少是 FFN、多少是真气泡?** 两条途径各答一半:**身份**由 node trace 定(§6.3,gap 里是 FFN,活跃 91.5%);**大小**由未插桩差分算术定(node trace 自带开销会把气泡转成忙碌,不能用它定量)——C0 比 A 每步多 ~10ms,减去可插桩 CPU 的 1.6ms,**真气泡/低效 ≈8ms/步 ≈130μs/层**,即 455μs ≈ FFN ~325μs + 气泡 ~130μs 的量级。更硬的定量证据在途:nsys batch=3 对比(若气泡是"CPU 慢半拍",b3 下 reshape 前 gap 应缩到只剩 FFN 那截)。
- **"切换"到底是 CPU 耗时还是 GPU 耗时?** **切换 100% 是 CPU 的活;GPU 没有"切换"这个动作。** GPU 只是个带任务队列的执行器,不知道 graph/eager 是什么——那是 CPU 的两种递任务方式。**CPU 每步账单实测**(C0,CUDA API 全量统计,每 rank):发射 205 个 eager kernel 花 **7.7ms**(单次 37.6μs)+ 启动 64 段 FFN 图花 **1.4ms**(21μs/段)+ memcpyAsync 0.5ms + 查流状态 0.05ms ≈ **9.1ms/步**;而 `cudaEventSynchronize` 每步仅 1 次(步末等采样)——**逐层气泡里 CPU 不是在等,是在忙着发**。
  **实锤在对照:同一个 cudaLaunchKernel,五种模式速度对照**——A(纯图)5.5μs、Fea(全 eager)3.2μs、Deager(ngram 全 eager)3.3μs、**D(ngram 混合)24.8μs、C0(suffix 混合)37.6μs**(中位都正常 3-4μs+重尾:多数发射正常,被卡的都在图/eager 边界,驱动要做流状态校验)。**慢发射是混合模式的病,suffix 和 ngram 都得**;全 eager 两家都恢复 3μs。Fea 每步发 758 个 kernel 才花 2.4ms,C0 发 205 个却花 7.7ms。**账目闭合:C0 发射+启图 9.1ms − A 的 0.5ms = 8.6ms ≈ 差分气泡 ~8ms/步**——GPU 侧差分与 CPU 侧 API 计时两条独立途径互证。**CPU 干活,GPU 站岗买单**,这就是"税的本体是 CPU 慢半拍的气泡"。
- **为什么并发(batch 变大)会把税"藏住"?(nsys b3 直测后改写——原"气泡被 GPU 忙碌挤掉"的猜想被证伪)** 先交代原猜想是什么意思,证伪才有着落:每层两条并行时间线里,GPU 有活干 ~390μs < CPU 杂务节拍 ~460μs,差额就是每层的饿肚子气泡;猜想认为 batch=3 时 GPU 每层的活变多(KV 读 3 份),忙的窗口会**长到盖过** CPU 节拍 → 气泡被从时间线上"挤"出去,预言 gap 应缩到只剩 FFN 的 ~380-400μs。**实测:GPU 确实变忙了(flash_fwd 翻倍),但离 460 的天花板还远,节拍纹丝不动,gap 455→447 没缩——气泡一个没少。** 答案比原猜想简单:**每步固定成本是"每步常数",不随 batch 涨;并发的拯救 = 纯摊薄。** nsys 直测 C0 在 1 路 vs 3 路并发:**步时 33.2→33.5ms 几乎不动**,却同时伺候 3 路 → 每 token 税 ÷3。微观三指标:②flash_fwd 10.8→22μs(GPU 每层活翻倍,被塞进原有时隙)、③cudaGraphLaunch 恒 64/步且单价 21μs 不变(CPU 杂务不随 batch 变)——这俩解释了步时为何能保持常数;**但 ①reshape 前 gap 455→447μs 没缩**,直接否了"GPU 变忙把每层气泡吃掉":每层微结构原样不动、气泡没消失,只是被 3 倍产出摊平。batch 扫描的效应曲线(C−Ceager 每 token 差 8.1→5.5→4.5ms)与此吻合。这就是 e2e 里 Ceager 对 C 的 summary 提升(+2%)远小于 replay(+21%)的主因(另一个:e2e 的 per-call tok/s 含共跑噪声)。(注:异步调度经 D/E 直测只值 ~0.5ms/步——ngram 在 batch=1 活下来靠的也是接受摊薄,见 §7.3 修正账本。**摊薄是贯穿全案的同一个机制**。)
- **Ceager 做了投机,为什么 replay 里只和 A 打平?** 投机不是免费加速器,是"用步时买 token"的生意。收支表:Ceager 每步贵 41%(24.0→33.9ms),每步多吐 43% token(1→1.43)——精确抵消。**盈亏平衡点 = 每步接受 33.9/24 = 1.41 个 token,冷树 summary 实测 1.43 恰好压线**。report 接受 3+ → 收入三倍支出不变 → 2.3x;e2e 热树抬接受+并发摊支出 → summary 微赚。支出侧逐项(七修后已基本闭合,±1ms):发射件数×10 +1.7(采样直测 2.2 vs A 0.5)+ launch 外 cond 等待 +1.2(采样直测,GIL/线程交接类,栈归因待 step1)+ sampler 0.82 + metadata 0.44 + propose/RPC 0.35 + 异步关 0.5 + 验证位置 ~0.5-1 ≈ **+5.5-6.5 ≈ 实测短 ctx 基础费 +6-8 ✓**;长 ctx 另加"里程费"斜线(GPU 侧注意力按量)。**Ceager 没有大病,是七笔小账;C0(混合)才有一笔 4.5ms 的大病(cuLaunchKernel 内驱动锁等待,七修定罪)**——这正是全 eager 方向正确的会计学表述:用一笔可消的大病换几笔难再优化的小账。**新旧账本对账**:早前"未归因 ~5.4"= 发射低估 1.0(旧用墙钟口径 F−A 0.7,CPU 直测实为 1.7)+ launch 外 cond 等待 1.2(新直测)+ ctx 斜线 ~3.2(旧表未分离 ctx 维度)——严丝合缝。口径提醒:CPU 时间项 ≈1:1 上墙依赖"异步关+GPU 有空隙"的条件,非恒等式,是 ±1ms 模糊的来源。若把小账里 spec 管线部分再砍掉,冷树 summary 即变 1.43x——"降固定税"方向的剩余奖品。
- **为什么不直接上 nsys,还先做那么多差分?** 差分便宜(跑个 replay)、能快速排除大块嫌疑;nsys 数据量大、要独占卡、还容易被"graph 不记录"误导(旧版栽的坑)。而且差分里的 Fea 和 §七的 ngram/eager 对照,恰恰是纠正 nsys 初判的关键——两种方法互相纠错。
- **nsys 会把别人进程算进来吗?** 不会,只给自己启动的进程注入探针。但别人同时占卡会拖慢我们的 kernel,所以测时独占了卡。

## 术语表

| 术语 | 一句话 |
|---|---|
| kernel | GPU 上跑的一个计算函数(一次矩阵乘/softmax/搬数据) |
| kernel launch(发射) | CPU 执行 `cudaLaunchKernel` 派活给 GPU;异步,发完不等 GPU |
| CUDA graph | 把一步内几百个 kernel 的发射录成一段"图",一次回放,GPU 连续跑不等 CPU |
| eager 模式 | 不用 graph,每个 kernel 都 CPU 现场发一次 |
| graph+eager 混合(piecewise) | 开投机后:FFN/norm 留 graph、注意力挖成 eager,每层在两者间切一次;本案真凶藏身处 |
| 异步调度(async scheduling) | CPU 在 GPU 跑当前步时提前为下一步备料;ngram 开、suffix 强制关。D/E 直测:batch=1 下只值 ~0.5ms/步,非税的主因 |
| GPU 空转气泡(bubble) | GPU 已准备好,却在干等 CPU 发下一个 kernel 的闲置 |
| memory-bound | decode 的瓶颈:每步读一遍全部权重,GPU 算力闲、主要等显存 |
| 每步固定税 | 开投机后每步凭空多花、与猜没猜无关的时间(C0 空草稿也交);实为混合边界税 ~10ms/步 |
| 按量税 | 与固定税相对:每多一个草稿位置多付的验证+KV读成本,草稿越长越贵;k=8 砍的是它(§七b) |
| k(num_speculative_tokens) | 草稿长度**天花板**:每步实发 0~k 个、按证据变;k=8 下平均每步才 ~2.4 个,长度仍步步变(§七b.1) |
| padding(SUFFIX_PAD_DRAFTS) | **定长器**:不足 k 就补必被拒的 dummy token 到恰好 k 个,把形状钉死;只买规整不买接受,实测 ~2ms(§七b.1) |
| 桶(cudagraph bucket) | vLLM 给一批常见 token 数各录一张图,运行时选最近的桶;变长草稿=每步换桶,定长=永远同桶(§七b.1) |
| ctx(上下文长度) | 该请求当前 KV cache 里的 token 总数=prompt+已生成;决定每步注意力读多少 KV,是"里程费"的里程(文中常用 prompt 字符数作代理) |
| 差分实验 | 造两个只差一件事的配置,耗时之差=那件事的成本 |
| 插桩(instrumentation) | 在代码关键处加计时器,量某段 CPU 代码耗时 |
| nsys | NVIDIA 系统级剖析器,录 GPU kernel 时间线/发射/空转;**默认不记录 graph 内部 kernel** |
| cuda_gpu_trace | nsys 导出的逐 kernel 明细(每 kernel 起止),用来找空隙在哪 |
| --cuda-graph-trace=node | nsys 选项,开了才逐个记录 graph 内部节点(否则整张图记成一条,FFN 隐身) |
| reshape_and_cache | 把算出的 K/V 写进 KV cache 的 kernel,注意力最后一步;每层空隙就在它前面 |
| C0 配置 | suffix 投机但草稿上限=0、永远空草稿——隔离"纯机制开销"的对照组 |
| C / Ceager | suffix k24,graph 混合 / 全 eager;两者对比证明"慢的是混合不是 eager" |
| D / Deager | ngram k8,graph 混合 / 全 eager;D≈Deager≈E 打平,证明混合税对 ngram 被"定长不搅桶+接受摊薄"抵消(旧说"靠异步不交税"已修正,§7.3) |
