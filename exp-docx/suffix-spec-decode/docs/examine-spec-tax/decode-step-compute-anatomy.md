# Decode 每步的计算量拆解(Qwen3-32B 实数)

> **缘起**:讨论"能不能砍 KV 减注意力开销"时,发现必须先讲清 decode 一步到底在算什么、哪部分大、哪部分随上下文涨。本文把一个 decode step 拆到零件级,全部用 Qwen3-32B 真实结构参数算,并分清 **FLOP 口径**和**访存口径**(decode 是访存瓶颈,两者不一样)。
> **一句话**:decode 每步 = **把整个模型权重读一遍生成 1 个 token**;时间几乎全花在**读 64GB 权重**上(和上下文无关),**注意力只是个随上下文长度 L 线性增长的小尾巴**——L=7K(summary)时占延迟 ~2.8%,L=20K(report)时 ~7.5%,要到 L≈几十万才反超权重。

## 零、先立三个概念(30 秒)

1. **一个 decode step = 生成 1 个新 token**。要拿到这 1 个 token,GPU 得让当前上下文过一遍**整个模型**(64 层 + 输出头)。
2. **两类活,性质完全不同**:
   - **权重矩阵乘**(QKVO 投影 + FFN + 输出头):和上下文长短**无关**,每步都是固定的一大坨;
   - **注意力**(新 token 对已缓存的 K/V 做加权):**和 KV 长度 L 成正比**,上下文越长越大。
3. **decode 是"访存瓶颈"不是"算力瓶颈"**:批量=1 时,GPU 的算力核心大部分时间在**等着从显存把权重读进来**,不是在等算。所以决定**延迟**的是**读了多少字节**,不是做了多少 FLOP——这条贯穿全文。

## 一、一步里到底有哪些部分(逐零件)

按数据流顺序,每层(共 **64 层**)依次做:

| 部分 | 干什么 | 矩阵形状(隐藏维 d=5120) | 和 L 有关? |
|---|---|---|---|
| RMSNorm ×2 | 归一化 | 逐元素,极小 | 否 |
| **Q 投影** | 算 query | 5120 → 64头×128 = 8192 | 否 |
| **K/V 投影** | 算 key/value(GQA 只 8 头) | 5120 → 8头×128 = 1024(各一个) | 否 |
| RoPE | 给 Q/K 加位置 | 逐元素,极小 | 否 |
| **注意力** | Q·Kᵀ → softmax → ·V | 对 **L** 个缓存 key 各算一遍 | **是,∝L** |
| **O 投影** | 注意力输出投回 hidden | 8192 → 5120 | 否 |
| **FFN(SwiGLU)** | 两升一降 + 门控 | gate/up:5120→25600;down:25600→5120 | 否 |
| 残差 add | 加回主干 | 逐元素,极小 | 否 |

64 层之后,做**一次**:

| 部分 | 干什么 | 形状 | 和 L 有关? |
|---|---|---|---|
| 末层 RMSNorm | 归一化 | 极小 | 否 |
| **LM head** | 投到词表算 logits | 5120 → 151936(词表) | 否 |
| 采样 | argmax / softmax 选 token | 词表上一次,较小 | 否 |

**所以"哪些部分"= QKVO 投影 + FFN + LM head(权重乘,固定)+ 注意力(∝L)+ 一堆逐元素小活(norm/RoPE/残差/采样,可忽略)。**

## 二、每部分多大(FLOP 口径,每步每 token)

用真实参数(d=5120,64 层,64 Q 头 / 8 KV 头,head_dim=128,FFN 中间维 25600,词表 151936):

| 部分 | FLOP/token | 占权重总量 |
|---|--:|--:|
| **FFN** | **50.3 G** | **78.6%** ← 最大头 |
| 注意力投影(QKVO) | 12.1 G | 18.9% |
| LM head | 1.56 G | 2.4% |
| **权重小计(与 L 无关)** | **≈ 64 G** | 100% |
| 注意力(∝L) | 32768×L×64 ≈ **2.1 G × (L/1000)** | 见下 |

- **权重乘固定 ~64 GFLOP/token**,正好对应"32B 参数 × 2(一次乘一次加)";其中 **FFN 独占近 80%**(升维到 25600 太贵)。
- **注意力随 L 线性涨**:L=2000 时才 4.2G,占总 FLOP 6.2%;要 L≈**3 万**才追平权重。

## 三、但延迟看的是"访存",不是 FLOP(关键)

decode 批量=1,算力远没喂饱,**每步时间 ≈ 从显存读的字节 / 显存带宽**。读的字节:

| 读什么 | 字节/token | 和 L 有关? |
|---|--:|---|
| **全部权重**(32B×2B,bf16) | **64 GB** | 否,每步都读一遍 |
| **KV 缓存**(K+V,8 头×128×64 层×2B) | **0.26 MB × L** | 是,∝L |

- L=2000:KV 读 0.52 GB,占总访存 **0.8%**;
- **权重那 64GB 才是每步延迟的绝对大头**——这也是[五常数成本模型](fixed-tax-conclusions.md)里 **24ms/步基线**的来源(读权重为主,叠加 kernel 启动/不重叠等开销)。

> **为什么两个口径差这么多**:注意力的 FLOP 占 6.2%,但它的**访存**只占 0.8%——因为注意力"读一堆 KV、只做很少算"(算术强度低),而 decode 是访存瓶颈,所以**看延迟要用访存那列**。凡本文说"注意力占比",延迟相关的都指访存口径。

## 四、注意力占比随上下文 L 变化(延迟口径)

| 上下文 L | 注意力 FLOP 占比 | **访存(延迟)占比** |
|--:|--:|--:|
| 2,000 | 6.2% | **0.8%** |
| 8,000 | 20.8% | **3.2%** |
| 32,000 | 51.2% | **11.6%** |
| 100,000 | 76.6% | **29.1%** |

**交叉点(注意力=权重):FLOP 口径 L≈3 万,访存(延迟)口径 L≈24 万。**

## 六、这对"优化 decode"意味着什么

把两类活分开,就能看清每种优化动的是哪块:

| 优化 | 动哪块 | 效果 | 适用 |
|---|---|---|---|
| **投机解码**(suffix/EAGLE) | **摊薄权重读** | 一次前向验 K 个草稿 token → 那 **64GB 权重读被 K+1 个 token 分摊** → **减步数** | 一直有效(权重读是大头) |
| **更深草稿** | 每步多验几个位置 | 草稿位置是**加算力**(FLOP),但**权重已在读**(访存不变)→ 访存瓶颈下每位置很便宜(实测 0.62ms/位置) | 有长可抄段时 |
| **KV pruning / 稀疏注意力** | **砍注意力那条 ∝L 的尾巴** | 省注意力那块 | **长上下文 或 高并发**(见 §四 + 下方并发修正) |
| 量化(权重 INT8/4) | **缩权重读的字节** | 直接砍最大头(64GB→更小) | 一直有效,但另一条线 |

**两个要点**:
1. **投机解码之所以是主杠杆**——它动的是"读权重"这个每步的绝对大头(把它摊到多个 token 上),不是动注意力那个小尾巴。这也是为什么[Ceager 对每个 tag 都不亏](../l0-l3-v2-results.md):省的是大头。
2. **KV pruning 减注意力**在**长上下文或高并发**才有肉——§五 的 2.8%/7.5% 是 **batch=1** 的数,**批处理会把它放大**(见下方并发修正):真要动得用**注意力类信号**(H2O:未来会不会被 attend)去挑剪谁,report 的 KV 恰是它要综合/引用的 summary(高重要性、不该乱剪),用"投机是否命中"当剪枝信号是错配的(表面可抄 ≠ 注意力重要,已判死见 [../explore-idea/idea-verdicts-and-standing.md](../explore-idea/idea-verdicts-and-standing.md) §一.9)。

### 并发修正:批处理把 KV 占比放大(2026-07-06 追加)

§五 的注意力占比是 **batch=1**。但生产是并发的,而 §六 表已指出:**权重读跨请求共享、KV 各读各的**。所以 batch=B 时每步每卡访存 = 权重 16GB(不变)+ B×(每请求 KV),**KV 占比随 B 涨**:

| 场景 | B=1 | B=3 | B=6 | B=9 |
|---|--:|--:|--:|--:|
| summary(L≈7K) | 2.7% | 7.8% | 14.5% | **20.3%** |
| report(L≈20K,实际单流) | 7.5% | 19.6% | 32.8% | 42.3% |

- **summary 实际 ~9 路并行** → KV 占每步 **~20%**(权重 16GB + KV 9×0.46=4.1GB)→ 砍 50% KV 可省 **~10%** summary decode(现实 ~5-8%,随并发浮动);这是**目前找到的最实在的 KV pruning 靶子**;
- **report 是单流串行**(每题末尾一个,无并发伙伴)→ 停在 batch=1 的 7.5% → 只省 3.8%。**report 上下文虽长,但没并发把它的 KV 撑成大头。**

**结论**:"砍 KV 减注意力"在 batch=1 可忽略、但**并发下对 summary 有 ~10% 真收益**——不过这是**独立的 H2O 式高并发 KV 压缩项目**(注意力信号 + 摘要质量验证),和投机/深注扫描正交。数据:`~/modify-code-runs/suffix-spec-decode/mechanism-sim/context_len.py`。

### 实测:批处理吞吐曲线——"权重摊薄"的直接证据(2026-07-07)

上面反复讲"权重读跨请求共享、批处理摊薄它"。直接量了一版(vLLM TP4、**CUDA graph**、独立请求、短上下文 200 token、每请求生成 200):

| 批 | 聚合 tok/s | vs 串行 | 批效率(加速÷批) |
|--:|--:|--:|--:|
| 1 | 43 | 1.0× | 1.00 |
| 4 | 165 | 3.9× | 0.96 |
| 8 | 312 | 7.3× | 0.91 |
| 16 | 576 | 13.5× | 0.84 |
| 32 | 951 | 22.3× | 0.70 |
| 64 | 1656 | **38.8×** | 0.61 |

- **聚合吞吐 43→1656(38.8×)**,而每请求 decode 只慢 1.65×(4.68→7.73s)——批处理的本质:**单请求略慢、总量猛涨**;
- **批 1-8 效率 0.91-1.00 = 近线性**(读 16GB 权重主导、GPU 没喂饱,多加一个请求几乎免费);**批 32-64 效率 0.70-0.61 = 转亚线性**(每步算 B 份 + 读 B 份各自 KV → 开始受算力/KV访存限制);
- batch=1 的 43 tok/s = **23.4ms/token**,正是 §三"读权重"主导的每步 ~24ms → 首尾闭环;
- **注意**:这是短上下文(KV 小、撑得住高批效率);上下文越长、亚线性拐点越早(见上"并发修正")。数据:`~/modify-code-runs/vllm-batching/`(run_batching.sh + res_batch_n*.jsonl)。

## 追问区(两个最容易误判的点)

### 追问 1:report 注意力 FLOP 占 39.4%,砍 50% KV 减 ~20% FLOP,每步明显吗?

**FLOP 确实 −20%,但延迟只 −3.8%,谈不上明显。** 因为 decode 是访存瓶颈,省 FLOP ≠ 省时间:

- FLOP 口径:注意力 39.4%,砍半 KV → 省 39.4%/2 = **19.7% FLOP**;
- 访存(延迟)口径:权重每步照读 **64GB**,KV 从 5.2GB → 2.6GB,总访存 69.2 → 66.6GB → **延迟只省 3.8%**。

你省掉的注意力 FLOP 本来就"藏在读权重的影子里免费做掉",砍了它 GPU 还得花同样时间读权重。**省的是本不要钱的算力,不是瓶颈。**(summary 更小,~1.4%。例外:高并发时权重读被摊薄、注意力相对变大,砍 KV 收益会涨;但 report 单流生成,就是 ~3.8%。)

### 追问 2:验证 2 个 token 的耗时,是单 token 的 2 倍吗?

**不是,约 1.03 倍——这正是投机解码能成立的根本。**

| 验证 token 数 | 耗时 | 相对单 token |
|--:|--:|--:|
| 1(普通 decode) | ~24ms | 1× |
| 2(1 真 1 草稿) | ~24.6ms | **1.03×** |
| K+1 | 24 + 0.62×K | 边际 **0.62ms/位置**(一步的 2.6%) |

一次前向验 2 个 token,**64GB 权重只读一遍**(两 token 共享),多的那个 token 只是多做一份 FLOP,而 decode 算力没跑满 → 这份算力**藏在同一次权重读的影子里、近乎免费**。所以边际 token ~0.62ms,不是再来一个 24ms。**若验 2 个真是 2 倍,投机就永远省不了时间;正因边际 token 极便宜,草稿一被接受就等于"一次权重读的价钱拿多个 token"。** K 涨到上千(prefill)才变算力瓶颈、才 ~线性。

**两个追问是一回事**:decode 每步 ≈ 读 64GB 权重的时间;FLOP 增(多验)或减(剪 KV),只要不突破访存瓶颈,都几乎不动延迟。省 decode 时间的正道 = **摊薄那次权重读**(投机)或 **缩小权重**(量化),不是动注意力小尾巴。

### 追问 3:那 token pruning 什么场景才真能加速?——大 batch × 长 prompt(2026-07-09 追加)

**对,它的主场就是 B×L 大到 KV 反超权重的服务场景。** 分界线可以精确算出来:每步每卡访存 = 权重 16GB(**跨请求共享,与 B 无关**)+ B×L×0.26MB(KV,**各读各的**)→ KV 追平权重的条件是 **B×L ≈ 25 万 token-positions**(64GB ÷ 0.26MB,聚合口径)。

| 场景 | B×L | KV 占访存 | 砍 50% KV 省 |
|---|--:|--:|--:|
| 我们的 report(B=1, L=20K) | 2 万 | ~7.5% | ~3.8%(追问 1) |
| 我们的 summary(~9 路, L=7K) | 6.3 万 | ~20% | ~10%(§六并发修正) |
| 大 batch 服务(B=64, L=8K) | **51 万** | ~67% | **~33%** ✅ |
| 长上下文服务(B=16, L=64K) | **102 万** | ~80% | **~40%** ✅ |

另外两条它真实立足的理由(不走带宽轴):
1. **HBM 容量 → 吞吐**(生产里常是真动机):KV 占满显存限死最大 batch;砍一半 KV ≈ 塞两倍并发 → 吞吐近 ×2。即使带宽没到 break-even 也成立——省的是**准入容量**,不是每步延迟。
2. **prefill 侧**:注意力 O(L²) 是算力瓶颈(与 decode 访存瓶颈不同轴),prompt 剪枝在超长 prompt 上还省 prefill/TTFT。

**所以本文判决的准确表述**:token pruning **对我们的 workload**(并发 ≤3、段内最多 ~9 路、L=7-20K → B×L ≤ 6 万,离 25 万远)救不了 decode;它的论文/产品立足于大 batch 长上下文服务——**和我们不是一个战场**,不是普遍无用。

## 测量陷阱:从日志反推每步耗时,两个坑(2026-07-07,深注扫描踩过并纠正)

**本轮(深注扫描 step-time 分析)一度报错、经用户逐个揪出的不一致数值——一眼对照:**

| 我一度报的数 | 场景 | 错因 | 真值 |
|---|---|---|---|
| 每步 **126ms** | 从 res 反推 step-time | `n_out`(流式块数≈步数)当成 token 数 → 虚高 ≈每步token(3.7)倍 | **33.8ms** |
| 每步 **8ms** | 从 traj 反推 step-time | 没除 TP worker=4(每步被 4 个 worker 各记一遍) | **33.8ms** |
| 每位置 **0.08ms** | 拟合斜率 | 用了 8ms 那版错 step-time | **0.32ms** |
| 每位置 **0.62ms** | 拿来对比 report | 那是 summary 的五常数口径、不同实验,**不可直接比高低** | 同量级,别比 |
| **"14×"** tokens/块 | 对账真 token vs n_out | 把 ×4 的 traj token 数 ÷ n_out(又混进了 worker 4 倍) | **~3.6×**(且 n_out≈步数) |
| 算式 **"3.72÷29.6=125.7ms"** | 手写 | 漏写秒→毫秒的 ×1000 | =0.1257 秒=125.7ms |

**根因就两个**(下面详述):**n_out≈步数不是 token 数**、**traj 被 TP worker ×4**。多数错数是这两个 bug 单独或叠加(如"14×"=3.6×token/块 混了 4×worker;"0.08"=正确斜率被 4× 步数压小)的产物。

**没受影响、结论仍成立**:主判决表的 **char/s**(`out_len_char/decode_s`,干净)和 **每步token / 接受率**(都是**比值**,×4 抵消)全对 → 路线1 结论(factor 是杠杆、gf4 甜点、depth 哑弹)**不变**。翻车的只有我额外做的 step-time 侧算。

---

深注扫描要验"每草稿位置成本",得从日志反推 decode 每步墙钟。踩了两个坑,每个都让 step-time 错一个整倍数,记下来免再犯:

**坑① `res` 里的 `n_out` 不是 token 数,是流式块数 ≈ 步数。** vLLM **每个 decode 步吐一个 SSE 块**、块里含该步接受的全部 token(report ~3.6 个/块)。所以 `n_out ≈ 真实步数`(实测 8860≈8852),**不是** token 数。用 `每步token ÷ (n_out/decode_s)` 反推,等于把"步数"当"token 数",**虚高 ≈每步token 倍**(report 3.7×:错算 **126ms**,真值 33.8ms)。
- 单位链自查:`3.72 tokens ÷ 29.6 (块/秒) = 0.1257 秒 = 125.7ms`——这个"每秒 29.6"其实是每秒**步数**不是 token,链子从一开始就错。

**坑② 轨迹日志被 TP worker 数重复计数。** `SUFFIX_TRAJ` 按 pid 写文件,**TP4 = 4 个 worker 各写一份、每步记 4 遍**。`glob traj_*.*` 全 sum → 步数/草稿/绝对 token 全 **×4**(比值如 tok/step、接受率**不**受影响,但**步数、每步耗时受影响**)。算每步墙钟必须 `原始记录 ÷ worker数`,否则 step-time 偏小 4×(错算 **8ms**,低于权重地板就该警觉)。

**正确姿势 + 两道物理自检**:
- `每步ms = 墙钟 ÷ (原始记录 ÷ worker数)`;墙钟 = Σ每 req 的 `max(t)−min(t)`(4 份日志时间戳相同,**墙钟不受 ×4**,只有计数受);
- 自检 A:算出的墙钟应 **≈ res `decode_s`**(实测 ratio 1.00 → 墙钟没算错);
- 自检 B:每步 ms 必须 **> 16.7ms**(读 16GB 权重的物理地板,§三)。低于它一定算错了——坑②的 8ms 就是这样露馅的。

## 术语表

- **decode step**:自回归生成里吐 1 个 token 的一次完整前向。
- **访存瓶颈(memory-bound)**:算力没喂饱,时间卡在从显存搬数据。decode(批=1)就是这样。
- **GQA**:分组查询注意力。Qwen3-32B 有 64 个 Q 头但只 8 个 KV 头,所以 KV 缓存比 MHA 小 8 倍(省显存+省 KV 访存)。
- **KV 长度 L**:当前上下文已缓存多少 token 的 K/V。decode 时从 prompt 长度涨到 prompt+输出。
- **FLOP 口径 vs 访存口径**:前者数算了多少;后者数搬了多少字节。decode 看延迟用后者。

## 附:数字怎么来的

全部用 Qwen3-32B `config.json` 实参手算(非测量):权重 FLOP = Σ各投影/FFN/LM头的 2×矩阵尺寸;注意力 FLOP = 4×n_q×head_dim×L×层数;KV 访存 = 2×n_kv×head_dim×L×层数×2B。上下文长度来自 `census_Ceager/q*/llm_calls.jsonl` 的 prompt_tokens+completion_tokens。可复算脚本逻辑见本文 §二~§五公式。
