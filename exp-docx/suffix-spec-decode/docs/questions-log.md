## Q8 参数意义 / 接受率悖论 / 测量陷阱(大白话,详解)
> 参数调整为什么会影响性能,大白话解释各参数意义?为什么接受率高了之后 summary 反而更慢?三个测量陷阱通俗讲解下。

### 8.1 各参数是什么意思,为什么动它会影响性能

先一句话打底:suffix 投机解码 = **打字时的"整句联想"**。模型每吐一个 token,系统拿"最近写的这一小串"去后缀树(用 prompt 内容建的索引)里查"上次这串后面跟的是什么",把后续整段当草稿填给模型,模型**一次前向**把草稿过目,对的全收、错的丢掉。**猜中就省 decode 步数,猜错就白验证**——所有参数都在调"赌注下多大、多有把握才下注":

| 参数 | 大白话 | 调它的后果(实测) |
|---|---|---|
| `max_tree_depth=24` | 联想时**往回看多少个 token** 去匹配(也是树的记忆深度) | 深了匹配更准,但建树查树更贵 |
| `num_speculative_tokens`(suffix 下默认=24)| **一次最多猜几个 token**(草稿上限) | 纯上限,一般不用动 |
| `max_spec_factor=1.0` | **胆量和证据挂钩**:结尾跟树匹配上 N 个 token,就敢猜 factor×N 个。匹配越长越有底气猜越长 | 试了 0.5 = 胆小一半:匹配 10 个只猜 5 个 → 每次赢得少,结果最慢(0.838x) |
| `min_token_prob=0.1` | 树里按**频率**估"这个位置后面跟 X"的概率,≥阈值才敢写进草稿 | 调低(0.05)= 冷门续写也敢猜 → 猜得多但命中低(28%),浪费验证 → 0.889x;调高(0.5)= 只猜十拿九稳的 → 命中高但下注少 → 0.917x |
| `max_cached_requests=10000` | **全局树背多少条历史请求的输出**(跨请求记忆) | 设 0 = 只用本请求 prompt 树,失去"抄别人/抄历史"的来源(0.872x) |

为什么"下注策略"影响速度:**赢 = 一个被接受的草稿 token = 省一步 decode**(一步 = 把 32B 权重从显存整读一遍,很贵);**输 = 草稿被拒 = 白搭了验证**。太激进输太多,太保守赢太少,默认 0.1 正好卡在中间——所以扫出来两头都比默认差。

### 8.2 为什么接受率高了,summary 反而更慢

因为**接受率是"命中率",不是"赢的总额"**。省时间靠的是**接受的 token 总数**(每接受一个省一步),不是命中的比例。

拿实测数字说:
- `factor=0.5`(接受率 41%):总共只敢猜 16275 个 token,接受 6712 个 → **总共只省 ~6700 步**;
- `prob=0.05`(接受率才 28%):猜了 34788 个,接受 9831 个 → 命中率低,**省的步数反而更多**。

打个比方:一个赌徒只赌十拿九稳的局,胜率 90% 但一晚只下 3 注,赢 30 块;另一个胜率 30% 但下 100 注,赢 300 块。**看胜率前者漂亮,看钱包后者厚。**

再加最关键的一层:**只要开了 suffix,每一步 decode 都要交"入场费"**——vLLM 明说开 suffix 就**关掉异步调度**(CPU 准备下一步和 GPU 算这一步没法再重叠,每步多一截等待),外加每步维护/查询后缀树。这笔税**每步都收、跟猜没猜中无关**。

summary 的账:
> 总耗时 ≈ 步数 × (每步原价 + 固定税) − 接受 token 省下的步数

summary 的照搬是碎的(中位一段 41 字),**不管怎么调参,能接受的 token 总量就那么点** → 省的步数抵不过每步都交的税 → 全部配置都在 0.84~0.96x 徘徊。接受率被调参推高,只说明"下注更谨慎了",赢的总额反而缩水,税一分没少交——**所以接受率高了还更慢**。

### 8.3 三个测量陷阱,通俗版

**坑① 热树:server 是有记忆的,别拿"背过题的它"做对比。** 全局后缀树默认把**之前请求的输出**都记下来(1 万条)。同一批题在同一台 server 上测第二遍,等于**开卷考试**——上一遍生成的答案它全背下来了,这次整段猜中。实测:prob=0.5 那台 server 之前跑过一模一样的 60 条 summary,重测假显 **1.9x**(有条 decode 从 6.1s 缩到 1.1s);换全新 server 冷启动重测,**只有 0.917x**。教训:配置 A/B 对比,**每个配置都得换全新 server**。这个"记忆"在生产里是真福利(重复相似 query 越跑越快),但做对比实验时它是污染源,俩事别混。

**坑② 并发:投机解码是"捡 GPU 的闲便宜",人多了就没便宜可捡。** decode 一步只算 1 个 token,但要把整个模型权重从显存读一遍——瓶颈在读显存,**GPU 算力大部分闲着**。验证草稿正好用这些闲算力,等于**免费搭车**。并发一大、batch 满了,GPU 算力吃紧,草稿验证要跟正经请求**抢算力**,搭车变加塞。所以串行(batch=1)测出的 1.76x 是**最漂亮的上界**,真实并发下会缩水——上线前得小流量再验。

**坑③ 数块数 ≠ 数 token:流式接口会"打包发货"。** 测速脚本原来按流式返回的"消息块"数 token。不开投机时,基本一块 = 一个 token,数得准;开了投机,一步验证通过 5 个 token 会**打包成一块**吐出来——块数严重少数,算出来的 tokens/s 假低(第一版就这么把 suffix 算成"更慢")。改成**数字符**(char/s)绕开。另一个小坑:temp=0 也不保证两次输出一模一样(batch 组合不同、浮点求和顺序不同),单条对比有噪声 → 最后用"同题配对 + 总字符÷总时间"来算,离群值带不偏。

## Q9 为什么重叠大但 suffix 没效果(summary,详解)
> 为什么 overlap-v2 指出 RESEARCH_SUMMARY_TEXT 有大量输出和输入重叠,但实际用 suffix 效果并不好?

**核心**:overlap 报告量的是"照搬的**总面积**",suffix 吃的是"**单段连续长度**"——summary 的照搬是**碎纸屑**,report 的照搬是**整页复印**。同样"重叠多",对 suffix 是两种完全不同的命。

### 9.1 v2 报告里被忽略的两行(数据自己就是答案)

| | Coverage@8(总面积)| 最长单段(中位)| 段数 | 阈值提到 @30 后 |
|---|--:|--:|--:|--:|
| SUMMARY | 54% | **41 字** | ~38 段 | **崩到 9.5%** |
| REPORT | 91% | **226 字** | ~170 段 | **仍有 74.5%** |

"阈值敏感度"表是铁证:把"算作照搬"的最短片段从 8 字提到 30 字,summary 覆盖率从 54% 崩到 9.5%——**它的重叠几乎全是 8~30 字的小碎片**;report 提到 30 字仍有 74.5%,是真的整段抄。54% 是把 38 个小碎片面积**加总**出来的,单看它会高估 suffix 能吃到的肉。

### 9.2 碎片对 suffix 几乎无肉可吃:每段要交三笔钱

**① 起步"对暗号"费,一段交一次。** suffix 不是先知:模型开始抄某句时,它结尾的上下文还是自己的过渡语,跟 prompt 树对不上,**猜不了**;要等真抄进去几个 token,树才"锁定"这段位置开始出草稿。一段 12 token 的碎片,前 3~4 个用来对暗号,只剩后半段能白捡。**38 段就交 38 次起步费;report 一段 226 字只交一次,然后收割一长串。**

**② 草稿长度跟匹配长度走(`max_spec_factor=1.0`),短段还没等胆子变大就结束了。** 刚锁上时匹配才 3 个 token → 只敢猜 3 个;匹配 6 个 → 猜 6 个。碎片段在"胆子养大"之前就抄完了,每步净赚 1~2 个;report 的长段能把草稿养到十几二十个 token,一次前向吞一大口。实测的逐位置接受数就是这个指纹:**pos0 接受 614、pos1 214、pos2 83、pos3 34**——绝大多数步只多收 0~1 个 token,正是"短段反复重新锁定"的形状。

**③ 微改写 = 投机失败 + 重新对暗号。** summary 抄的时候爱小改:换个连接词、调个"的/了"。字符级统计里这只是把长段断成两个短段、**面积照样计入 coverage**;但对 suffix 是一次实打实的草稿被拒,还得重新走一遍①。"看起来在抄"和"token 一字不差地抄"之间有折损,碎片式照搬折损最狠。

### 9.3 另一半账:没抄的地方照样交税

summary 还有 ~46% 内容不在任何照搬里(自己组织的话)。这些步 suffix 什么都赚不到,但**固定税照收**(开 suffix 就关异步调度 + 每步查树)。summary 本来就短(~1.7 千字),碎片里抠出的那点节省填不平全程的税——这就是 0.955x 且**怎么调参都翻不正**的原因。

### 9.4 一句话收拢

overlap 报告和 suffix 实验**不矛盾,反而互相印证**:report 是"91% 面积 + 226 字长段"→ 1.76x;summary 是"54% 面积但全是 41 字碎片"→ 略亏。**判断 suffix 值不值得开,别看 coverage 总量,看长片段(@30 阈值)的覆盖率**——这个指标下 summary 只有 9.5%,早就预言了它吃不到肉。
(小注:overlap 统计的是当时录制的输出,重放用 temp0,输出不完全一样——只影响幅度,不影响这套结构性解释。本目录 overlap-replay.md 用 replay 实际输出逐条验证了这一点。)

**代码级展开** → [why-high-overlap-no-speedup.md](why-high-overlap-no-speedup.md):按源码机制逐 token 模拟真实请求,i53(cov8=86%)vs i61(cov8=85%)同样重叠面积、实测 0.62x vs 2.9x;summary 全量理想上界仅 1.33x,交 18% 税后天花板 ≈1.09x——"调参救不了"钉死在机制上限。

## Q11 机制细节五连问(pattern 怎么取/多处出现/置信度/为什么还慢/async 税)

- **关 async scheduling 的代价**:异步 = CPU 排班下一步与 GPU 算当前步重叠(墙钟≈GPU 时间);suffix 每步净吐 1~25 不可预知、草稿是采样后 CPU 才算出的,下一步 batch 依赖本步全部结果 → 只能串行,CPU 段从"藏在 GPU 后面"变"串在 GPU 前面",每步多几毫秒,即 ~18% 固定税的主体。

## Q14 树的数据结构:不是 SAM/怎么限深 24/全局树怎么装多个请求
> ①它跟正常 SAM 实现似乎有区别,10k prompt 怎么限深 24、记录所有 ≤24 子串的位置?②SAM 是对单一序列建的,全局树怎么合并多个请求的序列?③`self._local_trees[req_id]` 是每请求一棵树?全局树也是每请求一棵吗?

**答**(详解已并入 18 号文档 §三-2):
- **前提纠正:它不是 SAM**(SAM 是我们离线分析脚本用的),是**深度封顶的广义后缀 trie + 频次计数**,没有 suffix link。
- **限深 = `_active_nodes` 滑动窗口**(suffix_tree.h:164-168):每序列一个 ≤24 个活跃指针的 deque;新 token 来时队尾压入根指针(新窗口)、**超 24 弹队头**(该窗口停止生长)、每个指针下走一步。树的根路径 = 每个起点的 ≤24 窗口 → 所有 ≤24 子串可从根走出,count=出现次数,位置靠 ref_seq/ref_idx 引用不拷贝。每 token O(24),10k prompt ≈24 万次操作。
- **全局树不拼接序列**(拼接会造出跨请求假子串):同一棵 trie 按 seq_id 挂多条独立序列,各自贡献 ≤24 窗口,**count 跨序列累加**(频率投票天然跨请求),`remove(seq_id)` 按 endpoints 单独摘除。朴素 trie 恰好让限深/多序列/删除三件事都容易——SAM 的全局结构跟它们都打架,而 speculate 最多回看 24,深索引无用。
- **布局**:本地树 dict、**每活跃请求一棵**(只装 seq 0 = 本请求 prompt+已生成,结束整棵 del);**全局树整个引擎单例一棵**(cache.py:80),装 ≤max_cached_requests 条序列,**只收各请求的输出**(prompt 不进全局树)。

## Q15 零基础:后缀树到底是什么、和 SAM 什么区别(手把手)
> 后缀树和 SAM 有什么区别?我只知道 Trie。/ 广义后缀 trie 是什么,跟狭义后缀 trie、跟 trie 什么区别?/ 我对 vLLM 的后缀树投机解码一窍不通,把我当小白手把手教。

**答**:已写零基础教程 → [../../19-suffix-tree-explained.md](../../19-suffix-tree-explained.md)(LMCache/claude-docx 同步一份)。四个概念一句话版:
- **Trie**:通用容器,喂什么词存什么词,查询从根走;只认"从头开始"的匹配。
- **后缀 Trie**:结构就是 Trie,只是喂的词固定为"一条序列的全部后缀"——因为"任何子串 = 某后缀的前缀",这样喂完就能查任意子串;节点 count 顺手等于子串出现次数。
- **狭义 vs 广义**:一条序列一棵 vs **多条序列共一棵**(count 跨序列累加、按 seq_id 分账、可单独删除)——vLLM 的全局树用的就是广义。
- **vLLM 的树**:限深(每条后缀只喂前 24)+带频次+广义的后缀 Trie(+路径压缩);**不是**教科书后缀树(那是 O(n) 压缩版,Ukkonen 构造)也**不是** SAM(endpos 等价类自动机,我们只在离线分析 build_matchlen 里用它)。
教程含:7 token 小序列逐 token 建树表(滑动指针队列)、speculate 手把手(枚举/投票置信度/score 择优/验证两种结局)、序列 B 加入/删除演示广义、§八 教科书后缀树(banana 图)与 SAM(endpos)对比表。

## Q16 min_token_prob 调低(0.05)为什么反而更慢?"多猜只是白猜"不对吗?
> 按道理接受率调低最多导致草稿不命中,但不会伤害原有的 best score candidate?

## Q19 为什么 token 级重叠和字符级重叠对加速的影响不一样?(大白话)
> 可能是什么原因导致的?你分析下。

**答**:字符和 token 是两把不同的尺子,而机器只用 token 尺子干活;字符尺子还有"汇率"问题。三个原因,重要性递减:
① **干活的单位是 token,字符只是包装**。suffix 的猜(匹配 token 序列)、验(比 token id)、赚(每接受 1 个 token 省一步)全在 token 空间;字符级统计=拿包装箱尺寸估货值。
② **"8 字符"这个门槛在不同语言里是不同的货(最响的发现)**:summary 组内 字符cov8 与 每token字符数(cpt) 相关 **+0.88**——cov8 几乎在测"这篇是英文还是中文"(cpt 范围 1.3~5.9)。英文 1 token≈4-6 字符,重复仨术语就计入 cov8;中文 8 字符=4-6 个 token,门槛严得多。本实验英文题多为分析型(慢)、中文题多为检索型(快),cov8 被语言带跑 → 组内 −0.23 的假反向。**控制语言后假象消失**:cpt 低半(偏中文)cov8→加速 r=+0.42(正!),cpt 高半(偏英文)r=+0.01(纯没用);而 harvest 两半都好使(+0.45/+0.74)——token 尺不受汇率影响。
③ **字符长段换算成 token 会缩水,机制起步费按 token 收**:i37 字符段均长 94 字符"看着长",cpt=5.9 → 仅 ~16 token,刚够锁定+滚两下;i61 字符段 79 字符"看着短",cpt=2.0 → ~40 token,滚到顶格。同字符长度,中文段 token 含量是英文的 2-3 倍。另有 BPE 边界折损(字符相同≠token 相同,段两端磨掉 1-2 个),短段被磨得更狠。
一句话:**字符口径是"外币"记账、每条请求汇率还不同;token 口径是机器结算的本币。跨语言负载下选指标必须用 token 口径(harvest)。**

> 我理解 harvest 就是草稿生效的 token 数,占据整个输出的比例?

**答**:方向对,差一层——harvest 是**光看文本形状估出来的"白捡区"占比**(数"前面已连续照搬 ≥3 token、且下一个还在延续"的位置 ÷ 输出总 token,analyze_replay_overlap.py:46),算它时**没有跑任何草稿/验证**;"草稿实际生效占比"是模拟器的 accepted/n_out(或 server 的 accepted 计数,但那是全 run 累计拆不到单条)。两者 80 条相关 **0.99**、几乎重合,但单条有出入:"锁 3 个才开张"的近似**偏严**,短匹配若续写独一份照样命中(i53:harvest 9.5% vs 模拟实际接受 19.7%);长段照搬时**非常准**(i61:71.0% vs 69.9%)。用法:harvest 当"这条有多少肉"的静态预估(不用跑 server 就能算),要实际生效数看模拟器 accepted 或 server metrics。已在 why-high-overlap 附录表头补了此说明。

## Q21 优化方向六问(多候选并行/async兼容/步级题级预测/抄vs总结/score算法/其他)
> 针对 deep research agent 解题过程做优化,六个思路求分析+补充。

**答**:已整理成文档 → [optimization-round2-analysis.md](explore-idea/optimization-round2-analysis.md)(含上界阶梯、oracle-候选选择实验、优先级表)。要点:
- **①多候选并行**:平手规则实为 `>=`(更长 match 赢)+本地优先,非"取第一个";oracle 实验(每步所有 match_len 候选都验证)显示 score 单候选已拿到 97-100% 收益(i53: 95→103 token,被打败步数仅 2-3%)——**零接受步的本质是"模型在说自己的话",所有候选都错,不是选错**。C++ 有 _speculate_tree 但 vLLM 无树验证,低优先级。
- **②async 兼容**:难点=下一步 batch 依赖本步接受数(1~25)+草稿是采样后 CPU 产物;EAGLE 系兼容因草稿产于 GPU 前向内。自改调度器不划算;**推荐降税**:propose 增量化(匹配状态机)、无匹配 K 步跳过、C++ 化;且并发下税被稀释(v2 端到端 summary x1.00)。
- **③预测**:步级预测省不掉税(税是 server 级)→只值"下注尺寸"自适应;题级路由可行(harvest r=0.90)但上界 12pp、端到端下更小,且需双池。进阶:前 K token 探针+LMCache KV 迁移中途换池。
- **④抄 vs 总结**:行为层=检索/罗列型抄、分析/综述型总结(代理:指令动词/URL密度/压缩比/单多源/语言);内部状态层=抄写签名是 induction head(注意力钉在 prompt 匹配位+逐步前移)+argmax≈1 低熵;prefill KV 预测力弱(读题≠作答)。路由用文本特征更便宜。
- **⑤score 算法**:score=Σ累积prob≈期望接受数,max score 已是 EV 最优;改进点:draft 级 score 地板(修 Q16 挤位)、全局树 margin δ/在线调权、成本感知 EV−λ·草稿长(并发)、count 平滑。全是 proposer 小改可 A/B。
- **⑥最对症**:summary 81% 零接受步 suffix 结构性够不着 → **换/混学习型 drafter(EAGLE3)**,不依赖照搬;或按阶段分池(report=suffix, summary=EAGLE)。降税包=L4+max_tree_depth(唯一没扫杠杆)。已否决勿走回头路:L3 膨胀、summary 调参。大局:v2 端到端全开已净赚,优先级=守住 report 收益+降税,非抢救 summary。

## Q22 oracle 实验是"全部草稿都命中 vs 真实"吗?
> oracle 实验我理解应该是对比最优情况(全部草稿都命中)和真实情况?

**答**:不是——"让草稿全命中"这个上界**不存在**:命不命中由模型实际要说什么决定,模型说自己的话时任何草稿都不会命中,这是物理约束不是决策失误。**oracle = 给某一个决策环节装"事后诸葛亮",单独称出该环节做错决定的损失**。本次装天眼的环节是"从多份候选草稿里挑哪份去验证"(答问 1 的"只验 best score 浪不浪费"),结论:选择环节几乎无损(97-100%)。你想的"最优 vs 真实"总对比对应上界阶梯的另一层——模拟器的"理想机制上界 1.33x vs 实测 0.955x"。完整阶梯(实测→免税→选择天眼→分叉天眼→硬天花板=harvest 形状)见 [optimization-round2-analysis.md](explore-idea/optimization-round2-analysis.md) §一。
**两层 oracle 合并读**:窄版(现有候选间挑最好,sim_multi_candidate)只 +1~2pp——score 几乎不漏;宽版(分叉无限宽全验证,sim_tree_oracle)summary 理想 1.33→1.50x(+11.3% 步)——因为被拒的大头是"模型说自己的话",树的任何分支里都没有,验证再多候选也全错。

## Q23 四个澄清 + EAGLE/MTP 讲解文档
> ①降税三件套小白版没懂,好处和收益是什么?②探针为什么量熵、怎么测?③induction head 本质是不是还是用熵?④score 地板是不是说"优化算法没意义、判断何时投机更有价值"?⑤另写一个 md 通俗讲 EAGLE3/MTP。

**答**:
- **①降税三件套**:税 18% 里 propose 的 CPU 是"每一步从头拿 24 个 token 枚举后缀配树"——相邻两步序列只差一两个 token,夹书签(增量匹配状态机)就能把几百次节点访问降到几次;连续 K 步无匹配的纯转述段隔几步再查(一条街都没货就隔几家问);Python 循环下沉 C++。**收益的特殊性:无条件退税**——别的优化要赌赢才赚,这个对所有请求所有步生效、与命中无关,预计能抠掉税里 propose 那一块(全场 3~6pp 量级)。
- **②探针的熵**:熵=下一个词的犹豫程度。照抄≈背书,分布集中、熵低、top1≈1;自己组织语言,同义说法多、熵高。**测量零成本**:decode 每步本来就算了全词表分布,H=−Σp·logp 或 top1 概率,vLLM 开 logprobs 即得,32~64 步取均值;照搬率用 harvest 同款匹配(毫秒)。照搬率是事实、熵是倾向,双信号都免费。注意探针路由是"迁池"(spec 是 server 级,不能中途开关),迁移靠 LMCache 搬 KV。
- **③induction head ≠ 熵**:熵看输出端分布形状;induction head 看中间层注意力(眼睛盯着原文位置逐步滑动)。常同时发生但可分(低熵不在抄:编号/套话;在抄不极低熵:边抄边换词)。**线上只能用熵+照搬率**(FlashAttention 不物化注意力矩阵);induction head 属离线研究/机制解释。你说"本质上还是熵"对落地版成立,对机制不成立。
- **④score 地板 ≠ "优化算法没意义"**:oracle 结论是"**挑哪份**"环节几乎无损(挑马眼光没问题);地板管"**这份值不值得交**"(有些场次所有马都是瘸的,别下注)。它就是最便宜的步级"何时投机"判断器——score 本身就是期望收益估计,加阈值即可。两者不矛盾:选择算法已近最优,参与决策还来者不拒。
- **⑤已写** → [../../20-eagle-mtp-explained.md](../../20-eagle-mtp-explained.md)(LMCache/claude-docx 同步):听诊器学徒比喻(EAGLE 吃大模型 hidden state 连猜 3~5 个,不需要书里有原文)、MTP=出厂自带多预测头(Qwen3 无→只能 EAGLE 后装)、vs suffix 互补对照表(report 池 suffix/summary 池 EAGLE 的依据)、落地清单。

## Q23 为什么"分叉支路无限宽"是树状投机的理论天花板?
> 树-oracle 实验里,为什么支路无限宽就是理论天花板?

**答**:线性投机的失败拆两类:(a) **选错支路**——真实续写在树里该上下文之后出现过,但草稿走了别的支路/被 prob 门槛掐掉;(b) **树里没有**——模型在说自己的话,文本形状决定,与怎么猜无关。树状投机(任意宽度 k)只能修 (a);**无限宽 = 所有支路全部提交验证,真实续写只要存在于树中任一路径就必然命中,(a) 修到零**。单调性:验证无损,多提交支路只多命中不干扰(错的只是白验证),故任意有限 k 的命中集合 ⊆ 无限宽 → 无限宽是这族方案收益的上确界。比方:单钩(线性)→多钩(k路)→一网打尽(无限宽);**一网打尽都只多捞 0.17x(summary 1.33→1.50x),说明水里没鱼——大头是 (b),再宽也捞不到**。两个边界:①保留了预算 m×factor(量的是纯宽度维度,放宽预算是另一杠杆已扫过);②免掉了全部验证成本(现实中无限宽成本发散),真实方案还要再扣有限宽/验证开销/tree attention 改造三笔。实现:sim_tree_oracle.py 直接拿真实输出过滤匹配位置集合、能走多深接受多深(等价全支路验证)。

## Q25 勘误:"碎片对机制无价值"说法不准
> token 级别的 @8 重叠率跟加速比也是正相关的,"碎片对机制无价值"不准确。

**答**:已修正(周报 §5 + variance §二)。精确表述:**段价值 ≈ max(0, 段长 − 锁定费 ~3 token),门槛按 token 算**。字符 cov8 失真是因为它的"8"落在门槛之下(英文 8~30 字符 ≈ 2~6 token,大量短于锁定费);token cov8 的"8"(≈30+ 字符连续)已越过门槛、每段净赚 ~5+,所以 token 口径各阈值都与加速显著正相关(tok cov8 合并 r=0.90,tok cov4 也有 0.85)。"无价值"只适用于**短于锁定费(~≤3 token)**的段(且非严格为零——独苗续写的 1~2 token 匹配也偶有命中,i53 模拟实际接受 19.7% > harvest 静态估计 9.5% 即此)。失真的是字符度量,不是"短段"概念本身。
