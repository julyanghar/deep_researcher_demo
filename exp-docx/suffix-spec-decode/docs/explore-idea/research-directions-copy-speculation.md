> 缘起:研究的原始 motivation = **LLM 会照搬输入/其他 agent 的输出作为自己的输出**。EAGLE 类学习型 drafter 与此正交,定位降为必比 baseline(见 [optimization-round2-analysis.md](optimization-round2-analysis.md) 问 6 追问)。本文把 P0 + D1~D4 五个方向展开成可执行的研究计划:每个方向讲清"**为什么做(解决什么问题)→ 优势与预期收益 → 具体怎么做 → 风险在哪**"。
> **2026-07-05 更新**:①按税拆分终局结论刷新全部证据基线(固定税=混合边界慢发射,Ceager=双口径冠军,异步直测仅 0.5ms,并发下垃圾草稿要付真钱);②新增 **P0:自适应投机门控(题型先验+前 32-64 token 探针)——用户选定的第一个动手方向**。
> 阅读顺序:赶时间只看 §〇(资产盘点)、§P0(先做的)、§一(主菜 D1)、§五(论文骨架+行动清单)。

## P0(用户选定先做):自适应投机门控——题型先验 + 前 32-64 token 探针决定"这条请求要不要投机"

### P0.1 为什么做这个:解决"投机是全局开关,但值不值得投机是每条请求自己的事"

现状的结构性错配:vLLM 的投机是 **server 级全局开关**——开了,**每条请求、每一步**都要发草稿、都要陪着验证;可我们的数据反复证明"值不值得投机"在请求间天差地别:

- **题型维度**:report 每步接受 ~3+(照抄输入,e2e 2.29x);summary 平均只有 ~1.4,且 **81% 的步是零接受**(创作段纯陪跑);JSON/规划类短调用几乎没得抄;
- **题目维度**:同是 summary,token 级重叠好的题 harvest 30%+,差的题 <5%——低重叠题的每一个草稿都是白发的;
- **并发维度(税账本的新证据,这条最要命)**:batch=1 时垃圾草稿搭权重读便车≈免费,**并发下要付真钱**——ngram 4.9% 接受率的洪水把 e2e summary 打到 0.75x,CeagerK8pad 的 13% 洪水让两段全输。**"发了不中"不再是无害的,是负资产。**

一句话:**系统对每条请求盲目地全力投机,而"该不该投"的信号(题型+开头几十个 token 的实际接受)其实便宜到近乎免费。** 把这个信号用起来,就是 P0。

### P0.2 优势与预期收益

**优势(为什么这是好的第一刀)**:
1. **完全无损**:门控只决定"要不要发草稿",验证语义一个字不动——关了=退化为普通 decode,开了=原样 rejection sampling。红线(无损性)自动满足,不需要质量复核也知道输出不变;
2. **信号免费**:探针期的接受数,proposer 在 `propose()` 里本来就看得到(每步传入的 `sampled_token_ids[i]` 长度 = 上一步接受数+1),零额外计算;题型先验也是编排器现成知道的;
3. **改动极小**:单文件 ~30 行,env 门控,和已有的 SUFFIX_PAD_DRAFTS 插桩同款做法,不碰调度器/验证器;
4. **是 D2(混合仲裁)的最小可部署切片**:D2 问"每步该用哪个 drafter",P0 问"这条请求该不该用 drafter"——同一族信号(接受 EMA/match 状态),先用单 drafter 的 on/off 把信号和阈值调通,D2 的风险就消了一半。

**预期收益(对着冠军 Ceager 算,分口径诚实报)**:
- **replay batch=1:≈0**。全 eager 下草稿近乎免费(Ceager 33.9ms ≈ C0 34.0ms),关掉也省不出什么——这一格是 sanity check(证无损无害),不是收益来源;
- **e2e 并发:这里才是钱**。三笔可期的账:①低重叠 summary 请求不再往共享 batch 里灌垃圾验证位(v2 数据:81% 零接受步的那些请求,每步白占 2-8 个验证位);②防洪水:在 ngram/k8pad 这类满发配置上,门控能把 0.75x/两段皆输救回接近不投机水平(强对照实验见 P0.5);③**解锁"报告大 k"**:目前 k 是全局的,summary 拖着 report 不敢开大;门控后低收益请求自动归零,k 可以按 report 的需要放开。量级估计:e2e summary 段 +3~8%(尾部题更多),report 段不动,系统对负载组成的鲁棒性显著提升(这条是质变,不是百分比)。

### P0.3 具体怎么做

**决策规则(两层,先验定参、探针定夺)**:

```
请求到达:
  题型先验(可选) → 定探针参数:
    REPORT 类   → 直接常开(先验足够强,2.29x 不需要探针再确认)
    SUMMARY 类  → 探针模式,W=48, τ=0.25
    JSON/短调用 → 直接常关(输出<100 token,探针本身都嫌贵)
  没有题型信息 → 一律探针模式(纯 server 侧也能跑)

探针模式:
  前 W 个输出 token:正常投机,累计 accepted_probe / drafted_probe
  W 结束:accepted_probe/steps_probe ≥ τ ? → 常开
                                      < τ ? → 关闭(不再 speculate,树照常喂)
  关闭期间,每生成 R=192 token 重探一次(重开 16 token 窗口)
    ——防"开头创作、中段照抄"的阶段结构被一票否决
```

**代码钩子(全部在一个文件里)**:改 [suffix_decoding.py](../../../../anaconda3/envs/lmcache/lib/python3.12/site-packages/vllm/v1/spec_decode/suffix_decoding.py) 的 `propose()`:
- 每请求状态字典 `{req_id: (tokens_out, drafted, accepted, gate_state)}`——`accepted` 从传入的 `sampled_token_ids[i]`(长度−1)累计,`drafted` 是自己上步发的;
- 被门控关闭的请求:跳过 `speculate()` 调用、`draft_token_ids.append([])`(空草稿路径已有,C0 就是这么跑的);**`add_active_response` 照常执行**(树持续喂新 token,重探时立刻可用);
- env 门控:`SUFFIX_GATE=1 SUFFIX_GATE_W=48 SUFFIX_GATE_TAU=0.25 SUFFIX_GATE_REPROBE=192`,不开=零行为变化;
- 每次门控切换打一行可 grep 日志(req_id/时机/当时接受率)——验收和事后分析都靠它。

**题型先验的接线(可选增强,三个方案按侵入度排)**:
1. **不接线(P0 默认)**:纯探针,server 侧自治,一行客户端都不改;
2. 客户端把 tag 塞进 OpenAI 请求的 `user` 字段/自定义 header,vLLM 侧透传到 request metadata(**透传路径待核**,是 P0 里唯一需要读 vLLM 接线代码的点);
3. 按 tag 分流到不同 server(重,违背单 server 原则,不做)。

**第 0 步(动服务器之前,半天、零 GPU):离线仿真定参**。我们有现成资产——v2 的 100 题 per-request 接受序列(harvest 数据)+ 机制模拟器:把门控规则跑在录好的接受序列上,扫 W∈{32,48,64}×τ∈{0.15,0.25,0.35}×R∈{128,192,256},输出每组参数的"误杀率(照抄段被关掉的损失)vs 省draft率",挑 Pareto 点。**参数不是拍脑袋,是拿历史数据仿真出来的**——这一步同时产出论文里的敏感性分析图。

### P0.4 解决了目前的什么问题(对号入座)

| 现存问题(实测) | P0 怎么解 |
|---|---|
| 并发下垃圾草稿付真钱(ngram 0.75x/k8pad 两段输) | 低接受请求探针期后自动闭嘴,洪水从源头掐断 |
| summary 81% 零接受步陪跑 | 这些请求 W 个 token 后归零成本(只剩喂树) |
| k 全局一刀切,summary 拖累 report | 门控后 k 按 report 放开,summary 自动豁免 |
| suffix 自家 CPU 机件 ~1.6ms/步 | 关闭请求省掉 propose+metadata+sampler 大部分(空草稿路径) |

### P0.5 评测方案(判定门槛预写死,防事后圆)

- **Sanity(replay40,batch=1)**:Ceager+gate vs Ceager——预期两者相等(±2%),且 REPORT 完全不受影响(先验常开)。不等 → 门控实现有 bug;
- **主战场(e2e 10 题)**:Ceager+gate vs Ceager——判定:SUM tok/s **不降**且尾部(逐题 decode 最慢的 3 题)改善,REP 持平;
- **强对照(洪水救援)**:CeagerK8pad+gate vs CeagerK8pad vs Ceager——判定:gate 版把 13% 接受率的洪水损失救回一半以上(SUM 从 23.7 拉回 ≥25);这一格是"门控价值"的最强直接证据,因为病人症状最重;
- 全程带门控切换日志,事后按题统计误杀率(照抄段被关掉的 token 数占比)。

### P0.6 风险与对策

1. **阶段结构误杀**(summary 开头是创作段,探针期接受低→关闭→中段引用搜索结果的照抄段错过):周期重探(R=192)兜底;增强版用**match 触发重启**——树喂着新 token,匹配长度 ≥4 时立刻重开(match 状态是 proposer 免费信号,D2 的信号①);离线仿真会直接量出误杀率;
2. **参数过拟合小样本**:定参用 v2 100 题仿真,评测用另 10 题,分离;
3. **题型透传接线不通**:退回纯探针模式(方案 1),损失的只是"report 免探针"这点小便宜;
4. **与热全局树交互**(e2e 热树把接受率抬到 40%,探针期可能都过阈值→门控形同虚设):那说明这个负载下本来就该全开,门控闲置无害——它是保险丝,不是常载调节器。

## 一、D1(主菜):多智能体输出级冗余的系统性利用

### 1.0 为什么做 / 解决什么 / 预期收益(总起)

**为什么做**:多智能体系统里"谁抄谁"不是随机的——它沿着任务图的边走(report 抄 summaries、兄弟对相似输入给相似响应)。**编排器手里攥着这张图,投机系统却在用一个无差别 FIFO 缓存撞运气**——这是全系统最大的一块被浪费的信息。

**解决什么问题(三个实测痛点)**:
1. **兄弟源基本没被吃到**:union 边际增益中位 12.9pp(prompt 14.3%→prompt∪兄弟 27.7%,几乎翻倍)——这块可收割量现状只有全局树在碰运气地捡(v2 实吃锚点:互抄使接受率 26.4%→42.5%,离上界还有大段距离);
2. **全局树跨题污染**:Q16 式"看着像、第一位就错"的假命中,以及 FIFO 驱逐导致的时机错失;
3. **收益不可携带**:全局树是 server 本地状态,双池路由/LMCache 迁移一发生,投机收益清零。

**优势与预期收益**:
- **收益的数字依据**:harvest→加速是 r=0.90 的剂量反应关系;把 summary 的可收割从 14.3% 抬到 27.7%,意味着大多数题跨过打平点(旧口径 15-20%;**Ceager 把固定税砍掉后打平点还会更低**,同样的 harvest 变现更多)。定性预期:summary 段从"平手"推向净正,report 段因定向树(无污染、即时)接受率再抬;
- **优势是结构性的**:定向(无污染)、即时(不等 FIFO)、可控(编排器决定谁喂谁)、可携带(hints 随请求走)——四条都是全局树**架构上给不了**的,不是调参能追平的;
- **学术定位**:decode 侧跨 agent 草稿复用截至知识截止未见成体系工作(待文献核查钉死),且和我们自己的 KVCOMM(prefill 侧)拼成"两层复用"的完整故事。

### 1.1 现象学先行:照搬的四种来源(taxonomy = 分类清单:把"抄来的内容从哪来"分门别类)

| 类型 | 定义 | 我们的实测 | 草稿源何时可得 |
|---|---|---|---|
| **T1 抄输入** | 输出照搬自己 prompt 里的内容 | report 抄 summaries(91%);检索型 summary 抄搜索结果 | **请求到达即有**(prompt 里) |
| **T2 抄自己** | 照搬自己已生成的前文 | 列表/模板重复(本地树命中) | 边生成边有 |
| **T3 抄兄弟** | 与兄弟 agent 输出逐字重合 | v2:同题 summary 互抄 union 31.7% **>** 被自己 prompt 覆盖的 27.8% | **兄弟运行时流式可得** |
| **T4 抄祖先** | 跨轮/跨会话照搬(上轮输出进本轮 prompt) | v2 多轮画像负载 | 上轮结束即有 |

T3 最有意思:兄弟 researcher **没读过彼此的输出**,重合来自"对相似输入的相似响应"——这是**数据流之外的冗余**,现有系统只有全局树在"撞运气"地吃它。

**T3 实测(2026-07-04 终审;经两次口径更正:初版误用字符级面积、二版误用 v1 数据,均由用户纠正)**

**口径说明(全部 token 级,先读这个再看表)**。所有指标都基于同一个原子量:
> **matchlen[j]** = 这条 summary 的第 j 个 token 结尾处,往回数、能在"资料库"里找到的最长**连续 token 串**的长度(SAM 后缀自动机线性时间算出;token 用 Qwen tokenizer 切)。

"资料库"换成谁,就得到不同口径:

| 口径 | 资料库 = | 怎么算 | 含义 |
|---|---|---|---|
| **互抄 harvest** | 同题**其他所有** summary 拼接(接缝插唯一负数分隔符,防拼接处拼出假匹配) | #{j: matchlen[j]≥3 且下一位还在延续} ÷ 总 token 数;题级聚合 = Σ可白捡 ÷ Σtoken(长度加权) | "先连抄 3 个锁定,之后的延续才算白捡"——兄弟源的可收割占比(打平点参照 15~20%) |
| **prompt harvest** | 这条 summary **自己的 prompt** | 同上 | suffix 现状(本地树)能吃到的部分 |
| **union 边际增益** | prompt∪兄弟 vs 仅 prompt,两次 harvest 相减 | harvest(prompt∪兄弟) − harvest(prompt) | 兄弟源在 prompt 之外的**净增量** = GADS hints 的净价值上界 |
| (tok cov8) | 同互抄 | ≥8 token 连续段的覆盖面积 | 仅初版 10 题用过,已被 harvest 取代 |

**数据(题级,n=每数据集题数)**:

| 数据集 | n | 互抄 harvest | prompt harvest | corr(互抄,prompt) |
|---|--:|--:|--:|--:|
| online100(v1 单轮) | 100 | 中位 11.4% | 14.2% | +0.30 |
| **online100_v2(v2 多轮)** | **100** | **中位 18.8%**(P25 15.4/P75 22.2/max 47.0) | 14.3% | **+0.40**(ρ+0.34;限定 q1-q10 也 +0.30) |
| l0v2×2 / tax_A(v2 多轮,同 10 题,local 缓存) | 3×10 | 中位 26~29% | 17~20% | −0.32~−0.64 |

**union 边际增益(online100_v2 全量)**:prompt-only 14.3% → **prompt∪兄弟 27.7%**,净增益中位 **12.9%**(P25 11.2/P75 16.1)。

**五条终审结论**:
1. **"此消彼长"撤回**——两个 100 题大样本都是正相关(可抄性公因子:好抄的题什么都好抄);负相关(−0.32~−0.64)是"同一 10 题 × local 搜索缓存"的条件特例,三次重跑=同条件复测而非独立复制;
2. **union 边际增益是 GADS 的关键数字**:兄弟源把可收割中位从 14.3% 抬到 27.7%,**几乎翻倍**;且 18.8% 的互抄里约 12.9pp 是 prompt 之外的净增量(重合仅 ~6pp)——兄弟源不是 prompt 源的影子,是真增量;
3. **多轮放大 T3 成立,幅度 ×1.65**(11.4%→18.8%);P25=15.4% 说明四分之三的多轮题兄弟源过打平点;
4. **local 缓存模式抬高互抄**——推导与核查:三个 local run(l0v2×2、tax_A)中位 26~29% vs online100_v2 限定同 10 题的 20.8%;**逐题配对核查(tax_A vs online100_v2 同题)**:9/10 题 local 更高、平均 +6.7pp,且每题 summary 条数两边几乎一致(27/24、9/9、27/27…,排除"库大小"混杂),题目文本指纹核对一致(确系同题同管线)。机制解释(待验):local 模式所有 researcher 从同一份冻结缓存取材 → 来源趋同 → 输出趋同;可用"题内兄弟 prompt 的 URL Jaccard,local vs online"直接验证。**GADS 评测须用 online 口径或标注此偏差**;
5. **输入 URL 无预测力**(prompt 的 URL 两两 Jaccard 仅 0.05~0.16,与互抄 r=0.13)——互抄源于"相似响应"而非"读了同样的网页";事前预测用题型/subquery 相似度或探针。

**⚠️ 口径 TODO**:①互抄含"同 researcher 跨轮自抄",按 researcher id 拆 T3/跨轮两块待测;②harvest 是"草稿源在手"的理想上界,现状全局树只吃到部分(v2 接受率 26.4%→42.5% 是实吃锚点),上界−实吃=GADS 空间。

**taxonomy 的操作意义**:每类的草稿源可得时机不同 → 决定系统该在什么时刻、把什么内容、送到哪个请求的草稿库里。这就是设计空间。

### 1.2 核心观察:编排器知道任务图,投机系统却在盲猜

deep-research 是一张 **DAG(有向无环图,大白话:一张"谁的输出喂给谁"的流程图)**:planner → researchers(并行)→ report。**照搬关系与图的边高度对齐**——report 抄的就是图上指向它的那几条边的产出。而现状:arctic 全局树是一个**无差别 FIFO 缓存**(FIFO=先进先出:谁先来存谁,存满一万条就把最老的扔掉——它不认识"哪条和当前请求有关"),命中靠撞运气,还有跨题污染(Q16 的"看着像、第一位就错")。

**编排器明明知道"report 的草稿源就是这 3 份 summary",却没有任何渠道告诉投机系统。**这个信息浪费就是 D1 的立足点。

### 1.3 系统设计:图谱感知的草稿源(Graph-Aware Draft Sourcing)

**先定义两个词**:
- **草稿源(draft source)**:投机解码猜词时"去哪里抄"的那份文本。suffix 现在的草稿源是 prompt(本地树)+ 历史请求输出(全局树);
- **hints(草稿提示,接口参数名 `draft_hints`)**:编排器随请求**附带**的一组文本——等于告诉 server:"这个请求的输出大概率会照搬这几份内容,请把它们当作这个请求**专属**的草稿源"。内容通常就是任务图上游的产出(比如给 report 请求附上它要汇总的 3 份 summary 原文)。

**接口**:编排器发起下游调用时,显式注册草稿源——

```
llm.generate(prompt, draft_hints=[summary_1, summary_2, summary_3])
```

server 端收到后,把这几份 hint 文本 tokenize、插进**该请求专属**的查找结构(树或 GPU 缓冲),投机时从里面抄。对比全局树的四个优势:

1. **定向**:无跨题污染(hint 就是这道题的上游产出);
2. **即时**:不用等 FIFO 恰好还没驱逐;T3 甚至可以在兄弟生成中途流式推送;
3. **可控**:编排器按图谱决定谁喂谁(KVCOMM 的跨 agent 思路在 decode 侧的镜像);
4. **可携带**:hints 随请求走——**双池路由/LMCache 迁移场景下投机收益不丢**(全局树是 server 本地状态,迁移即清零;这给了 hints 一个全局树永远没有的部署自由度)。

**实现分三层递进**(工程量递增,每层都可单独发数据):

- **L-a(1~2 天可出原型)**:复用 arctic 本地树机制——start_request 时把 hints 的 token 也插进该请求的本地树(树支持多序列,现成能力)。CPU 版,同步口径先验证命中假设;
- **L-b**:hints 放进 **GPU 环形缓冲**(ring buffer:显存里一块定长的 token 数组,写满了从头覆盖旧内容——存"最近的草稿源"刚好),配"无树化 GPU suffix 匹配"(方案 2:滑窗匹配+频率投票+变长延伸的张量实现)。**(2026-07-05 收益重估)**旧版说这层"把异步半边整个赢回来"——税拆分定案后要降调:异步 D/E 直测只值 0.5ms/步,suffix 自家 CPU 机件(propose+RPC)也才 ~0.3ms;L-b 的真实价值不在省这几毫秒,而在**结构资格**:草稿链全 GPU 张量化+定长 → 进异步/graph 白名单 → 与 vLLM 主线执行路径兼容(不再被强制走特殊路径),以及 hints 容量不受 CPU 树内存限制。做不做取决于 L-a 验证的命中率,而不是性能账;
- **L-c(屋顶)**:与 KV 层合并成**"两层复用"**——同一条图上的边,prefill 侧复用 KV(KVCOMM/blend:上游输出作为下游 prompt 不重算),decode 侧复用 token(上游输出作为下游输出的草稿源)。统一抽象:**agent 通信内容缓存**(LMCache 的自然扩展:一份内容,存 KV 也存 token 序列,一份冗余两次变现)。

### 1.4 可验证的主张(实验设计)

- **H1 定向 ≥ 撞运气**:同负载下,hints 专属树的接受量 ≥ 全局树(对照:arctic 默认);插桩已能分 local/global 记账,扩一个 per-source 维度即可分 taxonomy 统计"每个接受 token 抄的是谁";
- **H2 可携带性**:路由/迁移场景(探针+LMCache 搬 KV)下,hints 方案投机收益不掉,全局树方案掉到冷树水平;
- **H3 端到端**:对着新冠军 **Ceager** 比(它已把 e2e summary 打到平手/1.04x、report 2.29x)——GADS 的判定是把 summary 段推成**净正**(union 边际 12.9pp 的变现)且 report 接受率高于全局树版;对照组必须含 Ceager+全局树(现状最优)而不是旧 baseline,否则收益会被高估。

### 1.5 文献定位与风险

prefill 侧的跨模型/跨 agent 复用已有(DroidSpeak、我们的 KVCOMM);**decode 侧的跨 agent 草稿复用,截至知识截止未见成体系工作**——这是要用文献核查钉死的主张(见 §五行动清单)。若核查发现近似工作,D1 收缩为"图谱定向 + 两层合并 + 可携带性"的系统增量,故事仍然成立但调门降一档。

> **2026-07-07 文献核查(SoT / 并行生成轴,已查):** SoT 的后代(APAR、PASTA ICML'25、**Plato COLM'25**、Multiverse NeurIPS'25、DoT WWW'25)全是 **orchestration 级"大纲→并行段"= 角色②**,不碰 token 级 decode 草稿路由;我们的 **GADS(decode 侧图谱感知草稿源 = 角色③)不在其射程内——空地仍在**。
> 另核实三家的 **KV 复用全是"位置相关"**、**无一做位置无关复用**:① **Plato** 单调前缀累加(P→P+A∗→…),§4.2.2 明言"若只取依赖子集,B∗/C∗ 落到不同位置 → 生成全新 KV"(`/home/yilin/tmp/plato.txt` L353-356);② **PASTA** 预测块长提前分配 position id + 注意力掩码 + 单连续 KV 池,"generated once, not reused at different positions";③ **Multiverse** 训练式 Multiverse Attention 让分支在**相同位置**跑、Reduce 收敛到 max position、radix **原位拼接** KV 索引(`/home/yilin/tmp/multiverse.txt` L438-464),且**要 SFT**。→ 我们的**位置无关跨 agent 复用(CacheBlend/KVCOMM)vs 三者均可区分,且免训练**(PASTA/Multiverse 要 SFT)。
> **⚠️ 但这一轴不是判 GADS 新颖性的关键轴**——真正要查的是 §五#5 的 **spec-decode-drafting 轴**(REST / SAM-decoding / PLD-lookahead / GraphSpec / cascade drafting):**decode 侧"定向草稿源"若撞车,会撞在那条轴上,而 SoT 综述没覆盖它、仍未查**。GADS 新颖性未最终判定,以该轴核查为准。

## 二、D2:copy-state 感知的混合仲裁

### 2.0 为什么做 / 解决什么 / 预期收益(总起)

**为什么做**:suffix 和 EAGLE 的强弱段**完美互补**(下表),而且互补的边界有一个**免费且机制可解释的信号**(match 状态:锁定/雪球理论直接预测"match 长 = suffix 将赢")。别家做 adaptive drafting 调的是"猜几个",我们仲裁的是"**让谁猜**"——检索臂和神经臂在步级换手。

**解决什么问题**:单一 drafter 结构性留钱——suffix 在创作段每步 0(summary 81% 的步陪跑,P0 只能把这些步的成本归零,**赚不到它们的加速**);EAGLE 在照抄段只有 3~5、吃不下 20+ 的大口。仲裁是唯一能"两头都赚"的路。

**预期收益**:上界可以**零成本先算**(行动清单 #3:模拟器把每步取 max(suffix 实际, EAGLE μ≈4) 的混合 oracle 跑出来,半天、零 GPU)。粗账:summary 的零接受步若换成 EAGLE 的 μ≈4,每步接受从 1.4 → 2.5+,加上 Ceager 已把税砍掉,summary 有从平手到 1.3x+ 的想象空间——**但先让模拟器说话,再决定投入**。风险同样明确:EAGLE 臂要训练/装载 draft head,工程量数量级大于 P0。

**与 P0 的关系**:P0 是 D2 的最小切片(单臂 on/off),信号族相同(接受 EMA/match 状态)。P0 先落地,把信号可靠性、阈值方法论、门控日志基建全部调通——D2 到时只是把"关闭"换成"换臂"。

### 2.1 事实基础:两类 drafter 互不支配

| 生成状态 | suffix(检索) | EAGLE(学习) |
|---|---|---|
| 抄写段 | **一口 20+**(i61 有 93 个大步、13 次顶格 24) | 3~5 |
| 创作段 | **0**(summary 81% 零接受步) | 3~5 |

任何单一 drafter 都在留钱:EAGLE-only 浪费抄写段的大口,suffix-only 浪费创作段。→ "每步该用谁"是一个真实且未被好好解决的仲裁问题。

### 2.2 仲裁设计

- **信号**(从便宜到贵):①match 状态——proposer 反正要算,**零成本**,且我们的机制理论(锁定/雪球)直接预测"match 长=suffix 将赢";②近期接受 EMA(指数滑动平均:最近几步接受数的加权平均,越近权重越大——"这请求最近抄得顺不顺"的一个数);③熵/top-1 概率(采样端顺手);
- **策略**(从简单到复杂):规则版(match≥m → suffix,else → EAGLE)→ EMA 自适应 → bandit(多臂老虎机:边试边学"哪个 drafter 在这类请求上赢面大"的在线算法)/学习型;
- **先零成本验上界**:模拟器可以直接算"混合 oracle"(每步取 max(suffix 实际接受, EAGLE 假设 μ≈4))——**半天、零 GPU**,先知道这个方向的天花板再决定投入(行动清单 #3)。

### 2.3 差异化与风险

cascade drafting、adaptive speculation length(SpecDec++ 等)已有文献;我们的差异化候选:**检索×神经的步级互补仲裁**(不是级联验证)、仲裁信号来自检索侧 match 状态(零成本+机制可解释)、与 D1 的 agent 负载协同。风险:若文献已有近似,D2 降级为 D1 系统内的一个组件(仍有价值,不再是独立贡献)。工程注意:vLLM 单 server 单 method,原型要么改 proposer 接口共存双 drafter,要么先用模拟器+离线数据做算法研究。

## 三、D3:微改写鲁棒的检索草稿(定位:D1 内的增量优化)

**为什么做**:现象已钉死——模型抄写时爱做微小改写("一个'而'字打掉 6 token 草稿+2 步重锁"),每次微改写的真实代价不是那 1 个 token,是**斩断后的重锁成本(~3 步爬回大口)**。**解决什么**:把"抄写段被标点/连接词打断"的碎片化损失补回来。**预期收益(上界先泼冷水)**:树 oracle 实测只 +11%,D3 全部空间被它罩死——所以它是 D1 系统里的一个优化点,**不值得单独立项**,顺手做、别恋战。两个思路:

- **编辑容忍的重锁**:被拒后不退回 m=0,尝试"跳过 1 个 token 再对齐"(编辑距离 1 的容错锁定),把每次斩断的重锁成本从 ~3 步压向 ~1 步;
- **分叉预测**:统计模型爱改哪类位置(连接词/助词/标点),草稿在这些位置留宽度 2 的小分叉——依赖**树验证**(tree attention:一次前向同时验证多条分支草稿的注意力掩码技术,vLLM 尚未给检索式方法接线),排队等。

**上界提醒**:树 oracle 只有 +11%,D3 的全部空间被它罩住——**不是独立论文,是系统里的一个优化点**。红线:验证端不动(无损性保住,rejection sampling 原样)。

| oracle 上界 | 给某个决策环节开"事后诸葛亮"算出的理论天花板 |
