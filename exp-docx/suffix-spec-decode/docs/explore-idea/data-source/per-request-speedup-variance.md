# suffix 加速的题级差异报告:收益由"照抄量"决定,不由任务类型决定

> 承接主实验([summary.md](../../summary.md))和 replay 重叠分析([overlap-replay.md](../../overlap-replay.md))。主实验给的是**阶段平均**(summary ~0.95x、report ~1.76x);本报告钻进**单请求/题的粒度**,回答:同一个"summary"阶段,不同题的加速差多少?差异由什么决定?能不能预测和利用?
> 数据:60 条 summary + 20 条 report(每条来自不同题,从 DRBench 100 题均匀抽样),replay temp0 实测,长度归一 char/s 加速比。逐条明细见 [per_call_overlap.csv](../data/per_call_overlap.csv)。

---

## TL;DR
1. **"平均略亏"是假象**。同是 summary,加速比从 **0.71x 到 2.56x**;60 条里 **15 条真加速、33 条明显拖慢**,平均下来才显示 0.95x。
2. **决定加速的是 token 级照抄量,不是任务标签,更不是字符级 coverage**。token 级照抄占比与加速比 r=0.55;而 v2 头条的**字符级 Coverage@8,r=−0.23(负相关!)**,rp_cov30 r=0.009(几乎零)——**它对 summary 加速没有任何预测力**。
3. **两类题泾渭分明**:检索/罗列型(逐条转录来源、带 URL 原文)→ 照抄多 → 加速(可到 1.5x);综述/分析型(重组多源、自己组织)→ 照抄碎 → 拖慢(低到 0.71x)。
4. **有可利用的空间**:若能只对"真加速"的题开 suffix,summary 段吞吐可从 0.93x 翻到 **1.05x**(约 12pp 摆动)。但目前的预测信号是"事后"的(要先有输出),**做成上线路由需要一个"生成前"的轻量预测器 —— 这是值得投的方向**。

## 一、现象:同是 summary,加速比横跨 0.71x–2.56x

| 阶段 | min | P10 | P25 | 中位 | P75 | P90 | max | 分布 |
|---|--:|--:|--:|--:|--:|--:|--:|---|
| **SUMMARY** (60) | 0.71 | 0.78 | 0.82 | 0.88 | 1.03 | 1.14 | 2.56 | >1.1x:8 / 1.0–1.1x:7 / 0.9–1.0x:12 / **<0.9x:33** |
| **REPORT** (20) | 1.27 | 1.42 | 1.64 | 1.89 | 2.23 | 2.41 | 2.46 | **全部 >1.1x** |

**关键**:summary 不是"整体均匀略亏",而是**两拨题命运相反**——四分之一的题越过 1.0x(其中 8 条 >1.1x),一半以上的题掉到 0.9x 以下。用一个"平均 0.95x"去描述它,会同时误导两个方向:让检索型题的收益被埋没,也让人以为分析型题只亏一点点(其实亏到 0.71x)。report 则稳定得多:再差也有 1.27x,**永远赚**。

## 二、什么在驱动差异:token 级照抄量,字符级 coverage 会骗你

各特征与加速比的相关系数(Pearson r):

| 特征 | SUMMARY r | REPORT r | 说明 |
|---|--:|--:|---|
| **token 级 harvest 占比** | **+0.55** | **+0.90** | 锁定后能白捡的 token 占比,最贴机制 |
| **token 级 Coverage@8** | **+0.56** | +0.89 | token 级连续照抄 |
| token 级 Coverage@16 | +0.49 | +0.90 | 更长的 token 段 |
| token 级最长段 | +0.51 | +0.41 | |
| 字符级 Coverage@30 | +0.009 | +0.84 | summary 几乎没 ≥30 字段 → 全是噪声 |
| **字符级 Coverage@8(v2 头条)** | **−0.23** | +0.005 | **对 summary 反向、对 report 无关** |
| 输出长度(token 数) | +0.09 | +0.44 | summary 里长度几乎不影响 |

(2026-07-02 长度归一口径复核:方向全部复现且更强——合并 80 条 harvest 对加速 Pearson 0.90 / Spearman 0.90;summary 组内 Spearman 0.79 > Pearson 0.55,排序关联更稳、Pearson 被 i23 类离群拉低;字符级 cov8 依旧无用。数表见 [questions-log.md](../../questions-log.md) Q17。)

**这张表是整个实验最反直觉、也最重要的一张**:
- 让我们最初以为"summary 照抄很多、该加速"的那个指标(**字符级 Coverage@8 = 54%**),对 summary 的实际加速**不但没有正预测力,还是 −0.23 的负相关**。原因见 [questions-log.md](../../questions-log.md) Q9:字符级 cov8 把 8~30 字的**碎片**也计入,而其中大量片段短于锁定费(~3 token;8~30 字符在英文里只有 2~6 个 token),价值趋零——注意**不是"碎片一律无价值"**:token 口径的 cov@8/cov@4 与加速都显著正相关(+0.56/+0.51,见上表),门槛错位的是字符度量本身。cov8 高往往意味着"英文术语碎片多"而非"长段多",所以反而和加速轻微反向(另一半原因是语言混淆,见 Q19)。
- **只有 token 级指标预测得准**(r≈0.5–0.55)。suffix 在 token 上做投机,就得用 token 级的连续照抄来衡量。
- report 侧几乎所有 token 级 + 长字符段指标都强相关(r 0.84–0.90),因为它照抄的是**整段**,各口径都能捕捉;唯独字符级 cov8 依然无关(r=0.005)——**cov8 这个指标在两个阶段都没用**。

## 三、快 vs 慢:两类题长什么样(具象)

| | 慢(拖后腿) | 快(受益) |
|---|---|---|
| 例子 | i48 历史意识形态化 **0.71x** / i54 ADAS **0.74x** | i10 教学竞赛资源 **1.52x** / i17 二维半导体接触 **1.44x** |
| harvest | 5% | 44–52% |
| 题型 | **综述/分析型** | **罗列/检索型** |
| 输出形态 | 把多源**重组**成自己的要点结构、加粗小标题、英文分析散文 | 几乎**逐条转录**搜索结果,常含 **URL 原文**、来源名 |
| 压缩比 | 高(42K 字 prompt → 4K 字输出) | 低(8K → 0.6K) |

慢的例子(i48)开头:`**Key Information on the Ideologisation of History...** 1. **Authoritarian States...** - China and Russia use historical narratives to...` —— 全是模型自己组织的分析框架,token 级几乎不照抄原文。

快的例子(i10)开头:`1. 中国教育干部网络学院提供...一等奖课程的教学设计...网址：https://s.enaea.edu.cn/h/gxqnjsnlts 2. Bilibili平台上有第七届全国高校青年教师教学竞赛的多个组别决赛视频...` —— 逐条搬来源,**URL 是天然的长精确 token 匹配**,suffix 一命中就是一长串。

**判别直觉**:问的是**事实/资源/名录/数据**(答案 = 把来源摆出来)→ 照抄多 → 快;问的是**分析/评价/综述**(答案 = 归纳出观点)→ 转述多 → 慢。这跟报告为什么恒快一致——报告阶段的活儿本质就是"把各 researcher 的 summary 组织进最终稿",天然是"照抄+拼接"。

## 四、报告侧:方差更小但恒正

report 20 条加速比 1.27x–2.46x,**无一低于 1.27x**,预测特征更强(token cov16 r=0.90)。因为报告的照抄是结构性的(必然大段复述它 prompt 里的 summary),不像 summary 那样依赖"这道题恰好是不是检索型"。所以 **report 阶段可以放心全开**;真正需要斟酌的只有 summary 阶段。

## 五、对部署的意义

**① 不能按"阶段标签"一刀切开/关 summary 的 suffix。** 同一阶段内题与题差到 0.71x vs 2.56x,标签粒度太粗。

**② 完美路由的收益上界(summary 段,解码时间加权聚合)**:

| 策略 | summary 段吞吐 |
|---|--:|
| 从不开 suffix | x1.000 |
| 全开 suffix | x0.932 |
| 按 harvest>19%(打平点)开 | x1.031 |
| **完美路由(只在真>1x 开)** | **x1.053** ← 上界 |

→ 光把 summary 阶段的 suffix"该开才开",就能从 −7% 翻成 +5%,约 **12pp** 的空间。

**③ 但门槛信号是"事后"的。** harvest / token-cov 都要**先有输出**才能算,没法直接拿来在生成前决定开不开。所以要落地路由,有三条路:
- **最省事**:全局开 suffix。report 阶段(decode 大头)+ 检索型 summary 一起吃到红利,分析型 summary 各亏 ~7%;**只要 report 的解码占比够大,整体净赚**——这也是主实验的建议。
- **按阶段分池**:report 走开 suffix 的 server 池、summary 走不开的池(vLLM 的 spec 配置是 server 级、不能逐请求切)。稳妥但要两套服务。
- **值得投的方向**:训一个**生成前的轻量预测器**,从 prompt 特征(问题类型:事实/检索 vs 分析/综述;是否含大量待转录的结构化来源/URL)预测"这条会不会照抄",再决定路由。本报告已证明**目标信号(照抄量)与加速强相关且题间可分**,预测器有明确的学习目标。

## 六、方法与数据
- 加速比 = 同 index 的 suffix char/s ÷ baseline char/s(长度归一,消除两次 temp0 输出长度不同的干扰)。
- ⚠️ 口径注意(2026-07-02 勘误):[per_call_overlap.csv](../data/per_call_overlap.csv) 的 `speedup` **列**存的是**未归一的 decode 时间直接比**(analyze_replay_overlap.py:100),和本文正文的归一数值不是一个口径;桶级结论两口径几乎重合(harvest 四分位 0.82/0.97/1.08/1.90 vs 0.81/0.96/1.06/1.94),但单条差异可以巨大(极端如 i23:直接比 0.34x vs 归一 2.56x——两次运行输出 1218 vs 9242 字符),**引用单条必须自行归一**。详见 [why-high-overlap-no-speedup.md](../../why-high-overlap-no-speedup.md) §六勘误条。
- 特征:字符级用 SAM 求 matchlen(复用 v2 脚本);token 级用 Qwen tokenizer 分词后同一 SAM 逻辑;harvest = 锁定(≥3 token 已匹配)后可延续的 token 占比。脚本 `analyze_replay_overlap.py`,逐条 [per_call_overlap.csv](../data/per_call_overlap.csv)。
- 全部冷树、batch=1、单进程(避开三大测量坑,见 [summary.md](../../summary.md))。

## 附:局限(诚实小字)
- **样本**:summary 60 / report 20,每条 ~1 题;结论看分布和趋势,单条有噪声(如 i23 在 18% harvest 却 2.56x,是短输出撞上一个长匹配的放大效应)。要写正式材料建议扩到每阶段数百条。
- **可能的混淆**:本批"快"样本偏短、且多为中文罗列;但输出长度与加速几乎无关(summary r=0.09),token 级照抄量才是贯穿始终的解释变量,语言只是表象。
- **相关≠全部**:token 特征 r≈0.55 解释了 summary 约三成方差,余下受输出长度构成、单次匹配位置等影响 —— 这也是"预测器"仍需真正建模、而非单阈值卡的原因。
- **上界≠可达**:1.05x 是"完美先知路由"的天花板,真实预测器会打折;但它标定了这个方向值得追的量级。
