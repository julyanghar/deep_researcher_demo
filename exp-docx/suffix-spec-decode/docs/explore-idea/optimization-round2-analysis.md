## 〇、总纲:根因排序决定优化排序

## 一、上界阶梯:oracle 是什么意思、每层损失各是多少

**先澄清一个常见误读:oracle ≠ "让草稿全命中"。** 草稿命不命中由"模型实际要说什么"决定——模型说自己的话时,任何算法选任何草稿都不会命中,这是物理约束不是决策失误。**oracle 的正确用法是:给某一个决策环节装上"事后诸葛亮",把该环节做错决定造成的损失单独称出来。** 不同环节装天眼,得到不同层的上界:

```
实测(replay,长度归一)                summary 聚合 ≈0.955x
   ↑ 免掉 18% 固定税 + propose CPU + 两次运行输出差异
理想机制上界(模拟器,score 单候选)      1.33x            ← sim_all.json
   ↑ 候选选择开天眼:每步全部 match_len 候选都验证、取命中最好的
选择-oracle 上界                      仅 +1~2pp 接受量  ← 本轮 sim_multi_candidate
   ↑ 分叉处所有支路全验证(树状草稿 oracle,无限宽)
树-oracle 上界                        summary 1.33→1.50x(步数多省 11.3%);i61 3.32→3.67x
                                      ← sim_tree_oracle.py(并行 session 补测)
─────────────────────────────────────────────────────
硬天花板:只有"模型实际要说的 token 恰好连续出现在树里"的位置才可能被接受
        = harvest 决定的形状上限,任何选择/验证算法都翻不过去
```

**选择-oracle 实验**(方法:模拟器变体,轨迹按生产策略推进,每步反事实地把所有 match_len 候选各自与真实输出求 LCP、取最大,对比 score 选中者):

| 请求 | policy 接受(score 单候选) | oracle 接受(全候选并行验证) | score 被打败的步 |
|---|--:|--:|--:|
| i53(慢例) | 95(20%) | 103(21%) | 8/388(2%) |
| i37(cov8=94%) | 83(32%) | 87(34%) | 4/174(2%) |
| i10(快 summary) | 154(49%) | 154(49%) | **0**(0%) |
| i17(快 summary) | 154(54%) | 158(55%) | 4/132(3%) |

读法:**score 单候选已拿到 97~100% 的可拿收益**。零接受步的本质是"所有候选都错"(根因①),不是"选错了候选"(④)。

## 二、六问逐答(创新点就地合并展开)

> 统一比喻(和 18 号文档一致):suffix 投机 = 赌桌下注。**score = 这份草稿"预计能对几个 token"**(数学上就是期望值);赢一个 token = 省一步 decode;验证 = 开牌。各创新点(3.x 小节,编号沿用、供外部引用)按同一模板:现状哪里在漏钱 → 改什么 → 钱从哪里回来;审后状态(❌撤回 / ⏸搁置 / ✅保留)见各节头部批注。

### 问 1:相同 score 取第一个 / 只验 best candidate,是否浪费?能否并行投机多候选?

- 事实澄清:平手规则是 `>=`——**更长的 match_len 赢**(不是取第一个);本地树与全局树平手取本地。候选草稿的"计算"在 CPU 上近乎免费,浪费只可能出在"没验证"。
- 定量分两层:**窄版**(只在现有 match_len 候选间开天眼)+1~2pp(上表);**宽版**(分叉处无限宽全验证,sim_tree_oracle)summary 理想 1.33→1.50x、步数多省 11.3%——叠上真实树宽 2~3、18% 税、并发下草稿 token 抢算力三笔折扣后,**配不上引擎级改造**。
- 工程事实:arctic_inference C++ 里已有树状草稿(`_speculate_tree`/`use_tree_spec`)但 vLLM 0.18 没接线;真正门槛在**验证端**——vLLM v1 rejection sampler 只支持线性链,树验证需要 tree attention。**结论:低优先级,等上游。**
- (本问对应的创新点 = 树状投机,定量与判定已由 §一 的两层 oracle 给出,搁置,无单独 3.x 小节。)

### 问 2:能否让投机兼容 async scheduling?

- 难点:异步的前提是"不看本步结果就能排好下一步";suffix 下一步 batch 依赖本步**接受数**(1~25,验证完才知道)和**草稿**(采样后 CPU 才产出)。EAGLE/MTP 兼容是因为草稿在 GPU 前向内部产生、padded 形状可预留。
- 路线 A(不建议自己扛):padded 预留 + 草稿晚绑定,要改 vLLM 调度器核心。
- **路线 B(推荐):不救异步,把 CPU 段砍小**——展开见 3.5;两半税各占多少从未实测,裁决实验见 3.9。
- 缓解事实:税在真实并发下**自动稀释**(GPU 步变长、CPU 占比下降)——v2 端到端墙钟 −29%、summary x1.00。
- **追问:EAGLE 为什么能原生兼容异步?它不是同样"不知道命中几个"吗?** 是的,"接受数不确定"两家都有,但它可用 **padding** 解决(调度器每步按上限 k 预留槽位,实际接受数留在 GPU 上事后对账,浪费的位置就是 padding 的代价——vLLM 的 padded drafter batch,这正是校验代码要求 EAGLE+异步必须开它的原因)。**真正的分水岭是草稿的"值"在哪产生**:①EAGLE 的草稿头跑在 **GPU 前向尾部**,草稿值以 GPU 张量存在,CPU 调度器排 N+1 时**不需要知道值**——和普通异步 decode 的"最后一个 token 占位符晚绑定"同一招,值由 runner 在 GPU 上填;②suffix 的草稿要 **CPU 拿到本步采样结果之后**才能算(喂树+匹配),而异步时间线上 CPU 排 N+1 时 GPU 还没跑完 N、token 还没回传——**suffix 的 propose 在异步时间线上没有执行窗口**;③次要障碍:suffix 草稿 0~24 变长是它的经济学核心(按证据下注),按 24 padding 会每步塞满垃圾验证位,padding 不经济;EAGLE 固定小 k(3~5),padding 浪费有界。stale-by-one RFC 的本质就是给 suffix 在异步时间线上硬造一个窗口(用 N−1 的状态出稿),代价是接受率打折。
- **追问:能否"随开随关"(decode 中动态切换异步)?** 分三层答:①**现状不可行**——async_scheduling 是启动时定死的执行纪律,不是每步读的开关:开着异步时任何时刻都有一个"在途 step"没收尾,切换需要**排空流水线(drain)+ 换引擎主循环的纪律**,这是引擎手术不是配置活,vLLM 无此 API;②**即便做出来,粒度是整台引擎不是单请求**——一个 engine step 只能有一种纪律,混合 batch(有人在抄、有人在写)下批级切换会抖动(碎片式照搬正是高频抖动场景),要靠迟滞阈值压制,收益被切换气泡(每次 drain ≈1 步)吃掉;③**同样的目的有三条更省力的路**:题级双池路由(不投机的池天然开异步,§3.8 探针+KV 迁移就是它的加强版)、**EAGLE(草稿产于 GPU 前向内、原生兼容异步——"不投机时零税"和"投机"兼得,这正是它排优先级第 2 的原因)**、给 vLLM 上游提 padded/stale-draft CPU-drafter RFC(一步旧状态出草稿换取异步保留,接受率会掉,研究性质)。**且这个决策应等 §3.9 实验数据**:若 e2e 并发口径下异步损失已被稀释到几个点,"随开随关"的动机本身就弱了。

#### 3.5 降税三件套:夹书签、隔街问价、下沉 C++

> **⏸ 审后修订(Q24)**:用户指出的前提问题成立——三件套只砍得到税里 **propose CPU** 那一半,砍不到"关异步调度"那一半;而**两半各占多少是假设、没测过**。先做 §3.9 的税成分拆分实验,若异步占大头,这三件套的收益上限就小,优先级相应下调(且 EAGLE 的权重上升,它能把异步半边拿回来)。

**现状哪里在漏钱**:18% 固定税里有一部分是每步 propose 的 CPU:**每一步**都拿末尾 24 个 token 从头枚举后缀去配树——可相邻两步之间序列只多了一两个 token,上一步配到哪、这一步完全可以接着配(维护一个匹配状态,和 overlap 分析里 SAM 的 (v,l) 状态机同理)。另外,一条请求连续 50 步一个匹配都没有(纯转述段),还每步老老实实查树,纯白花钱。

**做法**:①增量匹配(查字典夹书签,别每次从第一页翻);②连续 K 步无匹配就隔几步再查(一条街 50 家都没货,隔几家问一次);③Python 逐请求循环下沉 C++ 批量。

**先钉一个容易误读的点(现状 vs 提议)**:现状**没有**"接着配"这回事——现有实现每一步都拿末尾 ≤24 token **从树根重新枚举** m=1..24,没有任何跨步状态("雪球变大"只是**结果**:接受的 token 成了序列结尾,下一步枚举自然配得更长;路还是从根重新走的,不是断点续传——[spec-optimization-six-questions.md](../spec-optimization-six-questions.md) §四.3 同一结论)。出草稿时确实走到过树的深处节点,但**函数一返回这个位置就被扔掉了**。下文"零成本"说的是**改造后**获取新状态的成本:提议要做的,就是花一个指针把这个位置存进请求状态("书签"),外加岔开时的回退。

**增量匹配具体怎么"接着配"**(拿"匹配 4 个、命中 2 个"的例子走一遍):设本步匹配 `A B C D`(m=4,对应树节点 N),沿 N 往下出草稿 `E F G H`;验证时模型实际说 `E F X…` → 接受 E F、第 3 位不等、纠正 token X 顶上,本步净吐 `E F X`。下一步的匹配状态分三种 token 拿:
- **被接受的 E F(零成本)**:草稿本来就是沿树走出来的路径,接受=证实序列沿这条路又走了 2 步——新状态就是"N 往下走 E、F 的那个节点 N′",**出草稿时已经走过,不用再查**;
- **纠正 token X(一次查找)**:它是模型自己的词,拿它从 N′ 往下试一步:走得通 → 匹配=7,O(1) 完事;
- **走不通才回退(岔开步)**:此时"最长匹配后缀"要从前端缩短(`B C D E F X`?`C D E F X`?…直到配上)——SAM 用 suffix link O(1) 跳;arctic 的树没有这种链,真实现要么给树补 failure 链(Aho-Corasick 式),要么只在岔开步退回全量枚举。
账的关键:**顺风步(正在抄)增量几乎免费,而那恰好是现在最贵的步**(匹配深、枚举长);岔开步要付回退,但岔开步匹配本来就浅、全量枚举也便宜。另两个精确点:①现在的行为是"枚举所有 m 按 score 择优"——能配上的所有 m 恰好等于最深节点沿 failure 链跳出来的集合,候选集能增量拿到,**行为可做到完全一致**;②匹配长封顶 24(树深),用滑动窗口处理,树的插入端已有同款技巧(`_active_nodes` 滑动队列,见 19 号教程)。

**为什么能赚**:直接削每步 CPU 段,对所有请求、所有步生效、与命中无关——这是对"税"本身动刀。(注意:砍不到"关异步调度"那半边税,那半边见 3.7 的附赠。)

#### 3.9(新增)税成分拆分实验:18% 里异步和 propose 各占多少

**动机**:全部降税类方案的价值都取决于一个没测过的数——18% 固定税里,"关异步调度"(三件套砍不到)和 "propose CPU"(三件套能砍)各占多少。

**做法**(半天):三台同配置 server 跑同 80 条 replay 对照——① baseline(异步开);② baseline + **手动关异步**(vLLM 的 `scheduler_config.async_scheduling` 是三态可显式置 False;CLI 若没暴露就一行 patch);③ suffix 默认。①vs② = 异步半边;②vs③ = propose+验证半边。辅助:给 `SuffixDecodingProposer.propose` 加计时日志直接量每步 CPU 毫秒。

**决策规则**:异步占大头 → 三件套降级、EAGLE 升级(唯一能拿回异步的方案);propose 占大头 → 三件套按原计划做。

### 问 3:能否预测"这步/这题适不适合投机"?

- **步级:省不掉税。** suffix 天生带步级门(无匹配→空草稿→退化普通 decode);而 18% 固定税是 server 级的,步级预测只能省草稿边际成本(batch=1 下很小)。步级信号的正确用途是**下注尺寸**(按最近接受历史调 score 门槛/budget),高并发下才值钱。
- **题级:可行但收益有限。** harvest 与加速 r=0.90、题型可分,预测器有明确 label(=L2);但上界 summary 段 12pp、v2 端到端全开时 summary 已 x1.00、且 vLLM spec 是 server 级 → 需双池。若做,**文本特征就够**(见问 4)。
- **追问(2026-07-04):不想开两台 server,怎么办?** vLLM 0.18 自带解:**`method="ngram_gpu"`**(speculative.py:50)——GPU 上的 prompt-lookup 检索投机,**在异步调度白名单里**(vllm.py:699/723)。单 server 即得"投机 + 异步兼得":不照搬的请求只付 GPU 内的廉价匹配(藏在 GPU 时间线里,≈零税),照搬段照样抄 prompt 出草稿,且完全贴"照搬输入"的 motivation。代价 vs suffix:无全局树(丢跨请求互抄红利)、无频率打分、固定草稿长 k、匹配窗口小——**能不能吃到 report 的肉要实测**。已排 **D 配置**(ngram_gpu, lookup 2~6, k=8)接在税拆分实验后自动跑(同 workload replay+e2e);断言升级为"投机与 async enabled 同时成立"。若 D 接近 suffix 的 report 收益 → 单池部署答案就是它;差多少也正好是论文的 ablation(树的频率打分/全局树/变长下注各值几个点)。研究向的正解仍是把 suffix 塞回异步时间线(stale-by-one 草稿流水线,保住树的独有优势)。
- **追问(2026-07-04):能否指定某些请求投机、某些不投机?** 能,且很便宜:草稿是逐请求给的,proposer 里对指定 req_id 返回空草稿即可(该请求退化普通 decode)——在已插桩的 suffix_decoding.py 加 ~10 行(env 正则过滤 req_id;客户端用 X-Request-Id 头按 tag 标记,0.18 的 header 透传需核实,不行就 req_id 前缀 hack)。可选变体"**只捐树不下注**":跳过 speculate 但仍 add_active_response,保住它对全局树的供体价值。**但账要算对**:异步税是 server 级的,被关掉的请求**拿不到 winnings 却照交税的大头**——batch=1 下"关"反而比"开"更亏(投机赢≥0 且验证近免费);真正的收益场景只有两个:①高并发下省掉 batch 前向里的垃圾验证位(Q24 的 λ 逻辑);②**科研消融利器**——同 server 同树同负载下做"该请求开/关"的干净对照,不用双池。要让某些请求"零税",仍然只有双池/探针迁移(见下条)。
- 进阶玩法(自家基建):**前 K token 探针 + LMCache KV 迁移**——普通池先生成 32~64 token,量熵和照搬率,spec 友好再迁 suffix 池,KV 走 LMCache 免重 prefill。
  - **熵是什么、怎么测**(Q23):熵=模型下一个词的"犹豫程度"。逐字照抄时下一个词几乎注定(近乎背书),分布集中 → 熵低、top1 概率≈1;自己组织语言时同义说法多,分布摊开 → 熵高。测量零额外成本:decode 每步本来就算出了全词表分布,H=−Σp·log p(或更便宜的代理:top1 概率),vLLM 开 logprobs 就能拿,32~64 步取均值。照搬率则拿探针段和 prompt 做 token 匹配(harvest 同款算法,毫秒级)。**照搬率是"事实"(已经在抄)、熵是"倾向",两个都近乎免费,照搬率为主、熵为辅。**

#### 3.8 前 K token 探针 + LMCache KV 迁移:先试写两行,再分班

> **✅ 保留**(排 EAGLE 之后)。

**现状哪里在漏钱**:题级路由(该开 suffix 的题才开)最好的信号是 harvest,但它要**先有输出才能算**——鸡生蛋;纯 prompt 特征预测又隔一层。

**做法**:所有请求先在普通池写 32~64 个 token(亚秒级),这段开头已经暴露"要抄还是要写"(照搬率、熵);判定 spec 友好的迁去 suffix 池。换池最大的成本本来是重算几万 token 的 prefill——**用自家 LMCache 把 KV 搬过去**,迁移费压到接近零。

**为什么能赚**:逼近"完美路由"12pp 上界的可实现版;且是自家基建独有的打法。风险:v2 端到端下全开已 x1.00,路由边际收益缩水——所以排在 EAGLE 之后。

### 问 4:模型什么时候抄原文、什么时候自己总结?能否从内部状态判断?

- **行为层(已实证)**:由"答案形态"决定——事实/资源/名录/数据 → 抄;分析/评价/综述 → 总结。生成前可观测代理:指令动词(列出 vs 总结)、URL/bullet 密度(URL=天然长匹配)、**压缩比**(prompt 长÷预期输出长:42K→4K 必重组,8K→0.6K 可摘录)、单源 vs 多源、语言。
- **内部状态层**:抄写的机制签名 = **induction head**——注意力钉在"prompt 中上次匹配位置的下一个 token"、逐步 +1 前移,同时 argmax≈1(低熵);转述时熵抬升、注意力弥散。可用 KVCOMM 的 attention 工具做小实验:逐步记录 top-attention 位置 vs 该 token 是否照搬。
  - **它和熵是两个信号,别混**(Q23):熵看**输出端**(概率分布的形状,犹豫不犹豫);induction head 看**中间层注意力**(模型的"眼睛"盯在哪——抄写时盯着原文位置逐步滑动)。两者常同时发生所以熵能当代理,但可分:模型可能低熵却不在抄(编号、套话、常见搭配),也可能在抄但熵不极低(边抄边换词)。**线上落地只能用熵+照搬率**——生产推理用 FlashAttention,注意力矩阵不物化,取出来要改 kernel;induction head 适合离线研究(解释机制、给预测器加特征)。
- **prefill KV 预测力弱**:prefill 是"读题",抄不抄是"作答"时逐段决定的,同一请求内抄写交替。诚实结论:题级路由用文本特征更便宜;内部状态强在步级——而步级省不了税,故偏研究价值。

### 问 5:prob 兜底 + max score 的算法能否优化?

先摆正:**score = Σₖ P(前 k 个全对) 恰好是"期望接受 token 数"的估计**——max score 本来就是期望收益最大化(草稿零成本假设下),oracle 实验也证实近乎最优。可改的是成本项和来源可信度,全为 proposer 侧小改、可逐项 A/B(冷树 80 条 replay 一晚扫完):

四个候选小改(score 地板 / 全局树 margin / λ / 校准)全部走过 Q24 审稿轮——**两条撤回、两条搁置**,逐个展开:

#### 3.1 score 地板:预计赢不到半个子儿的注,别下

> **❌ 审后基本撤回(Q24,2026-07-02)**:两处站不住。①它省不掉本步的投机开销——score 是 propose 跑完才算出来的,树也查了、CPU 也花了,地板只能省"下一步前向里的草稿位置",batch=1 下近乎免费;②实测得失账(6 条 summary、910 个有草稿步):θ=0.5 会砍掉 295 步,**少验证 314 个草稿 token,但丢掉 45 个真接受**——每个验证 token 的成本要 >0.14 步才回本,低并发下净亏。仅在重负载+配合校准后才值得重估。

**现状哪里在漏钱**:现在只要有匹配就出草稿,哪怕 score=0.2——意思是"这份草稿预计只能对 0.2 个 token"。这种注十有八九全错(i53 八成草稿是白验证),而每个草稿 token 都要在下一步前向里占位置、过 logits、过比对——赢面配不上占用。Q16 还抓过它的恶性形态:低置信长尾巴给 score"充值",把本来更稳的短草稿挤下台。

**做法**:一行判断——最终选出的草稿 score < 阈值(比如 0.5,"预计连半个 token 都赢不到")就不交,该步退化成普通 decode。

**为什么能赚**:砍掉的全是**期望为负**的注,省验证端的位置和 CPU(并发下这些不免费),顺带修挤位 bug;不碰任何"预计能赢"的注,几乎没有下行风险。预期小几个点,一晚 A/B 可证。
**别和 oracle 结论搞混**(Q23):oracle 实验说的是"**挑哪份**候选"这个环节几乎无损(挑的基本就是命中最长的);地板管的是另一个决策——"**这份(哪怕是全场最好的)草稿值不值得交**"。赛马比方:oracle 证明你挑马眼光没问题(挑的都是场上最快的马);地板说的是"有些场次所有马都是瘸的,这种场次别下注"。它其实就是**最便宜的步级'什么时候投机'判断器**——score 本身就是现成的期望收益估计,加个阈值即可,不用训练任何预测器。

#### 3.2 全局树 margin:道听途说要打折

> **⏸ 审后降级为待验证假设(Q24)**:用户质疑成立——v2 端到端里全局树是**明确净正**(同题 summary 互抄,接受率 26.4%→42.5%;关掉它反而 0.872x)。"global 险胜 local 的那部分是净负"目前只有间接证据(Q16 在 prob=0.05 下的挤位),默认参数+热树场景**未验证**。验证方法:server 里加一行日志(每步记录选中候选来源 local/global、score 差、实际接受),统计"险胜步"的接受率;或直接开 δ 跑 A/B。数据出来前不动手。

**现状哪里在漏钱**:草稿有两个来源——本地树(这道题自己的 prompt,"自家账本")和全局树(历史请求的输出,"道听途说")。现在只有**平手**时让本地赢,全局 score 高一点点就被采纳。可全局树的草稿来自**别的题**:开头看着一样,续写常常牛头不对马嘴,第一个词就错(Q16 实证过它挤位)。

**做法**:全局候选必须比本地高出余量 δ 才能上桌;进阶版做记账——本请求里全局草稿最近老是被拒,就把它的 score 再打折(在线自适应)。

**为什么能赚**:砍掉"看着像、实际错"的白验证;而全局树真正有用的场景(v2 端到端里同题 summary 互抄、接受率 26%→42% 那种)是**大额优势**,δ 打不掉——只过滤险胜,不过滤大胜。

#### 3.3 成本感知下注(λ = 位置费):桌子坐满时少押注

> **⏸ 审后搁置(Q24)**:当前负载下算力确实够(v2 端到端 9 路并发仍净赚 29%,说明验证没在挤兑),λ≈0 就是最优——这条是**为将来更高并发预留的旋钮**,现在做没有收益。

**现状哪里在漏钱**:max score 的前提是"验证免费"。batch=1 时近似成立(GPU 算力大量闲着,验证白捡);但生产是题内 9 路并发——桌子坐满后,你的草稿 token 每多占一个位置,别的请求就慢一点,赌注有了真实成本,现在的算法对此视而不见。

**做法**:目标从"期望赢最多"改成"**期望赢 − λ×草稿长度**",λ 按当前 batch 占用动态调:空桌 λ≈0(等于现状),满桌抬高——自动缩短草稿、放弃低把握的注。

**为什么能赚**:投机收益随并发缩水是实测过的(坑②);位置费把缩水从"被动挨打"变成"主动控制",忙时少亏、闲时照赚。

#### 3.4 概率校准:只见过一次的事,别当十拿九稳

> **❌ 审后撤回(Q24,实测回答"score 会不会影响命中")**:统计了校准曲线(6 条 summary):score 系统性**高估 ~2.5 倍**(各分桶 实际/预测 = 0.40~0.60),**但保序**——分桶单调不乱。保序意味着"挑哪份"不受影响(与 oracle 结论互证:选择近乎无损);高估影响的只是"值不值得交"的绝对判断,而它的两个下游客户(地板 3.1、λ 3.3)都已撤回/搁置——**校准单独做没有客户**。

**现状哪里在漏钱**:置信度=频率,"这串话在 prompt 里只出现过 1 次、后面跟的是 X"会得到 prob=1.0——**样本量是 1 的"必然"**。模型转述时完全可能换个说法,这份"必然"虚高 → score 虚高 → 择优失真。

**做法**:平滑——给没见过的续写留一点概率(如 (count+α)/(total+αV)),或把"仅出现 1 次"的路径置信度封顶 0.9。

**为什么能赚**:score 更诚实,3.1 的地板和候选择优判断更准。单独收益小,是 3.1/3.2 的配套件。

### 问 6(补充):针对根因①的方案

> **追问(2026-07-04):EAGLE 与研究 motivation 的关系。**(D1~D4 已展开成研究蓝图 → [research-directions-copy-speculation.md](research-directions-copy-speculation.md)) 用户定位:研究的原始 motivation 是"LLM 会照搬输入/其他 agent 的输出作为自己的输出",EAGLE 让小网络自己猜、与照搬现象无关 → **EAGLE 定位降为必比 baseline,不是论文主体**。新颖性方向沿 motivation 排:**D1** 多智能体输出级冗余的系统性利用(taxonomy:抄输入/抄自己/抄兄弟 agent,v2 三种皆有实测;编排器知道任务图 → 草稿源按图谱预取,与 KV 层复用(KVCOMM/blend)合并为"两层复用"叙事;decode 侧跨 agent 复用文献少,prefill 侧已有 DroidSpeak/KVCOMM);**D2** copy-state 感知的混合投机仲裁(抄写段 suffix 一口 20+ vs EAGLE 3~5,创作段反之,**互不支配 → 仲裁是真问题**,match 状态=免费信号);**D3** 微改写鲁棒的检索草稿(修"一个'而'字打掉 6 token");**D4** harvest 定律+税分解作为表征贡献。论文骨架:现象表征 → 图谱感知草稿源 + copy-state 仲裁 + 探针路由 → 端到端赢 EAGLE-only 和 suffix-only(EAGLE 在其中= baseline + 仲裁的一个臂)。行动:文献核查(REST / SAM-decoding / cascade & adaptive drafting / DroidSpeak);税拆分实验照跑(它定"不抄段该交给谁"的账)。


- **最对症:换/混学习型 drafter(EAGLE3/MTP)。** 81% 零接受步是查表法结构性够不着的;EAGLE 类靠小网络续写、不依赖照搬,任意文本典型 2~4 token/步。落地:核实 Qwen3-32B 有无现成 EAGLE3 head → 同 80 条 replay + 端到端各测一轮;或**按阶段分池**(report 池 suffix、summary 池 EAGLE)。
- **降税包**:路线 B 三件套 + `max_tree_depth` 调小(**全实验唯一没扫过的杠杆**,树浅 propose 便宜,代价是超长照搬多拆几口,report 侧可能小亏,值得扫一版)。
- **别走回头路**:L3 引用模式已否决(吞吐 x1.49 但膨胀 +72% 吃光、1/10 撑爆上下文);summary 调参已证无解(0.1 即最优)。

#### 3.6 max_tree_depth 调小:货架别按 24 层收管理费

**现状哪里在漏钱**:树深 24 意味着每个 token 入树要插 24 层、每步查树最多走 24 层。但 summary 实测**最长**匹配才 17(中位远低于此)——多数时候 18~24 层是白建白查的。它还是全实验唯一没扫过的参数。

**做法**:扫 {12, 16, 24} 三点,summary 和 report 各测一轮。

**为什么能赚 / 风险**:直接降建树查树的 CPU;风险在 report 侧——超长照搬段受"匹配+草稿 ≤ 树深"的结构墙限制,树浅了 300 token 的长段要多拆几口,可能小亏。所以用数据说话,别拍脑袋定。

#### 3.7 EAGLE3 混合:给"不在抄"的 81% 换一个会说话的猜题人(最对症)

**现状哪里在漏钱**:根因①是天花板——summary 81% 的零接受步里,模型在说**自己的话**,prompt 树里根本没有答案。suffix 是"只会背这份 prompt 的复读机":考题在书里就飞快(report 1.76x),不在书里就干瞪眼、纯交税。**这不是调参能修的,是查表法的物种极限**(树-oracle 上界 1.50x 也翻不过去)。

**做法**:换/混一个**学习型**猜题人——EAGLE3/MTP 这类"在大模型身边训练出来的小学徒"(零基础讲解 → [../../20-eagle-mtp-explained.md](../../../20-eagle-mtp-explained.md)):不需要书里有原文,靠学过大模型的说话方式直接猜,任意文本典型每步对 2~4 个。落地两条路:整机换 EAGLE 全量对比;或**按阶段分池**——report 池继续 suffix(零训练、已验证),summary 池用 EAGLE。

**为什么能赚**:把 81% 的"纯税步"变成"也能白捡 1~3 个的步",唯一直接打根因①的方案。**附赠**:EAGLE 的草稿产在 GPU 前向内部,天然**兼容异步调度**——18% 税里"关异步"那半边直接要回来。代价:需要 Qwen3-32B 的 EAGLE3 头(先核实社区权重,没有就得训),显存多占一份小网络。

**④ 本文独有、对方缺失的**:**税成分拆分实验(§3.9)**——对方"关异步是大头、查树是小头"的说法和本文 Q24 之前一样是**未测的假设**;此实验同时裁决双方的降税优先级,保留为高优。另:两层 oracle 的"上界阶梯"框架(§一)是本文独有的方法论沉淀。
