# 从零讲通:沿 provenance 图解码(Provenance-Guided Decoding)

> **缘起**:这篇把一次"paper 故事打磨"的对话展开成小白能从头读通的教程。原始那条回答讲的是——研究智能体(research agent)的解码怎么加速,核心 idea 是"给草稿分段、每段指定出处(provenance)",以及怎么把它拔高到"一张图、三个角色"的框架。这里从"大模型怎么蹦字"讲起,一路搭到那个框架为止。
>
> **读法**:
> - 只想要结论 → 看 **TL;DR** + **§8 一张图三个角色**;
> - 想读懂为什么 → 从 **§1 概念梯子** 顺着往下,每节只加一个新想法;
> - 想知道"这 idea 到底靠不靠谱" → 直接跳 **§10 追问区** 的 Q2(诚实风险)和 **§12 进阶**;
> - **本文管"原理/故事"**;后缀树的源码接线看 [../sim-suffix-explained.md](../sim-suffix-explained.md),机制账(下注经济学/成本常数)看 [proposal-mechanism-acceptance-uplift.md](proposal-mechanism-acceptance-uplift.md),各 idea 的判决看 [idea-verdicts-and-standing.md](idea-verdicts-and-standing.md)。三者互挂,不重复。

## TL;DR(一分钟版)

1. **研究智能体慢在两头**:一头是 **prefill**(把长 prompt 一次性读进去),一头是 **decode**(一个字一个字往外蹦)。两头都重。
2. 想用**投机解码**(先草稿、后验证)加速,但发现**命中率不高**(生产口径 ~26.7%)。
3. **原因**:朴素投机是"从整个语料里瞎找草稿",它不知道**这一段输出到底是照着哪一段来源写的**。
4. **洞察**:智能体其实**自己知道**——它做小结时知道在读哪几页,做报告时知道在综合哪几个小结。这层"谁来自谁"的关系叫 **provenance(出处/血统)**。**智能体早把这张图建好了,解码时却把它扔了,当成"从零生成"。**
5. **做法**:把这张图交给解码器——给草稿**分段**,每段**指定它的出处**,只从对的来源里找草稿。
6. **拔高成框架**:这张 provenance 图不止提命中率,它**同一个对象同时驱动三件事**——① prefill 该复用哪段 KV;② decode 哪些段可以并行;③ decode 每段照着哪个来源抄。**一张图,三个角色。** 这就是让故事从"缝了三个 trick"变成"看穿了一个结构"的关键。
7. **诚实提醒**:三个角色里,**①(KV 复用)和 ②(并行)最硬**,**③(靠出处提命中率)是最险的一条腿**——我们已有实验显示接受率被"内容可抄性"锁住(~29%),换来源子集不一定破得了(详见 §10 Q2)。所以故事的重量该压在 ①②,③ 当"我们 characterization 指出的机会"。

## §1 起点:大模型是"一个字一个字往外蹦"

**起点假设**:你只知道"大模型能生成文本",不知道它内部怎么一步步出字的。就从这儿起步。

大模型生成文本是**自回归**的——**回**头看**自**己已经写出的所有字,预测下一个字,写下来,再回头看(现在多了一个字),再predict下一个……如此一个 token 一个 token 往外蹦。

> **术语**:**token** = 模型眼里的最小文字单位,中文里大约 1 个 token ≈ 1.7 个汉字。下文"字""token"混用时,精确单位以 token 为准。
> **术语**:**自回归(autoregressive)** = "预测下一个要基于自己前面全部输出"。

**桥句(记住这句,后文反复用)**:**蹦每一个字都要回头看前面全部字。** 这句话是理解后面所有"为什么慢""为什么能投机"的地基。

---

## §2 生成分两段:prefill 和 decode

你给模型一段 prompt(比如"请总结以下资料:……几千字……"),它的工作分**两段**:

- **第一段 prefill(预填充)**:把你给的**整个 prompt 一次性读进去**,理解它。这一段是"读题",输入多长就要处理多长,但可以**并行**处理(一次前向把几千个 token 全过一遍)。
- **第二段 decode(解码)**:开始**一个一个往外蹦**答案。每蹦一个,就是 §1 说的"回头看全部 + 预测下一个"。这一段**没法并行**——第 2 个字得等第 1 个字定下来才能算。

**为什么要分两段?** 因为 prompt 是**已知的**(可以一口气全读),而答案是**未知的、要顺序生成**的(蹦出来才知道下一个基于什么)。一个"批量读已知",一个"逐个猜未知",性质完全不同,所以分开。

> **桥句**:**prefill = 一口气读已知的输入;decode = 逐个蹦未知的输出。** 两头都可能是瓶颈,后面会看到研究智能体**两头都重**。

---

## §3 为什么 decode 慢:每蹦一个字都要把整个模型读一遍

这里要加一个关键事实,才能理解后面所有加速手段的动机。

模型的"知识"存在几百亿个**权重**参数里。**蹦每一个字,都要把这堆权重从显存里完整搬一遍**去做计算。对我们这套 Qwen3-32B,权重摊到每张卡上约 16 GB,显存带宽约 960 GB/s——**光把权重读一遍就要约 16.7 毫秒**,这是每步的"地板价"。实测每步约 33.8 ms(地板价 + 其他开销)。

关键结论:**decode 慢不是慢在"算得多",而是慢在"搬得多"**——瓶颈是**把权重从显存搬到计算单元**(术语叫**访存受限 / memory-bound**),不是算力不够。注意力那部分计算其实只占每步约 6% 的浮点运算量、0.8% 的访存量(详见 [../examine-spec-tax/decode-step-compute-anatomy.md](../examine-spec-tax/decode-step-compute-anatomy.md))。

> **这解释了一个反直觉现象**:既然每步的大头是"搬那 16 GB 权重",那把**多条请求攒成一批**(batch)一起 decode,这 16 GB 只搬一次就能服务整批 → 吞吐几乎白涨。实测批到 64,聚合吞吐 38.8×。这个"权重读被摊薄"的道理,后面 §8 的"并行"角色会再用到。
>
> **术语**:**KV cache** = 为了不每步都重算"回头看前面全部字",模型把前面每个字的中间结果(K、V 两个向量)缓存下来,下一步直接读缓存。所以上下文越长,decode 每步要读的 KV 越多。记住"KV = 前文的缓存",§8 的"KV 复用"会用到。

## §4 研究智能体为什么"两头都重"

**研究智能体**(deep research agent)干的活:针对一个问题,派几个"记者"各自去查一批资料、各写一份**小结(summary)**,再由"主编"把这些小结**综合成一篇长报告(report)**。

拿这个真实结构套 §2 的两段:

- **decode 重**:它要蹦出**大量长文本**——每份小结几百上千 token,报告更长。全是逐个蹦。
- **prefill 也重**:写报告时,prompt 里要塞进**所有小结**当素材。我们实测报告阶段的 prompt 高达约 8.5 万字符(≈2 万 token)——光"读题"这一下就极重。

我们对端到端墙钟做过拆分:**小结生成占 ~32.8%、报告生成占 ~27.8%、检索占 ~31.8%**。前两块就是 decode/prefill 的钱。所以"加速研究智能体"绕不开这两头。

> **桥句**:研究智能体 = **记者写小结 → 主编综合成报告** 的流水线。这个"谁综合了谁"的结构,正是后面 provenance 的来源。整篇教程都用这个"编辑部"比喻,结论会落回精确表述。

## §5 投机解码:先草稿、后验证,验证便宜、蹦字贵

decode 一步一步太慢,有个经典加速法叫**投机解码(speculative decoding)**,思路特别朴素:

> 与其老老实实一步蹦一个字,不如**先猜一串草稿**(比如"接下来 5 个字大概是 X Y Z W V"),再让大模型**一次前向把这 5 个字同时验证**——猜对的部分白赚,猜错的部分丢掉、退回正常蹦。

**为什么这能省时间?** 回到 §3:每步的大头是"搬 16 GB 权重",而**一次前向能顺便验证好几个草稿位置几乎不额外花钱**(算力本来就闲着,访存那一下已经付了)。所以:

- 蹦一个"确定对"的字 ≈ 省掉一整步 ≈ 省 26~34 ms;
- 而多验证一个草稿位置 ≈ 只花 0.62 ms(eager 实测)。

**这就是投机解码的经济学**:验证极便宜、真蹦一步极贵 → 只要草稿**平均能对上一点点就赚**。盈亏线低到"单个草稿位置有 ~2.4% 命中率"就回本([proposal-mechanism-acceptance-uplift.md](proposal-mechanism-acceptance-uplift.md) §二)。

> **术语**:**草稿(draft)** = 提前猜的一串 token;**接受率 / 命中率(acceptance rate)** = 草稿被大模型验证通过的比例;**drafter / proposer** = 负责猜草稿的那个东西。

## §6 草稿从哪来:研究智能体的输出"很多是照搬来的"

投机解码的命门是:**草稿怎么猜得准?** 通用做法是训一个小模型来猜(EAGLE 类),但研究智能体有个更省事的来源——

**它的输出很多是"照搬"来的**:小结大段照抄原文资料、报告大段照抄小结。我们量过:报告的"照搬面积"高达 91%。既然输出很多是抄的,那**草稿根本不用猜,直接去来源里找"这几个字后面接什么"就行**。

具体实现叫 **suffix decoding(后缀式投机)**:把**已经出现过的文本**(你的 prompt、别的记者已写完的输出)建成一棵**后缀树**——一个能快速回答"这个片段在历史里出现过、它后面通常接什么"的数据结构。decode 到某处时,拿**当前上下文的结尾几个 token**(≤24 个)去树里查,查到就把"树里记录的后续"当草稿发出去。

> **后缀树本身怎么建、怎么查**,是另一篇的活:[../sim-suffix-explained.md](../sim-suffix-explained.md)(源码级手把手)。本文只需要你记住一句:**suffix decoding = 拿上下文结尾去"历史文本"里查后续当草稿**。
>
> **术语**:**local 树 vs global 树** —— local 树 = 只用**本请求自己的 prompt** 建的树;global 树 = 把**此前所有请求的输出**也堆进去的共享树(别的记者写完就进树,后来者能抄)。

## §7 问题来了:从"整个语料"里瞎找,命中率上不去

朴素 suffix decoding 有个隐痛:它把**所有能抄的东西堆成一棵大树**,拿上下文结尾去里面查。可**同一个结尾片段在一大堆地方出现过**,树里岔路很多,不知道该信哪条 → 猜出的草稿证据弱、发得浅。

看真实数字(生产口径,census 40 题):

| 环节 | 数字 | 读法 |
|---|---|---|
| **匹配率** | ≥97% 的步都能在树里匹配上 | "找得到候选"根本不稀缺 |
| **每步草稿长度** | 平均 2.9 个 token,其中 66% 只压 1~2 个 | 证据弱 → 只敢发很浅的草稿 |
| **接受率(命中率)** | **26.7%** | **这才是稀缺资源** |

三级漏斗:**匹配(97%,不缺)→ 下注(浅,均 2.9)→ 命中(26.7%,稀缺)**。瓶颈卡在最后一环。

**再往深看,内容分两半**(这半段决定了命中率的天花板):

- 结构化内容(报告照抄小结、查询计划)命中率高:**报告 59.5%、query_plan 39.4%**;
- 论述/创作型的小结命中率低:**~26.7%**,而且很多步是模型自己写的"过渡话",压根不在任何来源里。

对应到端到端速度:**报告段 2.29× 加速,小结段只有 1.00×**([copy-speculation §〇](research-directions-copy-speculation.md))。抄得多的地方快,自己写的地方白搭。

> **桥句(idea 的起跳点)**:命中率低的一大截原因是——**朴素方法不知道"这一段输出到底照着哪一段来源写的"**,只能在整片语料里翻。如果我们能告诉它"这一段就是誊自甲记者第 3 段小结",它就能对着那一段照抄,而不是瞎翻。**这个"这一段来自哪"的信息,就是 provenance。**

## §8 洞察:智能体自己知道出处,却在解码时把它扔了

**provenance(出处 / 血统)** = "这段输出是从哪段输入来的"的对应关系。用编辑部比喻:**报道每一段底下的那个"署名 + 出处脚注"**——这段来自甲记者的小结,那份小结来自第 1、2 页资料。

**关键洞察**:研究智能体在干活的过程里,**天然就知道这些出处**——

- 记者写小结时,**知道自己在读哪几页资料**;
- 主编写报告某段时,**知道自己在综合哪几份小结**。

也就是说,智能体**早就在心里建好了一张 provenance 图**(报告段 ← 小结 ← 资料页),可它把这张图**只用来组织工作流,到了解码引擎那一层就扔了**——解码引擎收到的只是"一段 prompt + 请生成",它把每个输出 token 都当成"从零凭空生成",完全不知道这段其实是**照着某个已知来源在誊写**。

> **这就是故事的 insight**:**现有服务栈把"结构化智能体的解码"当成"无结构的从零生成",丢掉了智能体自己的数据流。** 我们要做的,就是**把这张图捡回来,交给解码器**。

### 一张图,三个角色

把 provenance 图交给解码器,它**同一个对象能同时驱动三件事**(这是让故事上台阶的关键——不是三个独立 trick,是一个结构推出的三个后果):

| 角色 | 干什么 | 治哪个瓶颈 | 编辑部比喻 | 硬不硬 |
|---|---|---|---|---|
| **① prefill:KV 复用** | 报告的 prompt = 那些小结;而小结的 KV 在**记者写它时已经算过一遍了** → 别让主编重新 prefill,直接**复用小结的 KV** | prefill(§4 的 2 万 token 大头) | 记者读资料形成的"理解",直接交给主编,别让主编重读 | **最硬**(你的 KVCOMM/LMCache 老本行) |
| **② decode:并行分解** | 报告第 1 段抄小结 A、第 2 段抄小结 B,**互不依赖** → 可以同时解码 | decode(§3 权重读可摊薄) | 不同段来自不同记者、互不干扰 → 几段同时誊 | **硬**(现成软件栈可证) |
| **③ decode:草稿路由** | 生成某段时,**只从它的出处来源里找草稿**(而非整片语料)→ 证据强 → 敢发深草稿 → 命中率高 | decode(§7 的 26.7% 天花板) | 誊某段时照着它对应的那份小结抄 | **最险**(见 §10 Q2) |

一句话:**provenance 分解是同一个对象,它同时给你 prefill 的复用边界、decode 的并行边界、和 decode 的草稿来源。** 这就是"一张图,三个角色"。

## §9 手把手实例:一个能手算的编辑部小例子

整节用**同一个玩具例子**走完"建图 → 三个角色各怎么用"。数字凡标"示意"的是为讲解编的,标"实测"的是真实数据。

### 9.0 场景(小到能手算)

问题:**"A 县 2020 年人口多少?比 2010 年增长多少?"**

两页资料:
- `page_1`:"2020 年人口普查:A 县常住人口 **52,341** 人。"
- `page_2`:"2010 年人口普查:A 县常住人口 **48,900** 人。"

甲记者读了这两页,写出小结:
- `summary_A`:"A 县 2020 年人口 **52,341**,较 2010 年的 **48,900** 增长约 7%。"

主编写报告第 1 段,引用了这份小结:
- `report §1`:"……A 县 2020 年人口 52,341,较十年前增长约 7%……"

### 9.1 建 provenance 图(这张图智能体本来就有)

```
report §1  ←──  summary_A  ←──  { page_1, page_2 }
```

用人话:报告 §1 的这句话,**出处是 summary_A**;summary_A 的出处是 page_1、page_2。

### 9.2 角色③(草稿路由)——一步 decode 的显微镜

假设主编正在蹦 report §1,已经蹦到 "……A 县 2020 年人口",下一个字该是什么?看**朴素**做法 vs **provenance-guided** 做法在**这一步**的差别。

一步投机解码内部发生什么(**单步显微镜**,①→⑤ 有先后):
1. proposer 拿上下文结尾片段(如 "人口")去树里**匹配**;
2. 匹配到 → **发一串草稿**(能发多深看证据多强);
3. 大模型**一次前向**把草稿的每个位置**同时验证**(这一下便宜,§5);
4. **接受最长正确前缀 + 采样 1 个新 token**;
5. 更新上下文和树,进入下一步。

差别全在第 1、2 步"去哪匹配、证据多强":

| | 朴素(整片语料树) | provenance-guided(只查出处) |
|---|---|---|
| 第①步 去哪查 "人口" 的后续 | 整个语料——"人口"在几十处出现过,岔路多 | **只在 summary_A / page_1 里查** |
| 第②步 发多深草稿 | 证据弱 → 只敢发 1~2 个(如 "，") | 证据唯一 → 敢发 "5""2""3""4""1" 五连 |
| 第③④步 大模型验证后接受 | 接受 ~1(示意) | 接受 5(示意) |

**为什么 provenance 版敢发深、命中高?** 因为限定到出处后,"人口"后面**几乎唯一**地接 "52,341"——岔路没了,证据从"弱"变"强",于是从"发 1~2 浅注"变成"发 5 连深注"。这正对上 §5 的经济学:位置便宜(0.62 ms)、盈亏线低(2.4%),**只要敢发深、且深位置还有命中,就赚**。实测尾部位置命中率:p4≈18%、p8≈12%、p10-12≈5~9%,全在盈亏线之上——**当前策略其实"下注太浅"**([proposal-mechanism §二](proposal-mechanism-acceptance-uplift.md))。

> ⚠️ **教学缩参提醒**:例子里"接受 5"是示意。真实小结的接受率被内容形态压在 ~26.7%,不是随便就能到 5。这个例子演示的是**机制方向**(限定出处 → 证据变强 → 敢发深),**幅度**能不能兑现是 §10 Q2 的争议点。

### 9.3 角色①(prefill KV 复用)——省的是"读题"钱

主编写整篇报告时,prompt 里塞了 summary_A、summary_B、summary_C……(§4 的 2 万 token 大头)。但 **summary_A 这串 token 的 KV,甲记者在写出它的时候就已经算过一遍了**(那也是一次 decode/prefill)。

- **朴素**:主编的 prompt 从头 prefill 一遍,summary_A 又被重新读一次;
- **provenance-guided**:provenance 图告诉你 report 的 prompt 里"这一段 = summary_A" → **直接把 summary_A 生成时的 KV 搬过来接上**(LMCache/blend 式 KV 搬运),这一段不用重算。

这一刀治的是 **prefill 瓶颈**,而且**恰好是 KVCOMM/LMCache 的主场**——所以它是三条腿里**最硬**的一条。

### 9.4 角色②(并行分解)——独立段同时誊

report §1 抄 summary_A、report §2 抄 summary_B,**两段的内容互不依赖**(provenance 图上是两条不相交的链)。于是它们可以**塞进同一个 decode 批一起蹦**——回到 §3 的道理:那 16 GB 权重读**一次就摊给两段**,几乎白赚吞吐(实测批处理这条曲线,batch 64 到 38.8×)。

> **注意边界**:并不是所有段都独立。若 report §3 需要"综合 §1 和 §2 的结论"再写,那它得等 §1、§2。provenance 图恰好告诉你**哪些段是独立叶子、哪些有依赖**——这就是图的价值:**并行边界不是拍脑袋,是从图上读出来的**。

## §10 追问区(预判你会卡在哪)

### Q1. provenance 和"输入输出重叠 20%"是一回事吗?

**不是,差在"知不知道对应关系"。**

- **"重叠 20%"** 是个**事后统计的标量**:把输出和输入放一起,数有多少 token 段是重合的,得一个比例。它只告诉你"有多少能抄",**不告诉你哪一段抄的是哪一段**。
- **provenance** 是**一张带方向的对应图**:report §1 这一句 ← summary_A 这一段 ← page_1 这一行。它是**结构**,不是比例。

打比方:重叠率 = "这篇读书报告里有 20% 的句子在原书里能找到";provenance = "**第 3 句**誊自原书**第 47 页第 2 段**"。前者是统计,后者是**可拿来路由草稿/复用 KV 的地址**。投机解码要的是后者——你得知道"往哪查",光知道"有 20% 能查到"没用。

### Q2.(最要命)换成 provenance-局部草稿源,接受率真能破 26.7% 天花板吗?

**这是整个 idea 最险的一条腿,得诚实说。** 我们已经有一个**决定性实验**指向"悬":

- **idea #4(兄弟互抄增强)已判死**([idea-verdicts §一.4](idea-verdicts-and-standing.md)):给"对的来源"(同题兄弟输出)更高匹配优先级/更深索引/更大预算,**接受率纹丝不动**——0 个兄弟 29.2% → 满兄弟 29.5%,Δ+0.3pp。
- **根因**:token 级"可抄内容"本就 ≈0——兄弟输出↔输出 cov24 仅 **0.69%**(中位 0.00%)、输入↔输出 cov24 ~0.4%。**字符级"看着像"(union 覆盖 ~31.7%),token 级"没得抄"。** 接受率天花板由**内容可抄性**决定,不由**树怎么组织**决定。

**provenance-局部草稿路由,本质是"换/限定草稿源"的一种**——它和已判死的 #4 是**表亲**。所以 Q2 的诚实答案是:

- **对论述/创作型内容(小结主体):大概率破不了**,因为那些 token 根本不在任何来源里(是模型自己的过渡话),限定到哪个源都没用;
- **对结构化内容(报告照抄小结、清单/表格):本来就已经 39~59% 命中**,这里 provenance 可能帮的是"**敢发更深**"(§9.2 的深注),但增益来自"下注更深"(经济学),**不是"换源提率"**;
- **所以**:③ 号角色若要成立,论证必须落在"**provenance 让我敢在结构化段落发深注**"(经济学路线,[proposal-mechanism §三路线1](proposal-mechanism-acceptance-uplift.md)),而**不能**落在"换个源子集就破内容天花板"——后者我们的数据直接反对。

**给故事的处方**(也是原始那条回答的落点):**把重量压在 ①②(不依赖接受率的硬腿),③ 当"characterization 指出的机会"**,别让整篇的成败吊在最险的一条腿上。

### Q3. KV 复用不就是 prefix caching 吗?有啥新的?

prefix caching 只在**前缀逐字完全相同**时命中(比如两条请求开头一模一样)。而报告的 prompt 是把小结**重新拼装**进去的——顺序、分隔、上下文都变了,**逐字前缀对不上**,prefix caching 基本不命中。provenance 的新意是:它知道"prompt 里这一段**语义上等于**那份已算过的小结",于是能做**乱序、跨请求的 KV 复用**(blend/KVCOMM 式),这是 prefix caching 做不到的。

### Q4. 并行解码不是已经有异步 batching 了吗?

现有 batching 是"**碰巧同时到达的请求**攒一批"——被动、机会性的。provenance 的并行是"**同一篇报告内部,主动把独立段拆出来**并行",是**请求内**的结构化并行,粒度和来源都不同。两者叠加,不冲突。

### Q5. 这跟 cascade attention 什么关系?

没直接关系,别混。cascade attention 是"一批请求**共享同一段长前缀**时,那段前缀的 KV 只读一遍"——是 attention kernel 层的省访存。我们实测 vLLM 默认根本不开它、开了在 TP4 下也要 batch≥72 才被性能模型选中([../examine-spec-tax/cascade-attention-verdict.md](../examine-spec-tax/cascade-attention-verdict.md))。provenance 是**上层的数据流结构**,cascade 是**底层的 kernel 优化**,两者正交。

### Q6. global 树不是已经"被动"实现 provenance 了吗?

部分是,但**只在"命中率"不在"路由/并行/prefill"**。global 树把所有输出堆进去,后来者"碰巧"能抄到前面的——这是**无差别堆料**,不是**有向图**。它不知道"report §1 **就该**抄 summary_A",只是"summary_A 恰好在树里,查到了算运气"。而且实测 global 树相对 local 树,**只把接受率从 26 抬到 ~30、char/s +9%,早早饱和**——因为它抬的是"匹配质量",抬不动"内容天花板"(同 Q2 根因)。provenance 图的价值恰恰是把"被动堆料"变成"**有向、可路由、可并行、可复用 KV** 的结构"。

## §11 术语表(忘了回来查)

| 术语 | 一句话 |
|---|---|
| token | 模型的最小文字单位,中文 ≈ 1.7 汉字/token |
| 自回归 | 预测下一个 token 要基于自己已输出的全部 |
| prefill | 把整个 prompt 一次性读进去(可并行) |
| decode | 逐个往外蹦输出(不可并行,每步读一遍全部权重) |
| KV cache | 前文每个 token 的中间结果缓存,decode 时读它免得重算 |
| 访存受限(memory-bound) | 瓶颈是"搬权重/KV",不是"算" |
| 投机解码 | 先猜一串草稿,大模型一次前向同时验证,对的白赚 |
| 草稿(draft) | 提前猜的一串 token |
| 接受率/命中率 | 草稿被验证通过的比例(生产口径 ~26.7%) |
| suffix decoding | 拿上下文结尾去"历史文本"里查后续当草稿 |
| local 树 / global 树 | 只用本请求 prompt 建 / 堆入所有历史输出的共享树 |
| provenance | "这段输出来自哪段输入"的**有向对应图**(≠ 重叠比例) |

## §12 进阶:跟已有方案的关系 + 三条腿的诚实排序(可跳过,不影响理解)

**跟已有工作的定位**(每个都是"必比 baseline"):
- **SoT(Skeleton-of-Thought)**:先出提纲再并行填充——和角色②的并行**撞车**,但 SoT 是纯 prompt 技巧、不碰 KV/草稿。provenance 的并行是从**真实数据流**读出的边界,且能顺带做 ①③。
- **PLD / prompt-lookup decoding**:从 prompt 里找草稿——是角色③的**退化版**(只查本请求 prompt = local 树),没有跨源的 provenance 路由。
- **prefix caching / RadixAttention**:逐字前缀复用——角色① 的**退化版**(见 Q3,只能吃完全相同前缀)。
- **EAGLE 类学习型 drafter**:训小模型猜草稿——和 suffix 正交,是③的**另一条来源路线**,列为对照 baseline([copy-speculation §〇](research-directions-copy-speculation.md))。

**三条腿的诚实排序**(决定故事重量该压哪):
1. **① prefill KV 复用 —— 最硬**。不依赖接受率;治的是实测 ~28% 的 e2e 大头(报告 prefill);正好是你 KVCOMM/LMCache 的主场,好实现、好防守。
2. **② 并行分解 —— 硬**。不依赖接受率;现成 batching 栈可证;唯一要小心的是"段间依赖"要从图上正确读出(§9.4 边界)。
3. **③ 出处提命中 —— 最险**。直接撞我们已判死的 #4(接受率内容天花板);**只有走"经济学/敢发深注"这条论证**才站得住,**不能**走"换源破天花板"。当"characterization 机会",别当主承重。

**一句话结论**:这个 idea 作为**框架("一张图、三个角色")**是站得住的、也够顶会级的 insight;但它的**说服力要建在 ①②(不吊在接受率上),再用 characterization 图先证明"provenance-局部下确实敢发深、命中更高",才轮到 ③ 上场**。
