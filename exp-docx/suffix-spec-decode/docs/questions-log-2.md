## §零、共同地基：一步 decode 到底花多少钱（五常数模型 + 三级漏斗）

后面好几问都在问"成本"，先把成本模型摆出来，一次讲清。

### 地基 1：一步的钱拆成五个常数

我们把"每个 decode 步的耗时"拆成五笔可加的钱（实测，summary 段，单位 ms/步）：

| 记号 | 名字 | 值 | 什么时候交 |
|---|---|--:|---|
| ① | graph 基线 | 24.0 | 每步都交（读一遍 32B 权重的物理底价 + 整图回放） |
| ② | eager 附加 | +0.5 | 用 `--enforce-eager` 时（放弃整图、逐个发 kernel 的净代价） |
| ③ | spec 管线 | +1.9 | 投机机制开着就交（与草稿多少无关、与执行模式无关的"手续费"） |
| ④ | 混合边界税 | +8.1 | 只有 graph+eager 混合模式交（驱动锁等待 4.5 + 图段启动 1.4 + 杂项）——**全 eager 一刀消掉** |
| ⑤ | 草稿费 | 激活 5.8 + 0.62×位置数 | 这一步"有草稿"就交（详见 Q4） |

- **原生 A** = ① = 24.0
- **推荐配置 Ceager**（suffix + 全 eager）= ①+②+③+⑤ = 24.0+0.5+1.9+7.5 ≈ 33.9（草稿费按平均 2.7 个位置算 = 5.8+0.62×2.7=7.5）
- 七个配置预测=实测残差 0，模型可信。物证：[cost-anatomy §二](examine-spec-tax/suffix-vs-native-cost-anatomy.md)。

### 地基 2：三级漏斗（投机的收益怎么来的）

投机每一步做三件事，像一个越缩越窄的漏斗：

```
① 匹配：拿上下文结尾去两棵树里找（自己 prompt 树 + 全局树）
        ≥97% 的步能找到（门槛低到"最后 1 个 token 出现过"就算）→ 不稀缺
        ↓
② 下注：找到就抄它后面的续写当草稿，发 dr 个草稿 token
        平均 2.9 个/步，66% 只押 1-2 个 → 押得浅
        ↓
③ 命中：草稿进模型验证，和模型真正要说的 token 一致才算接受
        summary 命中率 26.7% → 这才是稀缺资源
```

> **桥句（全文回指）**：投机是一门"**用步时买 token**"的生意——每步多花点钱（②③的成本），换每步多吐几个 token（③命中的草稿）。赚不赚，看**买 token 的单价** vs **一个 token 的价值**。

## Q1：per-request-variance 说某些 summary 没加速，我又说 e2e 全部更快——矛盾吗？

**结论先行**：不是"每条都更快"（有 2 题亏、13 题平），但也**远比我初稿说的乐观**——**逐题看是 25 胜 / 13 平 / 2 负，中位 1.04x**。我最初用"45% 调用低于 A 中位"论证"很多没加速"，那个统计**是误导的**（下面纠正）。

### 正确口径：逐题配对（同一题，A 和 Ceager 都生成 summary，可配对）

e2e 里逐个 summary 调用没法 1:1 配对（temp0 下不同 researcher 内容不同），但**逐题**可以——每题把该题所有 summary 调用聚合成一个 tok/s 再比。40 题结果：

| 统计 | 值 |
|---|---|
| 逐题比值中位 | **1.04x**（均值 1.07x，范围 0.95~1.26） |
| 加速（>1.03x） | **25 题** |
| 平（0.97~1.03） | 13 题 |
| 亏（<0.97） | **2 题**（q37 0.95x、q38 0.95x——都是 A2 里命中率垫底的难抄题：爵士钢琴/游戏画像） |

**所以 summary 绝大多数题都加速**，只有 2 题轻亏——replay 里那种 0.71x 的重亏，在 e2e 热树下**没有复现**（最差也就 0.95x）。

### 纠正我初稿的误导统计（第三处自纠）

初稿用"Ceager 有 45% 的调用低于 A 的中位数"暗示"很多亏"——**错**：两条大面积重叠的分布，本来就约一半落在对方中位线以下；**若两者完全相同会有正好 50% 低于对方中位**，所以 **45% < 50% 恰恰说明 Ceager 略好**，不是亏。同理"16% 低于 20 tok/s"也没对照——A 的 p10 也是 18.4，A 一样有一堆低于 20 的调用。**用"占对方中位以下的比例"来讲输赢是错的口径**。

（逐调用分布本身没错，仅供看形状：Ceager p10/中位/p90 = 18.4/26.7/36.6，A = 18.4/25.6/32.7——整条右移一小格。）

### 和 replay 变异分析的关系

[per-request-speedup-variance.md](explore-idea/data-source/per-request-speedup-variance.md) 讲 **replay 冷树**下逐条输赢分化（i48 0.71x 这类重亏）；e2e 热树下分化仍在但**大幅收窄**（最差 0.95x），聚合从冷树 ~1.00x 抬到 1.07x。**不矛盾**：热树把冷树的重亏磨成了轻亏、把聚合翻正。

> **落回精确**："summary 1.07x"= 聚合口径;逐题 25胜/13平/2负、中位 1.04x;"端到端回归率 2/40"是题级 decode 口径。三个口径都指向同一结论(多数加速、极少数轻亏),别用"占对方中位以下比例"这种错口径讲输赢。

## Q2："0.62ms/位置、盈亏线 2.4%、pos10-12 还有 5-9%"——展开讲

这三个数讲的是**赌场经济学**，一句一句拆。

**"下注一个草稿位置 = 花 0.62ms 赌一把"**：eager 下，每让草稿多深一个 token，这一步就多花 0.62ms（去验证它）。这是实测的按量单价（Q4 讲怎么测的）。

**"赌赢 = 省一整步 ≈ 26ms"**：如果这个草稿位置被接受，等于这个 token 不用再单独走一步 decode 生成——省下一整步 ~24-34ms。

**"盈亏线 = 2.4%"**：花 0.62ms 赌，赢了省 ~26ms。所以只要**赢面 ≥ 0.62/26 ≈ 2.4%**，这一注的期望就是正的（下注平均能赚）。像赌场里赔率 1:42 的赌桌——只要你的胜率超过 2.4% 就该一直下注。

**"pos10-12 还有 5-9%"**（实测逐位置命中率）：

| 位置 | pos1 | pos2 | pos4 | pos6 | pos8 | pos10 | pos12 | pos13+ |
|---|--:|--:|--:|--:|--:|--:|--:|--:|
| 命中率 | 45.7% | 29.5% | 13.7% | 8.1% | 5.7% | 4.1% | 2.9% | 0% |

**关键**：一直到 pos12（2.9%）都还在盈亏线（2.4%）之上——**理论上这些位置都值得赌**。但当前策略平均只赌 2.7 个位置就收手了（预算公式 budget=match_len×factor 在弱匹配时给的额度小）。**pos5-12 这些还有赚头的位置，被放着没赌**——这就是"深注扫描"（[提案路线 1](explore-idea/proposal-mechanism-acceptance-uplift.md)）的机会。

> **诚实警告（幸存者偏差）**：上表的 pos12=2.9% 是**当前策略下敢赌那么深的那些注**的命中率——它们是"树很自信"才发的。如果强行在**不自信**的匹配上也赌到 pos12，新增位置的命中率会**低于**这个survivor率，可能跌破盈亏线。所以"深注扫描"要实测，不能照着这张表线性外推。好在盈亏线只有 2.4%，容错垫厚。

## Q3：research-directions 说"兄弟源没吃到"，我又说"+1.8pp 已吃满"——谁对？

> **这一节被用户逐问打磨了 4 轮，我错了 3 次。下面直接给最终诚实结论 + 三个方法定义（用户点名要澄清），过程中的错法留作教训。**

### 最终结论（一句话）

**判不了——现有数据不足以定生死，得靠控制变量消融。** 我上一版据 v2 的"42.5%"把 idea #4 从 ❌ 降级到 ⚠️，**那个降级站不住**：干净口径下 census（密兄弟）summary 接受率 28.3% ≈ v2 E1（疏兄弟）26.4%，**兄弟几乎没把接受率抬起来**，反而印证我最初的 +1.8pp。所以 idea #4 回到 **⚠️ 真·存疑**（不是"已吃满"也不是"有大肉"），唯一的判据是消融实验。

### 我错在哪（3 次，同一个坏习惯：编解释 + 不配口径的快速"确认"）

| 轮 | 我的说法 | 错因 |
|---|---|---|
| 1 | "v2 多轮画像 vs census DRBench，题不同" | 前提错：查 workload，两者都 DRBench、题目重叠 |
| 2 | "census union 15.8% << v2 31.7%，坐实雷同度低" | 指标错：我算的是"窗口命中率"，v2 是"并集覆盖面积"；换对口径 census 也 ~31% |
| 3 | "food 一样但 42.5 vs 28.3 差 14pp" | 口径不对等：42.5% 是 v2 手工切窗、混着 report；干净全 tag 窗口 v2 45% vs census 37%，差仅 ~8pp |

### 排查结果：所有"差异"证伪，食物/参数/题目全一样

| 维度 | v2 | census | 判 |
|---|---|---|---|
| 题目 | DRBench | DRBench（重叠） | 一样 |
| suffix 参数 | `{"method":"suffix"}` num_spec=24 | 同左（默认） | 一样 |
| union 覆盖（**同口径**） | 31.7% | q1-10 **30.8%** | 一样 |
| 全 tag 逐窗口接受率中位 | 44.8% | 37.0% | 差 ~8pp（口径含手工切窗成分） |

关键推论：census 是"密兄弟 + union 31%"，summary 接受率 28.3%；v2 E1 是"疏兄弟"，26.4%。**28.3% ≈ 26.4%** → 干净口径下密兄弟没抬起 summary 接受率，`26.4→42.5 翻倍`很可能是 v2 手工窗 + 并发实时互命中的高估。

### 三个方法定义（用户点名澄清）

**① 接受率怎么算**：= 被接受草稿 token 数 ÷ 发出草稿 token 数（per-draft-token）。两种取法：
- **轨迹法（census 用，干净）**：SUFFIX_TRAJ 每步记 `acc`（=`len(sampled_ids)−1`=被接受草稿数）和 `dr`（发出草稿数），接受率 = **Σacc/Σdr**，靠 req_id join 到 tag → **能只取 summary**。census summary=28.3%。
- **server 窗口法（v2 只能用这个）**：vLLM 周期打印的 "Avg Draft acceptance rate"，一个时间窗内 accepted/drafted、**全 tag 混**。v2 的 42.5% 是手工挑"summary 占主导的窗"——没轨迹插桩（7/5 才加），没法干净按 tag 隔离，**这是口径不对等的根**。
- **轨迹法已验证（2026-07-06）**：census 轨迹全 tag 聚合 **38.8%** ≈ vLLM /metrics 计数器（地面真值）**38.9%**（差 0.1pp）≈ server 窗口中位 37.0% → 三源一致，`acc`/`dr` 记录正确。**分 tag（轨迹独有能力）可信**：REPORT 62.6% / QUERY_PLAN 38.9% / SUMMARY **28.3%** / SUPERVISOR 31.8% / INITIAL_Q 25.4%。**方法论定论：分 tag 接受率一律用轨迹法，不用 server 窗口。**

**② union 覆盖定义**：一条 summary 的输出里，被"同题更早 summary 拼接"中出现过的 **≥8 字连续块**覆盖的**字符并集面积** ÷ 输出总字符。算法：逐字符滑窗，`S[i:i+8]` 在 prior 中出现就把第 i..i+7 位标"已覆盖"（重叠只算一次），最后 覆盖字符数/总字符数。
- **我第一次的 15.8% 错在**：算的是"每 4 字取一窗、命中窗数/总窗数"（采样点命中率），不是覆盖面积——两者对不上。用并集面积重算 → census 30.8% ≈ v2 31.7%。
- **⚠️ 但 31.7% 是字符口径,会误导（2026-07-06 token 口径复核，用同一并集覆盖算法在 census 683 对上重算）**：

  | 兄弟↔兄弟（并集覆盖，census 683/723 对，中位） | char cov8 | token cov8 | token cov24 |
  |---|--:|--:|--:|
  | prior（每条 vs 更早，周报口径） | 35.5%（≈周报 31.7%✓） | 11.8% | **0.0%**（均值 2.3%） |
  | all（每条 vs 同题所有其他） | 47.2% | 21.0% | **0.0%**（均值 3.9%） |

  两种"兄弟"定义（只对更早 / 对所有同题其他）**关键的 token cov24 都是 0**——长段可抄性与定义无关地接近零。（并集覆盖算法：逐位滑窗，`seq[i:i+N]` 在源出现就把第 i..i+N-1 位标覆盖、重叠去重，覆盖位数/总长；≠ 窗口命中率。）

  > **并集覆盖 == SAM 覆盖率（同一个数，2026-07-06 证明+实测）**：周报用"字符级 SAM"、我用"定长 N-gram 窗口并集"，两者算出的**覆盖位置集合相同**。证明：位置 j 落在某个长度 ≥N 的逐字匹配段里 ⟺ 存在一个含 j 的 N 窗是源子串（"匹配段的任意 N 子窗还是匹配"，两个方向都成立）。**实测**：同一批 census 兄弟对，两法均值 37.07% vs 37.08%，逐条最大差 1.5pp（还只是我快速版用空格分隔符造的 1 字符虚假延伸）。差别只在**方法/效率**：SAM 建源的自动机、near-linear 一遍给出每位置最长匹配，**一次得到所有 N 的 cov**（cov8/cov24 同一趟）；我的 N-gram 哈希集是定长、每个 N 各扫一遍、写起来简单。**前提**：SAM 覆盖率必须也是"被 ≥N 匹配覆盖的位置并集"这个定义才等价；若换成"贪心不重叠最长匹配"或"只标匹配结束位"就会低一些。周报的 `Coverage@8 = ≥8字逐字片段覆盖率` 正是并集定义 → 与我一致。

  和**周报 §3 的输入↔输出**（char 54% → token cov8 12.9% → cov24 0.4%）**同一个 char→token 塌陷规律**——兄弟 token 级可抄性 ≈ 输入，都是短段、长段几乎零。31.7% 字符块全是同题实体/术语短片段。**周报漏做了兄弟的 token 换算、拿字符 31.7% 建了"兄弟→翻盘"叙事,该叙事被此表 + 决定性实验(接受率 flat)推翻。评可抄性一律 token 口径。**（注:早前报的"cov24 0.69%"是窗口命中率、非并集覆盖,口径已对齐周报重算。）

  > **char→token 落差的性质:大头是"同号不同尺"口径假象,不是分词丢可抄性（2026-07-06 拆解,追加问答）**。census 摘要是**中文**,1 token ≈ **1.70 字**,所以 `cov8` 两边不等长:char cov8 = 8 字 ≈ 4.7 token,**token cov8 = 8 token ≈ 13.6 字**——token cov8 其实在要求一条 1.7 倍长的逐字段,自然更低。实测覆盖率梯子(census 179 对,中位):char cov8 **35.4%** / char16 **8.2%** / char32 **0.0%**;token cov2 **71.5%** / token cov8 **12.5%**。对齐长度看:**token cov8(12.5%）≈ char 约 14 字**（落在 char16 的 8.2% 与 char8 的 35% 之间）→ char8→token8 那个 3× 落差**几乎全由"8 字 vs 13.6 字"的长度差解释**,不是分词把可抄内容"丢"了(2-token 级 token 匹配反而比等长字符更宽松,tok2 71.5%)。
  >
  > **真正非假象、且承重的事实是"匹配段本身就短"**(内容属性,与口径无关):兄弟间逐字**字符**匹配段——中位 **2 字**、P90 7、P99 18、≥32 字仅 **0.2%**;逐字 **token** 匹配段——中位 **1 token**、P90 4、P99 12、≥24 token 仅 **0.16%**。两种粒度一致地说:兄弟共享的是实体/数字/术语/短语这类**天生就短的碎片**,不是整句照抄 → 这才是 token cov24 ≈ 0 的真因。**所以"字符高 token 低"里,要修正的是"塌陷=分词丢信息"这个读法(其实是口径长度差);要坚持的是"长逐字段几乎不存在"这个结论(内容属性,换 token 口径才看得见,字符 cov8 会把它盖住)。**
  >
  > **能不能让 token 覆盖逼近字符覆盖?** ①只想数字对齐 → 等长比就已对齐(char14 ≈ token8),gap 是定义造成的、无需治;②想造出长 token 段(cov24>0)让草稿更长 → **不行,除非改生成**:长段不存在是因模型在转述,字节级 drafter/复制后重编码都变不出原文没有的长段。唯一能造长段的是**强制逐字引用**(L3 实验),实测抬了可抄量(suffix tok/s +49%)但输出膨胀 +72% 倒贴、质量平手、伤 1 题 → 净亏（见 [l0-l3-v2-results.md](../l0-l3-v2-results.md) E3）;③即使填了 gap,**接受率也不动**(对兄弟数实测全平 ~29%,内容天花板),多出的短段只走"更多短草稿→每步 token +9% 且早饱和"老路,非真加速。

**②b 追问:兄弟 token cov8 只 ~10%,为什么接受率有 28%?** 因为**接受率和覆盖率不是一个数,不能直接比**——三处错位(census token 级并集覆盖,中位):

  | 草稿来源 | cov2 | cov8 |
  |---|--:|--:|
  | 自重复(抄自己前文) | 32.9% | 1.6% |
  | prompt(抄输入) | 66.4% | 10.7% |
  | 兄弟(更早同题) | 69.6% | 10.6% |
  | **三者并集(草稿树真能抄的)** | **89.6%** | **22.3%** |

  - **①来源错位(最大)**:10% 是**只算兄弟**;草稿树里还有 prompt + 自重复 + 全局。并起来 cov8 = **22.3%**,已贴近 28%。且 **prompt(10.7%)贡献 ≈ 兄弟(10.6%)**——兄弟不是接受率主力(对上"关全局树接受率不降反升")。
  - **②长度错位**:cov8 要 8-token 连排;投机解码吃**变长匹配**、短到 2-3 token 也接受。并集 cov2 = **89.6%**。接受率 28% 落在 union cov8(22%)与 cov2(90%)**之间**。
  - **③分母错位**:覆盖率分母=所有位置;接受率分母=**只在有匹配处发出的草稿**(挑好打的打)→ 不被任何单个覆盖率卡住。
  - **一图串起来**:接受率=每草稿 token 命中率,草稿 ~2 token。**首位命中 ~46%**(靠遍地短料 cov2≈90%)→ **往后快速衰减**(要长段、而 cov8 才 22%、cov24≈0)→ 加权平均 **28%**。cov2 高 / cov8 低 / cov24≈0 / 逐位衰减 / 率 28% 是**同一件事的五个侧面**:短料撑起首位命中、长料没有所以伸不长。（数据:`~/tmp/accept_vs_cov.py`,10 题）

**③ +1.8pp 是什么**：单个兄弟**在请求生成途中**才入树的边际增量（请求内前后配对，事件组 −2.5pp vs 对照组自然漂移 −4.3pp）。它测的是"途中到货的零头"，不是兄弟总价值——但现在看，它和 census 的"密兄弟≈疏兄弟"结论方向一致（兄弟净贡献确实小）。

### research-directions 的"上界"怎么摆

union 边际 +12.9pp（14.3%→27.7%）是**离线 SAM 的可抄上界**（"完美 drafter 能抄多少"）——它说"兄弟里含可抄内容"没错，但**"含内容"不等于"能转成接受"**。census 实测：料在（union 31%）、但接受没涨（28.3%≈26.4%）→ **要么被动捕获已把能转的都转了，要么这些料本就难被 drafter 命中**。两种都指向"增强空间小"，但都需消融坐实。

## Q4：五常数里的"草稿费"具体包含什么？为什么？和草稿长度有关吗？

草稿费 = **激活费 5.8 + 0.62 × 草稿位置数**，两部分性质完全不同：

### 第一部分：激活费 5.8ms —— "开赌桌费"，交一次，不随草稿长短变

只要这一步**有草稿**（哪怕只 1 个），就得从"普通 decode 模式"切换到"投机验证模式"，这个切换本身收 5.8ms：准备验证用的张量、走 spec 采样路径（要同时给"真 token"和"草稿"排 logits）、把结果在 GPU/CPU 间搬运。**它是个台阶函数**：0 个草稿 = 0，≥1 个草稿 = 5.8，然后**再多草稿也不涨**。

- 已归因：rejection sampler + spec metadata = 1.26ms；
- **未归因 ~4.5ms**：嫌疑是"有草稿时 logits 要多行处理 / spec 采样的额外 GPU kernel / 输出拷贝路径切换"——**这是税表最后一块砖，当前头号系统研究标的**（[提案](explore-idea/proposal-mechanism-acceptance-uplift.md) 系统线）。

### 第二部分：0.62ms × 位置数 —— "里程费"，随草稿量线性涨

每让草稿多深一个 token，就多花 0.62ms 去验证它（多算一个位置的前向、多读一份 KV）。**这部分随草稿长度线性增加**。

### 怎么测出来的（四点拟合，残差 0）

| 配置 | 草稿位置数 | 实测草稿费 | 拟合 5.8+0.62×位置 |
|---|--:|--:|--:|
| C0eager（空草稿） | 0 | 0 | 0（无草稿不开桌） |
| CeagerK8（k8变长） | 2.4 | 7.3 | 5.8+1.49=7.3 ✓ |
| Ceager（k24变长） | 2.7 | 7.5 | 5.8+1.67=7.5 ✓ |
| CeagerK8pad（定长8） | 8.0 | 10.8 | 5.8+4.96=10.8 ✓ |

**回答"和草稿长度有关吗"**：激活费**无关**（台阶），按量费**线性相关**。所以一句话——**"有没有草稿"很贵（5.8 一次性），"草稿多长"便宜（0.62/位置）**。这个不对称是理解 Q6（k8pad 为什么更慢）和 Q7（混合投机成本）的钥匙。

## Q5："50k 字符 summary 每步多付 4-6ms（⑨斜线）"——画出来

"斜线"指的是"**每步比原生 A 多花的 ms，随 prompt 长度画成图，是一条上斜的线**"。用实测数据画（replay，summary 逐条按 prompt 字符分桶）：

```
 每步比 A 多的 ms
 13 |                                              ● 12.2   ← Ceager
 12 |                                        ╱
 11 |                                ● 11.1
 10 |●━━━━━━━━━●━━━━━━━━━━━━━●━━━━━━━━━━━●   ← C0（平线，10.0 恒定）
    |● 10.1    ●10.1        ●9.9        ●10.0
  9 |   ● 9.6
  8 | ● 8.3  ← Ceager 起点
    +----------+-----------+-----------+-----------+→ prompt 字符
      8-12k      12-20k      20-40k      40-52k
```

**两条线，两种病**（出租车比喻）：

- **C0（混合空草稿）= 平线**：不管 prompt 8k 还是 52k，每步都恰好多花 10.0ms——这是**纯起步价**（混合边界税，Q0 的 ④）。手续费和"文件多厚"无关，几道章就是几道章。
- **Ceager（全 eager 带草稿）= 斜线**：短 ctx 多花 8.3ms，长 ctx 多花 12.2ms。= **起步价（F+管线 ~7）+ 里程费**。里程费 = 每个草稿位置验证时要读一遍**该请求的 KV cache**，ctx 越长 KV 越长，读得越久 → 每个位置的 0.62ms 在长 ctx 时涨到 ~0.9ms。50k 处比 8k 处多的 ~4ms 就是这段里程费。

**这就闭环了"长 prompt summary 亏损"**：34-52k 的 summary 正好踩在里程费最贵的区段，而它们的命中率又不高（付了里程费还常常不中）→ 净亏。**治法是策略不是系统**：ctx 感知——上下文太长的 summary 就少押注或不押（因为里程费贵、赢面又没变高）。

## Q6：C0−A=10 是混合成本；你是说 suffix 每步成本随 ctx 涨吗？疑问：随草稿量涨吗（似乎不会）？但 k8pad 为什么更慢？

三个小问，逐个答，全部用 Q4 的"激活费 vs 按量费"钥匙。

**① "C0−A=10 是混合成本"——对。** C0−A 那条平线（10ms）= 混合边界税 ④(8.1) + spec 管线 ③(1.9)，全是"和内容无关"的固定开销，所以是平的。

**② "每步成本随 ctx 涨吗"——是的（但只有 Ceager 那条斜线涨）。** 见 Q5：里程费部分随 ctx 涨。这是"每个草稿位置验证读 KV，ctx 越长越贵"。

**③ "随草稿量涨吗？似乎不会"——一半对。** 你的直觉大概来自"固定税与草稿量无关"这句——那说的是**激活费/边界税**（台阶，有草稿就 5.8，不随长短）。**但按量费那 0.62/位置是随草稿量涨的**。所以完整答案：**每步成本随草稿量涨，但只涨那小的一项（0.62/位置），大的激活费不涨**。

**④ "那 k8pad 为什么更慢"——正是被这 0.62/位置咬的。** k8pad 用 dummy token 把每步**强行填满 8 个草稿位置**，而默认策略只押平均 2.7 个。多出的 5.3 个位置 × 0.62 = **3.3ms**：

```
Ceager（默认，2.7 位）  = 24.0 + 2.4 + 5.8 + 0.62×2.7 = 33.9
CeagerK8pad（填满 8 位）= 24.0 + 2.4 + 5.8 + 0.62×8.0 = 37.2   （+3.3ms，实测对上）
```

而且那 5.3 个多押的位置**全是 dummy、必被拒**——纯烧钱不赢。**e2e 里更惨**：并发下这些废位置的验证成本被批共享放大，接受率从 40% 崩到 13%（[G-e2e 0.97x](examine-spec-tax/overnight-verdicts-2026-07-06.md)）。

> **一句话钉死**：k8pad 慢**不是**激活费变了（还是 5.8），是**被强行拉高了草稿位置数 → 按量费爆炸 + 废位置零命中**。这反过来印证 Q4 的模型：草稿费真的是"激活（固定）+ 按量（线性）"两段。

## Q7：做 suffix + EAGLE 混合投机，要不要考虑每步成本增加？

**要，而且它是设计的第一约束——不算这笔账，混合很可能"赢了命中率、输了总时间"。**

先补一个新术语：**EAGLE** 是"模型式草稿"——不像 suffix 那样查树抄现成的，而是用一个**小 draft 模型跑一次前向**猜下一批 token。它的强项正好是 suffix 的弱项：suffix 吃"逐字重复"，EAGLE 吃"分布相似"（paraphrase 型内容）——summary 恰是 paraphrase 型，理论上是 EAGLE 的主场。

**用五常数模型算这笔账**，混合会动三个地方：

1. **多一次 draft 前向（新成本，可能很贵）**：suffix 查树才 0.16ms，EAGLE 要跑一次小模型前向——这个可能是**几 ms**（取决于 draft 模型多大）。这是混合相对纯 suffix 净增的每步成本，直接进"分母"。
2. **激活费别付两次（架构红线）**：如果做成"串联"（suffix 先试，不行再叫 EAGLE），最坏情况每步开两次赌桌 = **激活费 5.8 交两遍** = 灾难。**必须做成"择一/并联共享验证"**——一次激活费，选更好的草稿源，或合并两边草稿一起验证。
3. **验证位置可能变多（按量费涨）**：如果合并两边草稿一起验，位置数上升 → 0.62/位置那项涨。

**决策规则**（还是那个盈亏公式）：

```
净收益 = 每步接受 token 数 ÷ (1 + 每步附加费/24) − 1
         ↑ EAGLE 抬高它（分子）        ↑ EAGLE 也抬高它（分母：多一次 draft 前向）
```

EAGLE 把 summary 命中率从 26.7% 往 report 级（60%）抬 = **分子涨**；但每步多付一次 draft 前向 = **分母涨**。**两者净算才知道值不值**——关键变量是"draft 模型多快"。draft 模型越小越快，混合越容易净赚。

**务实建议**（写进[提案路线 3](explore-idea/proposal-mechanism-acceptance-uplift.md)）：
- **不要串联**（避免双激活费）；
- 优先做**按 tag/ctx 路由到单一 drafter**（report/结构化 → suffix 抄得准；summary paraphrase → EAGLE），而不是每步两个 drafter 都跑——路由零额外每步成本，只在"选谁"上花一点 CPU；
- 若一定要真并联，用**共享验证**（两边草稿合并成一个验证批），把激活费摊成一次。

> **一句话**：混合投机能不能赢，取决于"EAGLE 抬高的命中率"能不能盖过"多一次 draft 前向 + 可能的双激活费"。**成本模型（尤其激活费不可付两次）是这个设计的硬约束，不是事后优化。**
